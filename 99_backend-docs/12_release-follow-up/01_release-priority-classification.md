# 6/15リリース後の未完了機能・優先度整理

## 目的

6/15リリース時点で未実装、未完了、またはリリース後対応予定となっている機能を分類し、今後の開発優先順位を判断しやすくします。

この資料は共有向けに再編集した整理です。Backlog上のステータスは実装実態とずれている可能性があるため、打合せでは現在の実装状況、未完了範囲、残対応の期限を確認します。

## 分類の見方

| 区分 | 判断軸 | 打合せでの扱い |
| --- | --- | --- |
| 1. リリースに間に合わせたかったが、間に合わなかった機能 | リリース品質、DB移管品質、営業開始前提に関わるもの | 機能要望ではなく、原因、恒久対応、再発防止、確認範囲を確認する |
| 2. リリースには間に合わないと聞いていたが、リリース後早く導入したい機能 | 顧客利用、営業活動、運用価値への影響が大きいもの | 短期導入候補として、優先順位と着手時期を決める |
| 3. リリース後に順次開発を予定していた機能 | 通常ロードマップとして扱えるもの | 既存ロードマップ内で順序、仕様確定時期、担当範囲を整理する |

## 打合せで確認したいこと

| 確認テーマ | 確認内容 |
| --- | --- |
| リリース品質、DB移管品質 | 1の項目について、原因、恒久対応、再発防止、本番確認済み範囲を確認する |
| 短期導入順 | 2の項目について、営業開始と顧客利用への影響が大きい順に並べる |
| 通常ロードマップ | 3の項目について、仕様確定時期、着手順、リリース予定を確認する |
| Backlog整理 | Backlog上のステータスが実態とずれている可能性があるため、更新対象とクローズ条件を確認する |

- この資料は会議メモではなく、共有可能な開発優先度の整理です。
- 人名、担当者評価、会議経緯、内部事情は記載しません。
- Backlog番号は管理上の参照であり、現象の根拠そのものではありません。
- 実装判断に必要な詳細仕様は、必要に応じて正式な画面仕様または運用手順へ昇格します。
