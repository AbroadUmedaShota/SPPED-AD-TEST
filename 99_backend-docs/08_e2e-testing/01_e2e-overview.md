# E2E-001 E2Eテスト概要

## 基本情報

| 項目 | 内容 |
| --- | --- |
| 対象 | SPEED ADの主要業務フローを利用者導線として確認するテスト整理 |
| 現状 | 静的JSONをデータソースとするフロントエンドモック開発フェーズ |
| 主な確認手段 | 手動テスト、ブラウザ確認、将来のE2E自動化 |
| 正本参照 | `docs/ハンドブック/テスト/03_TESTING_GUIDELINES.md` |

## 位置づけ

現行のテストガイドラインでは、自動化された単体テストや結合テストの導入は行わず、開発者による手動テストを中心に品質確認する方針です。

このカテゴリでいうE2Eテストは、利用者が実際に通る画面導線を起点に、画面遷移、データ表示、操作結果、エラー表示、レスポンシブ表示を横断して確認するための整理です。

## テスト種別の違い

| 種別 | 目的 | 現在の扱い |
| --- | --- | --- |
| 手動テスト | 仕様通りのUIとインタラクションを人が確認する | 現行の中心 |
| E2Eシナリオ | 利用者導線を通しで確認する | 手動確認手順として整理中 |
| E2E自動化 | ブラウザ操作をスクリプト化して回帰確認する | 将来候補 |
| 単体テスト | 関数や小さなモジュール単位で確認する | 現段階では未導入 |
| 結合テスト | 複数モジュールやAPI連携を確認する | 現段階では未導入 |

## 確認観点

- 画面仕様と既存UIパターンに沿って表示されること。
- 主要画面遷移が期待どおりに動くこと。
- JSONやモックデータの内容が画面に正しく反映されること。
- 検索、フィルタ、ソート、ページネーション、バリデーションが期待どおりに動くこと。
- Chrome、Firefox、Edge、Safari確認可能環境で表示崩れがないこと。
- desktop、tablet、mobileの代表幅で操作できること。

## 差分メモ

- 既存ガイドラインは手動テスト中心であり、E2E自動化の実装方式までは決めていない。
- 一部仕様ではCypressが例示されているが、リポジトリ全体の自動E2E標準として確定しているわけではない。
- この資料では、実装済みの自動テストがあるかのようには扱わない。

## 未確認事項

- E2E自動化を正式に採用する場合のツール選定。
- CIで実行する範囲と、手元確認に留める範囲。
- 本番環境、ステージング環境、開発環境のどこまでを自動操作対象にするか。
