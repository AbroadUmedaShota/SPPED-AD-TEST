## 6. 名刺データ化の確定メカニズム（実測で判明）

別作業者がデータ化・照合を扱う際の前提になる重要事項。

- **OCRが「1入力」としてカウントされる**。OCR＋人間で同一値が2つ揃うと項目が確定し、次グループへ進む。
- OCRが正確に読める項目（メール・URL＝ASCII）は**人間1入力で確定**。OCRが誤読しやすい項目（**FAX・氏名・電話番号**）は**人間2人の一致入力が必要**。
- 名刺に値が無い項目（例: 電話番号2）は**空のままで確定**できる。
- **確定済みの項目は入力欄が disabled（グレーアウト）になり再入力不可**＝システム側で自動的にブロックされる（オペレーターが避けるのではなく、システムが受け付けない）。
- 1グループ内でも項目単位で確定し、未確定項目だけが残る。全項目確定でレコードが次グループへ前進。

> 完了判定の注意: matching_list の「未入力」は **1つも入力が無い件数**で、入力途中（一部項目が未確定）の記録は未入力に出ない。**完了は「各グループ作業可能件数が0」かつ「matchingの一致+不一致=総件数」で判断**する。

## 7. 自動化のハマりどころ（再実施者向け）

- データ化入力一覧の「作業可能件数」は**全アンケート横断のグローバルキュー**。特定アンケートの進捗判断には使えない（matchingの総件数=一致+不一致で見る）。
- 自動入力で**確定済み(disabled)の入力欄に値を入れようとすると操作がタイムアウトし、レコード処理全体が失敗**する。enabledな未確定項目のみ入力すること。
- 作業画面は名刺webp＋画像処理で**ブラウザのレンダラがクラッシュ（Target crashed）しやすい**。数件ごとにブラウザを再起動する設計が必要。
- **無関係カードの「スキップ」は他アンケートの実レコードにも作用しうる**（グローバルキューのため）。自分の対象（画像URLの surveyId や名刺内容）以外には触れない設計にする。
