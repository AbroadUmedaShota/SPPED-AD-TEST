# E2E-004 自動化検討

## 基本情報

| 項目 | 内容 |
| --- | --- |
| 対象 | Playwright/Cypress等による将来のE2E自動化候補 |
| 現状 | 自動E2E標準は未確定 |
| 方針 | まず手動E2Eシナリオを安定化し、自動化候補を絞る |
| 対象外 | この資料追加時点でのテストコード、CI、依存追加 |

## 自動化の優先順位

| 優先度 | 対象 | 理由 |
| --- | --- | --- |
| 高 | ログインからダッシュボード表示 | ほぼ全導線の入口であり、破損検知価値が高い |
| 高 | 回答URL表示から回答送信完了 | 来場者向けの主要体験で、JSONとの整合確認が必要 |
| 中 | アンケート作成から関連設定ページ遷移 | 操作数が多く、導線欠落を検知しやすい |
| 中 | 名刺データ化設定と見積表示 | 請求や通知に接続するため、回帰確認価値がある |
| 中 | お礼メール設定の状態表示 | 会期・データ化状態との接続が複雑 |
| 低 | 実メール送信、請求確定、削除処理 | 副作用が大きく、専用環境とデータ初期化が必要 |

## 安定化の前提

- クリック対象には、テキストだけでなく安定したIDやdata属性を付ける。
- テスト用シナリオUIは本番要件ではないため、E2Eでは明示的に検証補助として扱う。
- 日付や時刻に依存する画面は、固定データまたは固定時刻の前提を用意する。
- 一覧の並び順や件数は、モックJSONの内容とセットで確認する。
- 実送信、請求確定、削除など副作用がある操作は、localまたは専用環境でのみ扱う。

## ツール候補

| ツール | 向いていること | 注意 |
| --- | --- | --- |
| Playwright | 複数ブラウザ、レスポンシブ、トレース、アクセシビリティ寄りの確認 | 導入時はテストデータと待機条件の設計が必要 |
| Cypress | 画面操作の可視化、フロントエンド中心のE2E | 既存仕様で例示はあるが、標準採用は未確定 |
| 手動チェックリスト | 仕様未確定箇所やデモUIの確認 | 回帰頻度が高い箇所は自動化候補にする |

## CI化する場合の入口

1. local静的サーバーを起動する。
2. 代表シナリオを1本だけ自動化する。
3. 画面遷移、主要表示、コンソールエラーなしを確認する。
4. テストデータ初期化手順を固定する。
5. PR時に軽量E2E、リリース前に広めのE2Eを実行する。

## 未確認事項

- PlaywrightとCypressのどちらを標準にするか。
- CIで対象にするブラウザと画面幅。
- テストアカウント、テストデータ、通知送信の安全な分離方法。
- 本番類似環境で副作用のある操作をブロックする仕組み。
