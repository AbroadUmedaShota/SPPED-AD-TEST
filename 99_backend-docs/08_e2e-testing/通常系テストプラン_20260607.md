## 0. スコープと定義

### IN（通常系＝ハッピーパス）
- 正しい認証情報・権限・手順で機能を使ったとき、期待する業務アウトカムが得られること
- 各画面の主要アクション（CRUD・送信・DL・設定保存・アップロード）が正常完了すること
- E2Eの主要フロー（作成→回答→名刺入力→照合→完了データ→ユーザーDL→お礼メール）が一気通貫で成立すること
- 権限Lv別に「到達できる機能が仕様どおり表示・正常動作」すること

### OUT（今回除外）
- 異常系・エラー系の網羅（ただし下記ガードレールで「必須バリデーションの生死確認」のみ最小限IN）
- 越権・権限違反の能動的検証（既知 BUG-ADM-04 など）
- 負荷・性能・スケール／セキュリティ・ペネトレーション
- 本番環境・本番課金・本番メール送信

## 1. 前提・ガードレール（御局らでんの批判を反映・実行前に必須）

> リリース判定の信頼性を担保するため、ケース実行前に下記を満たすこと。

### G-1【最重要】既知バグとハッピーパスの交差を切り分ける
- BUG-ADM-04（Lv2が制限URLに直到達）が未修正の間、**Suite I の Lv2 ケースは「Lv2でメニュー外画面を直URLで開く→アクセス拒否される」ことをP0前提条件**として置く。拒否されなければ Suite I を `blocked` とし、Lv3以上のみでハッピーパスを通す。
- 「既知バグ画面の通常系を通す」場合は、当該ケースに既知バグIDを注記し、"正常に見える越権"を Pass と誤記録しない。

### G-2【最重要】「P1 FailはPM判断でリリース可」を手続き化
- P1 Fail 許容条件を事前列挙（例：回避策あり／影響はLv4管理者のみ／翌営業日までに修正コミット）。
- Fail発生時、QA部長が「影響範囲・回避策・修正見込み」1枚を作成→PMが読了の上で記録付き判断（口頭可否のみは不可）。

### G-3【最重要】STG環境前提の単一障害点を監視
- 実行前チェックリスト（担当サイン必須）：
  - [ ] STGで課金が発火しない（または依頼確定を「直前停止」運用）か
  - [ ] 会期日付の操作可否と、操作前/後/復元値の記録手順
  - [ ] メール送信先が STG ドメインに**強制リダイレクト**される設定か（本番アドレス誤送信防止／SRE沙花叉に確認）
- 前提が崩れたら Suite E・M は実施不可。エスカレーション先を明記。

### G-4 必須バリデーションの「生死確認」を最小限IN
- 各フォーム系Suite（A/B/D/E/G/H）に「必須項目の空送信」「文字数上限+1」の**2ケースのみ**P1で追加（異常系の全量ではなく、バリデーションが生きているかの確認）。

### G-5 代表値選定の属人化防止
- 設問タイプ×必須×名刺添付の全網羅はしない代わりに、「境界の低/中/高」「必須/任意の代表組」の**選定ルール**を本書に明記（§5）。テスターが迷わず同じ絞り方をできるようにする。

### G-6 テストデータの独立性
- 各Suiteに「前提データ」「実行後のデータ状態」を明記。削除系ケースは自分が作った複製のみ対象。
- Suite M（E2E）の直前に初期投入/リセット手順を挟む。生成物には Suite 固有プレフィックス（例 `TEST-M-…`）を付け相互汚染を防ぐ。

### G-7 P2スキップ可の条件明示
- 「スキップ可」とできるのは**今回リリースで非公開の機能**に限る。公開機能はP2でもスキップ不可（必要ならP1格上げをPM判断）。

## 5. 粒度方針・設問タイプの代表値ルール（G-5）

- 階層: Suite → シナリオ(1業務フロー) → ケース(1操作) → バリエーション(値/状態)。記述は **ケース〜バリエーション** まで。
- **設問タイプ（2026-06-08 STG実機で確認＝11種）**: 自由記述 / 単一選択 / 複数選択 / 尺度・評定尺度(**3/5/7段階を個別**) / **ドロップダウン回答** / 数値入力 / **マトリクス（単一）** / **マトリクス（複数）** / 日付・時刻 / 手書きスペース(**Premium**) / 説明カード。
  - ※当初プランの8種に対し、実機は **ドロップダウン・マトリクス単一・マトリクス複数の3種が追加**。
  - ※**エディタは「設問を追加」→ 各設問カード内の `select` でタイプ切替**（タイプピッカーUIではない）。手順記述はこれに準拠。
  - ※**手書きスペースは Premium 枠**。非Premiumアカウント(abroadcrew02)では作成/回答の実走に別アカウントが必要。
  - ※マトリクスは「行（行を追加）」「列（列を追加）」を個別入力。回答画面では表形式（行×列のラジオ/チェック）で描画。
- 軸: 必須ON/OFF、名刺画像添付ON/OFF。
- 絞り方ルール:
  - 各タイプは「必須ON・名刺添付ON」で基本1ケース
  - 評定尺度は 3/5/7 を個別（UIが変わるため）
  - マトリクスは単一で代表確認（複数は同型・差分のみ）
  - 「必須OFF」「名刺添付OFF」は代表1タイプで確認（機能共通のため）

### 実機で判明した通常系の挙動（ケース化推奨・2026-06-08）
- **回答送信時、名刺未添付だと確認モーダル**「名刺データが添付されていません。このまま送信しますか？」→「はい/いいえ」。`はい`で送信＝名刺なし送信(D-3-05)。
- **ログアウトは確認ダイアログ**「ログアウトしますか？」→ 承諾で /login。ログアウト後の保護ページ直URLは /login へ遷移（A-3-02）。
- **エディタ離脱時に未保存ガード**（beforeunload/確認）。
- 新規作成直後の survey は **設問0件**から開始（「設問を追加」で1件目を作る）。
