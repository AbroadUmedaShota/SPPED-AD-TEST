API契約

GET ?resource=cases|screens|runs|results

一覧取得。ページ初期表示時に `cases` と `screens` を一括取得する。

POST { action: "createRun", payload: {...} }

環境、テスター、ブラウザ、Viewportを指定して実行回を作成する。

POST { action: "upsertResult", payload: {...} }

`run_id` と `case_id` をキーに結果、実績、証跡URL、Backlog等の不具合リンクを更新する。

{ ok: boolean, data?: any, error?: string, updatedAt?: string }

成功・失敗を同じ形で返す。エラー時はページ側で再試行せず、手動保存を案内する。

公開方式

GitHub PagesはHTML/CSS/JavaScriptの静的公開。動的な保存処理はApps Script Web Appへ委譲する。

GitHub Pages docs open_in_new

Apps Script境界

Web Appとして `doGet(e)` と `doPost(e)` を実装し、Spreadsheetへの読み書きとバリデーションを担う。

Apps Script Web Apps open_in_new

制限への対応

Sheets/APIの割当を避けるため、初期ロード一括取得、結果保存単位の更新、指数バックオフ検討を前提にする。

Sheets API limits open_in_new

運用ルール

ステージング実行時の扱い

verified_user

正常系のみ

今回の項目出しは正常系に限定し、例外入力、権限違反、破壊的操作は別フェーズに分ける。

outgoing_mail

実送信を避ける

メール送信、請求確定、外部通知は送信直前の表示確認までを基本にする。

photo_camera

証跡を残す

テスト結果は画面スクリーンショット、CSV名、URL、ブラウザ、実行日時をセットで残す。

bug_report

不具合は分離

不具合が出たケースはテスト結果にリンクを残し、再テスト対象として別Runに引き継ぐ。
