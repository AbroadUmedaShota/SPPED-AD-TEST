# テスト計画書

対象: ログイン機能（ユーザー側 / 管理者側）
対象環境: 本番ステージング `https://stg.speed-ad.com`
作成日: 2026-06-03

## 1. テスト目的

- ユーザー側ログイン（`/login`）と管理者側ログイン（`/admin/login`）が、正規資格情報で正しく認証され、それぞれの初期画面（ユーザー: ダッシュボード `/dashboard`、管理者: 管理者トップ `/admin/top`）へ到達できることを確認する。
- 不正資格情報・未入力・権限境界など、認証の異常系で意図しないログイン成功や情報漏えいが起きないことを確認する。
- 実ステージング環境に対する自動E2Eスモークを確立し、回帰確認の起点とする。実データを更新する副作用（送信・保存・削除）はスコープ外とする。
- 認証情報（メール/パスワード）が成果物・ログ・コミットに残らない運用で実行できることを確認する。

## 3. テストアイテム

- ユーザー側ログイン画面 `https://stg.speed-ad.com/login`（メール+パスワードフォーム、ログインボタン）
- 管理者側ログイン画面 `https://stg.speed-ad.com/admin/login`（フォーム構造はE2E実装時にinspectで確定）
- ログイン成功後の到達画面: ユーザー=`/dashboard`（title「アンケート一覧」）、管理者=`/admin/top`
- 認証の異常系処理（誤パスワード、未入力、未登録メール）
- セッション確立（storageState保存・再利用）

## 4. テストスコープ

### スコープ内
- 正常系ログイン（ユーザー / 管理者、各正規資格情報）
- 異常系ログイン（誤パスワード、空入力、形式不正メール）
- ログイン後到達画面の表示確認・表示阻害コンソールエラー0の確認
- パスワード表示切替・アカウント記憶などフォーム挙動（実画面に存在する範囲）

### スコープ外（理由付き）
- パスワード再設定メールの実送信、新規アカウント作成の確定（実データ・実メール副作用が残るため。`要確認`: 専用使い捨て環境が用意されれば対象化可能）
- Googleログイン（外部IdP依存、自動E2E困難。`要確認`）
- 管理者画面内の更新系操作（保存・削除・確定・送信。観測要件の制約に従い実行しない）
- MFA・IP制限・認証基盤内部（10番仕様で対象外明記）
- 負荷・性能・大量同時ログイン（本スモークの範囲外）

## 5. テスト終了条件

- 正常系E2E（ユーザー/管理者ログイン）がいずれも Pass する。
- 異常系E2Eで「誤資格情報がログイン成功にならない」ことを確認できる。
- 認可境界E2E（R002/TA005）で「ユーザーセッションが管理者画面 `/admin/top` に到達しない」ことを確認できる（期待挙動が Q2 で未確定の場合は、到達できないこと自体を最低ラインとして確認し、確定後に判定を精緻化する）。
- テスト計画・分析・設計・テストケースの各レビューで未解決の P0 / P1 指摘が残っていない。
- 実行できない項目（人間実行ケース・副作用ありケース）は未実行として最終レポートに可視化されている。
- 認証情報が成果物・実行ログ・コミットに残っていないことを確認済み。

## 8. 後続成果物の品質ゲート

- テスト分析: R001〜R004（中〜高リスク）について、正常系・異常系・認可境界・情報漏えいの各観点（TVxxx）を最低1つずつ導出すること。
- テスト設計: 正規/誤り資格情報、空入力、形式不正、権限差の境界・同値分割を明示すること。
- テストケース: コードベース（認証情報非漏えいの静的検査）/ E2E自動（正常・異常・認可）/ 人間実行（セキュリティ目視・Google・パスワード再設定）に分離すること。

## 9. テスト環境

- 対象: `https://stg.speed-ad.com`（実バックエンド・実データ）
- 実行基盤: `.pwtest/` の Playwright（Chromium headless）。`npx playwright test --project=staging`。
- 認証情報: 環境変数 `SPEEDAD_EMAIL` / `SPEEDAD_PASSWORD` / `SPEEDAD_BASE_URL`（管理者用は別変数を実装時に定義）。成果物・コミットに書かない。
