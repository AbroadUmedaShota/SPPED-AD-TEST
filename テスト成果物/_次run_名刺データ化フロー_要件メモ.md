# 次run 要件メモ: 名刺データ化 一連フロー E2E（シナリオ／ユースケーステスト）

このメモは現在のログインrun完了後に立ち上げる **2本目の run-test-process** の入力。ユーザー指示（2026-06-03）を記録。

## テスト分類の位置づけ
本フローは **シナリオ（ユースケース）ベーステスト** として構築する。これらスキル群はシナリオ/ユースケースを正式サポート（計画でアプローチ種別、分析/設計で 事前条件→主フロー→代替→例外→事後条件 に分解、ケースで多段TCへ展開）。
ユーザー（送信者）と管理者（データ化作業者）をまたぐ多段ジャーニーのため、計画フェーズで対象アクター・ゴール・リスク・優先度を特定し、設計でフロー分解する。

## 対象フロー

### 324アンケート（通常プラン版）
1. 既存アンケート(324)に対してアンケート送信 ＋ 名刺送信（名刺画像をアップロード）
2. 管理者画面へ移動 → 名刺データ化画面（`/admin/data_input_*`）で名刺入力
3. 管理者側で名刺データ／名刺画像が問題なくダウンロードできること
4. ユーザー側画面からも問題なくダウンロードできること
   - ※通常プランだとユーザー側DLは「後日（翌営業日相当）」になる見込み。同日完結しない可能性 → タイミング制約として扱い、当日は「DL導線が出る/処理中表示」までを確認、実DLは後日検証 or 別途。

### 325アンケート（オンデマンドプラン版）
- 上記と同じ一連フローの **オンデマンドプラン版**。
- オンデマンドは「最後まで到達できる」見込み（同日でユーザー側DLまで完結しうる）。
- 通常版(324)との差分（プランゲーティング）に注意 → [[bizcard-plan-gating]] 参照（スピードプランでプレミアム限定はオンデマンドのみ等）。

## 実行方針（ユーザー承認済み）
- **副作用フル実行を承認**: 実アンケート送信・実名刺アップロード・管理者実入力・実DL まで行う。324/325 を使い捨て台帳として使用。
- **名刺画像はダミー生成**: 架空の氏名/会社/連絡先のテスト名刺画像をこちら側で生成（個人情報を含まない）。
- 認証情報は環境変数のみ・成果物非記載（ログインrunと同方針）。権限レベル Lv1〜Lv4 アカウントあり（`.pwtest/.auth/cred-map.md` 参照、gitignore済み）。

## 確認ポイント（テスト観点の種）
- アンケート送信が成功し回答可能状態になる
- 名刺画像アップロードが受理される
- 管理者: 名刺データ化画面に対象が出現し、入力→確定できる
- 管理者: 名刺データ(CSV等)＋名刺画像のDLが取得でき中身が妥当
- ユーザー: DL導線・実DL（プラン/タイミング差で同日 or 後日）
- 324(通常) vs 325(オンデマンド) のプラン差・到達範囲差

## 既知の重大バグ（リグレッション再現確認・必須）

- **REG-001（高）**: 名刺データ入力画面の**グループ1の作業画面でEnterキーを押すと「カーネル画面（クラッシュ/エラー画面）」に遷移する**問題が過去に観測された（ユーザー記憶ベース・2026-06-03申告／現存未確認）。
  - 対象: `/admin/data_input_list` のグループ1 → `/admin/data_input_screen`（作業画面）。関連: admin観測要件 STG-ADM-07 / STG-FLOW-DATA-01。
  - テスト観点: 入力欄フォーカス時およびフォーム上でのEnterキー押下で、クラッシュ/エラー画面へ遷移**しない**こと（誤送信・予期せぬsubmitが起きないこと）。各入力欄・各グループ（特にグループ1）で確認。
  - 期待結果: Enterで画面破壊が起きない（正常な次フィールド移動 or 無反応 or 明示的な確定のみ）。再現したら E2E-BUG として起票。
  - 分類: E2E自動で再現を試み、UI崩壊の検知が難しい部分は人間実行で補完。
