## 1. 文書目的

- 本書は、不具合報告フォーム画面の要件を定義する。
- 本書を正本とする。
- 15 番（ヘルプセンター）および 21 番（サポートサイト分離）との役割定義矛盾は、本書で Simple／Detailed の内部分岐方式を正式採用として解消する（§10 参照）。

## 2. 画面概要

- SPEED AD ダッシュボード利用中に発生した不具合・質問を、利用者が現場で即時に送信できる入口画面。
- ヘルプセンター、FAQ、共通フッター、プレミアム加入画面などから遷移してくる。
- 単一画面内に「入力フォーム」と「送信完了メッセージ」を同居させ、排他表示で切替える。
- 想定利用者は、一般利用者（簡易に不具合を伝えたい）、社内オペレーター（詳細な環境情報・社内メタ情報を送りたい）、ヘルプセンター／FAQ で自己解決できなかった利用者。

## 3. 画面要件

### 3.1 上部情報

- 画面タイトル「不具合報告」を表示する。
- ダッシュボード共通のヘッダー／サイドバー／フッターを注入する。

### 3.2 報告タイプ選択（共通セクション）

- ラジオボタンで **Simple（簡易報告）** または **Detailed（詳細報告）** を選択する。
- 既定は Simple とする。
- 選択内容に応じて説明文を切替える。
  - Simple: 「必要最低限の情報で素早く報告できます」
  - Detailed: 「発生環境や期待される動作など、より詳細な情報を提供できます」
- Detailed → Simple に戻した場合は、詳細セクションの入力値を初期化する。
- Simple → Detailed 切替時は、既に入力済みの詳細値は保持する（純粋な追記挙動）。

### 3.3 不具合の内容（共通セクション、両モードで表示）

- **不具合の種別**（必須）: 以下 5 分類から単一選択。
  - 表示崩れ
  - 機能しない
  - データ不整合
  - エラーメッセージ表示
  - その他
- **不具合の概要**（必須、複数行）
- **関連アンケート ID / URL**（任意）
- **再現手順**（必須、複数行）
- **実際の動作**（必須、複数行）

### 3.4 報告者情報（Detailed モードのみ、任意）

- 氏名
- メールアドレス（フィードバック希望時に入力）
- 会社名

### 3.5 発生状況（Detailed モードのみ）

- 発生日、発生時刻
- 発生環境（すべて任意）
  - デバイス（PC / スマートフォン / タブレット / その他）
  - 機種名（自由入力）
  - OS（Windows / macOS / iOS / Android / その他）
  - ブラウザ（Chrome / Firefox / Edge / Safari / その他）
  - ブラウザバージョン（自由入力）
  - SPEED AD 利用画面（ユーザー向け／管理者向けで分類された約 36 種）
- 各選択肢で「その他」を選んだ場合は、対応する自由入力欄を必須化する。
- **期待される動作**（Detailed モード時は必須）
- エラーメッセージ表示有無チェックボックス
  - ON にしたときはエラー文入力欄を表示し、入力を必須化する。
- スクリーンショット添付（§3.7）

### 3.6 管理情報（Detailed モードのみ、任意）

- 関連プロジェクト ID
- 影響モジュール（レポート作成 / キャンペーン設定 / API 連携 / ユーザー管理 / その他）。「その他」選択時は自由入力欄を必須化する。
- **深刻度**（ラジオ 4 段階）
  - 緊急（サービス停止、データ損失）
  - 高
  - 中
  - 低
- 社内メモ（自由記述）
- 担当者候補

### 3.7 スクリーンショット添付

- ファイル選択ダイアログからのアップロードに対応する（**単一ファイル**）。
- 受入形式は **PNG / JPEG / GIF** に限定する。
- サイズ上限は **5 MB** とする。
- プレビュー表示とクリア導線を設ける。
- 現行 HTML に「ドラッグ＆ドロップ」文言があるが、現行実装では対応していない（§11 将来計画）。

### 3.8 プライバシーポリシー表示

- 送信ボタン直上にプライバシーポリシーへのリンクを表示する。
- 外部タブで開き、`rel="noopener noreferrer"` を付与する。

### 3.9 送信完了画面

- 送信成功後は入力フォームを隠し、完了メッセージ（「ご報告ありがとうございました！」）を表示する。
- 「別の不具合を報告する」ボタンからフォーム状態を初期化して再入力できるようにする。

## 4. 状態と挙動

### 4.1 初期表示

- 報告タイプは Simple で初期化する。
- 詳細セクション（`#detailed-report-section`）は非表示で初期化する。
- エラーメッセージ入力欄、および各「その他」自由入力欄は非表示で初期化する。

### 4.2 モード切替

- Simple / Detailed の切替時に詳細セクションの表示状態を切替える。
- Detailed → Simple では、詳細セクション内の全入力値・チェックボックス・添付ファイルを初期化する。
- Simple → Detailed では、既存入力値は保持する。

### 4.3 バリデーション

- 送信押下時にクライアント側でまとめて検証する。
- 検証失敗時はフォーム上部にエラーバナーを表示し、該当フィールドに個別エラーを表示する。
- 必須条件は以下のとおり。

| 条件 | 対象項目 |
| --- | --- |
| 常時必須 | 不具合の種別、不具合の概要、再現手順、実際の動作 |
| Detailed 時必須 | 期待される動作 |
| Detailed かつエラーメッセージ表示 ON | エラーメッセージ本文 |
| Detailed かつ「その他」選択時 | デバイス、OS、ブラウザ、利用画面、影響モジュールの対応自由入力欄 |

- スクリーンショットはサイズ超過・MIME 不一致時に個別エラーを表示し、入力を取消す。
- メールアドレス書式検証、文字数上限は現行実装に存在せず、§11 将来計画とする。

### 4.4 送信処理

- 送信ボタン押下時は、二重送信防止のためボタンを非活性化し「処理中...」表示に切替える。
- Simple モードでは詳細セクションの値は送信しない。
- 選択肢（種別・深刻度・デバイス・OS 等）は日本語ラベルに変換して送信する。
- 発生日と発生時刻は 1 つの日時文字列（`YYYY-MM-DD HH:MM`）に統合して送信する。
- スクリーンショットは Data URI 化して送信する。
- タイムアウトは 15 秒とし、タイムアウト時は「送信がタイムアウトしました」とエラー表示する。

### 4.5 送信成功時

- 入力フォームを隠し、送信完了画面を表示する。
- 「別の不具合を報告する」押下時はフォーム状態を初期化し、入力フォームを再表示する。

## 6. 送信方式

### 6.1 送信先

- 現行の送信先は **Google Forms** の `formResponse` エンドポイントである。
- 自社バックエンドは未保有のため、Google Forms に依存している。
- フォーム ID、各フィールドのエントリ ID、運用担当者は社内運用資料を別途参照する。

### 6.2 CORS 回避方式（hidden iframe + form POST）

- `fetch` / `XHR` で Google Forms へ直接 POST するとプリフライトが拒否されるため、以下の方式を採る。
  1. ユニークな名前の非表示 iframe を動的生成する。
  2. `<form action="<Google Forms URL>" method="POST" target="<iframe name>">` を動的生成し、全項目を hidden input として追加する。
  3. `form.submit()` で実送信する。
  4. `iframe.onload` 発火をもって成功と判定し、送信完了画面を表示する。
  5. 15 秒経っても `onload` が発火しない場合はタイムアウトエラーを表示する。

### 6.3 送信成功判定の既知の限界（偽陽性リスク）

- `iframe.onload` は HTTP レスポンスが返った時点で発火する。
- Google Forms がバリデーション失敗・フォーム削除・エントリ ID 不整合等でエラーページを返した場合も `onload` は発火する。
- クロスオリジン制約で iframe の body を読むことができないため、**クライアント側では原理的に成功の正否を確認できない**。
- 結果として、送信失敗でも送信完了画面が表示されるリスクがある。
- この偽陽性は現行方式の制約として受容し、将来的には自社バックエンド化により解消する（§11）。

### 6.4 送信ペイロードの取り扱い

- Content-Type は `application/x-www-form-urlencoded`（標準 form POST の既定）。
- 空文字・未入力値は送信対象から除外する。
- スクリーンショットは Data URI 文字列として 1 フィールドに格納する。Google Forms 側のテキスト上限により、大きい画像は切捨または送信失敗する可能性がある（§11）。

## 7. 画面遷移

### 7.1 本画面への流入元

- ダッシュボード共通フッターの「お問い合わせ」リンク
- ヘルプセンター画面の CTA
- FAQ 画面の CTA（問題が解決しなかった場合）
- FAQ 画面の検索結果 0 件時のサポート誘導
- プレミアム加入画面の問い合わせ導線

### 7.2 本画面からの外部遷移

- プライバシーポリシー（`abroad-o.com/rule.html`、別タブ）
- 送信先（Google Forms、iframe 内）

## 8. 非機能要件

### 8.1 アクセシビリティ

- `<label for>` によるラベルと入力の関連付けを全フィールドに設ける。
- 標準的な `<form>` 要素によりキーボード操作（Tab 送り、ラジオの矢印キー選択、select 開閉）は成立する。
- `aria-live` / `role="alert"` の付与、スクリーンリーダー対応は現行未実装（§11 将来計画）。

### 8.2 セキュリティ・プライバシー

- プライバシーポリシー表示は送信ボタン直上に固定掲示とする。
- 報告者情報・社内メモは PII（個人識別情報）を含み得る。現状は Google Workspace 側のアクセス権・保存期間ポリシーに従う（運用定義は TBD）。
- 添付ファイルは MIME とサイズの 2 段検証のみ。マジックバイト検証は行わない。
- 送信先 URL はハードコードで環境別分岐は持たない。
- CSRF トークンは Google Forms 方式の特性上持たない。自社バックエンド化時に別途設計する。

### 8.3 外部依存

- Google Forms: フォーム削除・エントリ ID 再発行時は全送信が無効化される。運用上の監視が必要。
- 外部 CDN（Tailwind、Material Icons ほか）: 障害時はスタイルが破綻する可能性がある。送信ロジック自体は同一オリジン JS に実装されているため、CDN 障害下でも動作見込み。

### 8.4 レスポンシブ

- メインコンテナは最大幅固定で中央寄せ。
- 環境入力（デバイス／ブラウザ／OS など）は広い画面で 2 カラム、狭い画面で縦積みに切替える。

## 9. 関連実装ノート

- 現行ルーティングでは、共通 `src/main.js` が `data-page-id="bug-report"` を検知して本画面用の初期化関数を呼び出す。
- `02_dashboard/speed-ad-不具合報告-form/` 配下に React + TypeScript 版のフォーム実装が存在するが、**現行ルーティング・ビルドパイプラインからは参照されていない未参照プロトタイプ**であり、本書の仕様対象外とする（§11 将来計画で整理方針を決定する）。

## 10. 関連仕様書との役割定義

- 本書では、15 番（ヘルプセンター）と 21 番（サポートサイト分離）の間に存在した役割定義の矛盾を、以下のとおり整理する。
  - 15 番: 「不具合・一般問い合わせ兼用で、種別選択を必須」
  - 21 番: 「構造化された不具合報告専用」
- **正式採用**: 本画面は不具合・一般問い合わせ兼用とし、問い合わせ種別は `reportType` の **Simple（一般問い合わせ・簡易不具合報告相当）／ Detailed（構造化された不具合報告相当）** で内部分岐する。
- これにより 15 番の兼用方針を維持しつつ、21 番の構造化要素を Detailed モードとして包含する。
- 21 番が示す WordPress 側 `support.speed-ad.com` への移設計画は本書では扱わず、21 番側の将来計画として保留する。

## 11. 将来計画（Phase 2 以降）

現行実装のギャップ・改善候補を集約する。本書で仕様として保証しない項目。

- **送信先の自社バックエンド化**: Google Forms 依存を解消する。エンドポイント、認証、レート制限、JSON 応答、スクリーンショットの multipart/form-data 受信、CSRF 対策を設計する。
- **送信成功判定の堅牢化**: 自社中継または Google Apps Script Web App によるラップで、JSON 応答ベースの正否判定を可能にする。
- **スクリーンショット送信方式の見直し**: Google Drive 直アップロード＋ URL 送信、または自社バックエンド＋ S3 保管に切替える。現行の Data URI 丸ごと送信は Google Forms 側のテキスト上限に抵触するリスクがある。
- **ドラッグ＆ドロップ実装**: HTML に文言はあるが現状未実装。DnD を実装するか、文言を削除する。
- **文字数上限（maxlength）の整備**: 全 textarea / input で未設定。巨大貼り付けによる送信破綻を防止する。
- **メール書式検証**: 報告者メールアドレスの形式検証を追加する。
- **アクセシビリティ強化**: `aria-live`、`role="alert"`、フォーカス移動の実装。
- **React 版プロトタイプの整理**: `speed-ad-不具合報告-form/` を `_archive/` へ移動、または HTML 版を React 化する際のリファレンスとして維持する方針を決定する。
- **添付ファイル検証の強化**: MIME / サイズだけでなくマジックバイト検証を追加する。
