## 1. 文書目的

- 本書は、FAQ ページの要件を定義する。
- 本書を正本とする。
- 旧「本書が正」方針は破棄済み。

## 2. 画面概要

- エンドユーザー向けの「よくある質問」ページ。
- 以下 4 要素で構成する。
  - ヒーロー（タイトル＋メイン検索窓＋人気キーワードチップ）
  - カテゴリカード（Bento 風のカテゴリ一覧、プレビュー質問付き）
  - 全質問リスト（カテゴリごとにまとまったアコーディオン）
  - 解決しなかった場合の CTA（ヘルプセンター／お問い合わせ導線）
- データは静的 JSON（`data/faq.json`）を 1 回取得し、以降の検索・フィルタ・描画はクライアント側で完結する。
- バックエンドは静的 JSON の配信のみ。サーバ連携・認証・DB は扱わない。

## 3. 画面要件

### 3.1 ヒーロー
- タイトル、サブタイトル、メイン検索窓を配置する。
- 検索窓にはクリアボタンを併置する。入力がある間のみ表示する。
- 検索窓の下に人気キーワードチップ（`popularKeywords`）を並べる。クリックでその語を検索実行する。

### 3.2 スティッキー検索バー
- ヒーローが画面外にスクロールアウトしたタイミングで、画面上部に縮小版検索バーを固定表示する。
- ヒーロー検索窓とスティッキー検索窓は同一の検索状態を共有する。片方に入力した内容はもう片方にも反映する。

### 3.3 パンくず
- 「ダッシュボード > FAQ」を表示する。

### 3.4 カテゴリカード
- カテゴリ単位のカードをグリッドで並べる（1 / 2 / 3 カラム）。
- 各カードに以下を表示する。
  - カテゴリ名、説明、カテゴリアイコン
  - 件数バッジ（そのカテゴリに属する質問数）
  - 注目質問の抜粋（最大 3 件）
  - 「すべて見る」導線
- カードクリックで同ページ内の該当カテゴリセクションへハッシュジャンプする。
- 検索実行中はカテゴリカード領域を非表示にする（ジャンプ先との整合性維持のため）。

### 3.5 全質問リスト
- カテゴリごとに見出しを置き、その配下にアコーディオン形式で質問を並べる。
- 各質問アイテムは、質問行（トリガー）と回答本文（展開領域）で構成する。
- 注目質問は `Pick` バッジを表示する。

### 3.6 アコーディオン挙動
- **単一選択制**とする。任意の質問を開く時点で、既に開いている他の質問は自動的に閉じる。同時に開くのは最大 1 件。
- 既に開いている質問の質問行を再度押下したときは、その質問を閉じる。

### 3.7 検索ステータス表示
- 検索実行中は、結果一覧の上部に「検索キーワード」「ヒット件数」「クリア導線」を表示する。
- 検索語が空になった時点で非表示に戻す。

### 3.8 結果 0 件メッセージ
- 全カテゴリでヒット 0 件のとき、検索結果 0 件メッセージを表示する。
- 導線: 「検索をクリアしてすべて表示」「サポートに問い合わせる（`bug-report.html`）」。

### 3.9 フィードバック UI
- 各回答末尾に「この回答は役に立ちましたか？」セクションを設ける。
- 「はい」「いいえ」の 2 択ボタンを提供し、投票後はお礼メッセージに差し替える。
- 同一質問に対する投票は 1 回のみ受け付ける。

### 3.10 CTA セクション
- 一覧末尾に「解決しませんでしたか？」セクションを配置する。
- 導線: 「ヘルプセンターを見る（`help.html`）」「お問い合わせ（`bug-report.html`）」。

## 4. 状態と挙動

### 4.1 初期表示
- `data/faq.json` を取得する。
- 取得中はローディング表示のみ、失敗時はエラーメッセージのみ、成功時のみ本体領域を表示する（3 領域の排他制御）。
- 取得失敗時は定型メッセージ「情報の読み込みに失敗しました。時間をおいて再度お試しください。」を表示する。

### 4.2 検索の基本方針
- 検索状態は「メイン検索窓の入力値」を単一の真実とし、スティッキー検索窓で入力された値も一度メイン検索窓に同期したうえで検索処理を実行する。
- 検索処理は debounce 250ms で間引く（連続入力時の過剰な再描画を防ぐ）。
- 検索語空のときは全件表示に戻す。

### 4.3 検索ロジック
- **フィルタ: AND 検索**
  - 入力値を空白区切りで単語に分解し、**全ての単語**が「質問文＋回答文」に含まれる質問のみを残す。
  - 判定は小文字化した文字列で行う（大文字小文字を区別しない）。
- **ハイライト: OR 描画**
  - フィルタで残った質問の文面に対し、いずれかの単語に一致する箇所を `<mark>` タグで強調表示する。
  - AND フィルタ後のためどの単語も最低 1 回は一致しており、OR でハイライトしても見た目が崩れない。

### 4.4 IME（日本語入力）対応
- 日本語変換中（`compositionstart` 〜 `compositionend` の間）は検索を発火しない。
- 変換確定（`compositionend`）時点で改めて debounce 付きで検索を実行する。

### 4.5 検索のクリア
- 以下 4 つのいずれかの操作でクリアする。
  - 検索窓のクリアボタン
  - 検索ステータス内の「クリア」ボタン
  - 結果 0 件メッセージ内の「検索をクリア」ボタン
  - 検索窓で `Escape` キー押下（入力あり時）
- クリア時は検索語を空にし、カテゴリカード領域を再表示し、メイン検索窓にフォーカスを戻す。
- クリア直前に展開中だった質問があれば、クリア後に同じ質問を再度展開した状態で復元する。

### 4.6 アコーディオンの高さ制御
- アコーディオンの高さは CSS トランジションで滑らかに開閉する必要があり、そのために JS が展開直前に本文の実高さを計測してスタイル変数へセットする運用とする。
- ウィンドウ幅変更時は、展開中の質問について高さを再計算する。

### 4.7 カテゴリジャンプ時のフォーカス
- カテゴリカードからの `#cat-<id>` ハッシュジャンプ後、到着先セクションへプログラム的にフォーカスを移す（キーボード操作者がそのまま Tab で読み進められるようにするため）。

### 4.8 スティッキー検索バーの表示切替
- ヒーローが画面外に出たタイミングで表示、戻ったタイミングで非表示とする。
- 表示／非表示の切替には CSS トランジション（フェード）を用いる。
- 非表示中はスクリーンリーダーの読み上げ・Tab 順から完全に除外する。

### 4.9 フィードバック投票
- 投票時に以下を行う。
  - 計測用イベント（GA ラベル `FAQ / Feedback / <questionId> - <vote>`）をコンソールへ出力（実送信は画面外の確認事項）。
  - 当該質問 ID を投票済みとして sessionStorage に記録する。
  - ボタン群をお礼メッセージに差し替える。
- 再描画時は投票済み質問については最初からお礼メッセージを表示する。
- 同一質問・同一タブ内での二重投票は不可とする。

## 5. データ要件

### 5.1 データソース
- 実ファイル: `data/faq.json`
- 取得方式: クライアント側の GET 1 回。以降はメモリ上で検索・描画を完結する。

### 5.2 ルートスキーマ

| キー | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `popularKeywords` | string[] | 必須 | 人気キーワードチップとして表示する語の配列。空配列ならチップ領域を非表示。 |
| `categories` | Category[] | 必須 | カテゴリ配列。`questions` が空のカテゴリは描画対象から除外。 |

### 5.3 Category

| キー | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `id` | string | 必須 | カテゴリ識別子。カテゴリアンカー（`#cat-<id>`）と専用アイコン・色の適用キーに使用。 |
| `name` | string | 必須 | 表示名。カード見出し・一覧見出しに使用。 |
| `icon` | string | 任意 | アイコン名。未指定時は ID に応じた既定値、それも無ければ汎用ヘルプアイコンにフォールバック。 |
| `description` | string | 任意 | カテゴリ説明。空なら説明文は描画しない。 |
| `questions` | Question[] | 必須 | 質問配列。 |

### 5.4 Question

| キー | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `id` | string | 必須 | 質問識別子。DOM の関連付け、フィードバック記録キーに使用。 |
| `question` | string | 必須 | 質問文。検索時は一致部分をハイライト。 |
| `answer` | string | 必須 | 回答文。改行は表示時に `<br>` として反映。 |
| `isFeatured` | boolean | 任意 | `true` のときカテゴリカードのプレビュー質問に優先採用し、一覧側で `Pick` バッジを表示。 |

### 5.5 カテゴリカードのプレビュー選定
- 各カテゴリのプレビュー質問（最大 3 件）は以下のルールで決定する。
  1. `isFeatured === true` の質問を先頭から最大 3 件
  2. 不足分を `isFeatured` が偽の質問で先頭から補完

## 6. 永続化

### 6.1 フィードバック状態（sessionStorage）
- 保存先: **sessionStorage**。キー名 `faqFeedbackStatus`。値は `{ <questionId>: true }` の JSON 文字列。
- 保存タイミング: 「はい」「いいえ」投票時。
- 読み込みタイミング: 画面ロード時。パース失敗時は空オブジェクトにフォールバック。
- **寿命: ブラウザタブが閉じるまで**。タブを閉じると消える。別タブ・別セッションには引き継がない。
- 目的: 同一タブ内の再訪問・再描画時に投票済み状態を復元し、二重投票を防ぐ。匿名性を保ちつつ UX を維持する折衷案。

### 6.2 ヘルプセンター（15 番）との永続化方針の差異
| 項目 | FAQ（本書） | ヘルプセンター（15 番） |
| --- | --- | --- |
| ストレージ種別 | sessionStorage | localStorage |
| キー名 | `faqFeedbackStatus` | ヘルプセンター側で個別定義 |
| 寿命 | 同一タブが閉じるまで | 180 日（期限切れで破棄） |
| 意図 | 軽量な二重投票防止に限定 | 訪問間にまたがる長期的な投票済み状態の保持 |

- FAQ は「使い捨て的な一覧ページ」の位置づけのため、長期保存は行わない。
- ヘルプセンターは「何度も参照する記事ページ」の位置づけのため、180 日間は投票済み状態を引き継ぐ。

### 6.3 その他の状態
- 検索語、展開中の質問 ID はメモリ内のみで管理する（リロードで初期化）。

## 7. 非機能要件

### 7.1 アクセシビリティ
- 検索フォームに `role="search"` を付与し、ヒーロー版／スティッキー版で `aria-label` を区別する。
- アコーディオンの質問行と回答本文は `aria-expanded` / `aria-controls` / `role="region"` / `aria-labelledby` で関連付ける。
- 閉状態の回答は `hidden` 属性で DOM から非可視化する（キーボード操作者が閉じた本文に紛れ込まないようにする）。
- ローディング／エラー／検索結果件数の各告知領域にライブリージョンを設定する。
- 検索窓の `Escape` キーでクリアを発火する。
- `prefers-reduced-motion: reduce` のユーザーに対しては、アコーディオン展開・フィードバック切替・カード Hover 等のアニメーションを無効化する。

### 7.2 パフォーマンス
- 検索入力は debounce 250ms、ウィンドウリサイズは debounce 150ms で間引く。
- スクロールハンドラは passive 登録とする。
- 検索は全件クライアントフィルタで完結する。件数増加時のページング等は画面外の確認事項とする。

### 7.3 セキュリティ（XSS 対策）
- JSON 由来の全文字列は、HTML へ挿入する前に必ず HTML エスケープする。
- 検索ハイライトは**エスケープ済みテキストに対してのみ** `<mark>` を挿入する。
- 処理順序は以下を固定とし、逆順にしないこと。
  1. 生文字列を `escapeHtml` でエスケープ（`<`, `>`, `&`, `"`, `'` を実体参照化）
  2. エスケープ済みテキストに対して `highlightText` でキーワード一致箇所を `<mark>` 化
  3. 回答本文は最後に改行（`\n`）を `<br>` に置換
- 逆順にすると、挿入した `<mark>` 自体がエスケープされて表示が壊れる、あるいは JSON 由来の生文字列内 `<script>` 等が実行され得る。どちらの事故も避けるため、この順序は必然。
- カテゴリアンカー URL に含めるカテゴリ ID もエスケープする。

## 8. 外部リンク / 他画面との関係

- ヘルプセンター（`help.html`）からの導線で流入。FAQ 側の結果 0 件時および末尾 CTA からヘルプセンターへ戻す導線を提供する。
- お問い合わせ（`bug-report.html`）: 結果 0 件時および末尾 CTA から遷移する。
- パンくず、ヘッダー、サイドバー、フッターは共通レイアウトとして注入する。

## 9. 本番化要件

本セクションでは、完成済み画面を本番環境へ移す際に、画面からは直接確定できないため開発会社と確認する事項をまとめる。API・DB・ログ・権限などの詳細は、画面上の確定仕様ではなく実装時の確認観点である。

### 9.1 API 契約

以下は確認用のたたき台であり、画面上の確定仕様ではない。詳細スキーマ・パスは開発会社と合意して確定する。

| エンドポイント | 用途 | リクエスト | 主要レスポンス |
| --- | --- | --- | --- |
| `GET /data/faq.json`（静的配信） | FAQ データ本体の取得 | なし | 200 FAQ JSON（§5 スキーマ）／ 404 ファイル未配置 ／ 5xx 配信基盤障害 |
| `POST /api/faq/feedback`（必要な場合） | 回答への有用度フィードバック送信 | `{ questionId: string, vote: "yes"\|"no" }` | 200 `{ ok: true }` ／ 400 不正パラメータ ／ 429 レート超過 |

- FAQ 本体は認証不要の静的配信（CDN 経由推奨）とする。CMS 化する場合は管理系 API を別途 PRD で定義する。
- フィードバック送信 API は本書執筆時点では画面外の確認事項（§9 の「その他未対応」に記載の通り）。実装時は未認証・匿名送信とし、レート制限（同一 IP 単位、具体値は実装時確定）を設ける。

### 9.2 権限・認可

| 利用者種別 | 取扱 |
| --- | --- |
| 未ログインユーザー | 読取のみ可。フィードバック送信可否は §9.1 の API 実装方針に従う |
| ログイン中ユーザー | 読取可。フィードバック送信可 |
| コンテンツ編集者 | 静的 JSON 直接更新または CMS 経由（方針は別途決定） |

### 9.3 業務エラー時の挙動

| エラー種別 | ユーザー表示 | システム挙動 |
| --- | --- | --- |
| FAQ JSON 取得失敗（4xx/5xx/ネットワーク） | 「情報の読み込みに失敗しました。時間をおいて再度お試しください。」（§4.1 準拠） | 自動リトライなし。本体領域は非表示 |
| FAQ JSON パース失敗 | 上記と同じメッセージ | コンソールにエラー出力、本体領域は非表示 |
| フィードバック送信失敗 | サイレント失敗可（UX 最優先）、またはトースト「送信に失敗しました」 | sessionStorage の投票済みフラグは付与しない |
| 検索ヒット 0 件 | §3.8 の 0 件メッセージ | エラー扱いはしない |

### 9.4 データ永続化

- **サーバ側**: FAQ 本体は静的 JSON ファイル。CMS 化時はカテゴリ・質問テーブルを新設（主項目: `id` / `category_id` / `question` / `answer` / `is_featured` / `updated_at`）。
- **クライアント側**: フィードバック投票済みフラグを sessionStorage に保存（§6.1 参照）。
- フィードバック集計を実装する場合は、質問 ID 単位のカウンタテーブルを新設（主項目: `question_id` / `yes_count` / `no_count` / `updated_at`）。個人情報は保持しない。

### 9.5 ログ・監査

- アクセスログ: CDN / 配信基盤の標準ログで足りる。個別の監査ログは不要。
- フィードバック API 実装時は送信イベントログ（質問 ID・投票値・タイムスタンプ・IP）を保管。個人識別情報は含めない。
- 保管期間は運用方針に従う（集計用途であれば 90 日を想定、開発会社と確認）。

### 9.6 想定データ量

- 質問数: 数十〜数百件規模を想定。全件クライアントフィルタで支障が出た時点でサーバ検索へ移行（§7.2 参照）。
- 同時閲覧数: ダッシュボード全体のアクセス規模に従う。静的配信のため CDN キャッシュで吸収。

### 9.7 対応ブラウザ・デバイス

- 想定対応: 最新版 Chrome / Edge / Safari / Firefox（直近 1 年以内のメジャーバージョン）。
- レスポンシブ対応: PC（横幅 ≥ 1024px）／タブレット／スマートフォン（横幅 ≥ 375px）。
- IE および非 Evergreen ブラウザは対象外。

### 9.8 受け入れ基準

- [ ] FAQ JSON 取得成功時のみ本体領域が表示される（ローディング／エラーとの 3 領域排他、§4.1）
- [ ] 検索結果が空白区切り AND フィルタで絞り込まれ、ハイライトは OR で描画される（§4.3）
- [ ] エスケープ → ハイライト → `<br>` 変換の順序が保たれ、`<script>` 等の生文字列が実行されない（§7.3）
- [ ] IME 変換中は検索が発火せず、確定時に debounce 付きで発火する（§4.4）
- [ ] 同一タブ内で同一質問への二重投票が不可（§4.9 / §6.1）
- [ ] アコーディオンが単一選択制で動作し、展開中の他質問が自動的に閉じる（§3.6）
- [ ] `prefers-reduced-motion: reduce` 時にアニメーションが無効化される（§7.1）

## 10. 画面外の確認事項 / 未対応事項

### 10.1 サポートサイト分離時の確認事項（21 番との関係）
- `17_support_site_separation_spec.md` に基づき、FAQ ページを WordPress 側（`https://support.speed-ad.com/faq/`）へ移設する場合の確認事項を整理する。
- 移設後のダッシュボード側は、FAQ への遷移を外部リンクへ切り替える。
- 移設時点でも現行の検索体験を壊さないため、検索クエリは `?search=<query>` 形式で受け渡し可能な仕様を維持する。
- 本書の仕様は「ダッシュボード内 FAQ ページが存在する期間」の仕様であり、分離後は 21 番および WordPress 側の仕様書に主導権が移る。

### 10.2 その他未対応
- 多言語化（i18n）: カテゴリ名・質問文・回答文の言語切替は未対応。対応時は辞書化または言語別ファイル分割を別途設計する。
- フィードバック集計: 現状はクライアント側のコンソール出力のみ。サーバ連携・GA 実送信は画面外の確認事項とする。
- 検索結果のページング／並び替え UI は未提供。件数増加時の対応は画面外の確認事項とする。
- カテゴリカードのプレビュー質問 3 件は `isFeatured` 優先＋先頭補完の単純ロジック。クリック履歴など動的選定は未対応。
