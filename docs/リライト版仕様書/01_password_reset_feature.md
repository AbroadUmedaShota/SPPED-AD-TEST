# パスワードリセット機能 要件定義書

| 項目 | 値 |

| 対象画面 | `02_dashboard/forgot-password.html` / `reset-password.html` / `reset-password-complete.html` |
| 正本区分 | 本書を正本とする |
| owner | product |

## 1. 文書目的

- 本書は、ログイン前のパスワードリセット機能の要件を定義する。
- 本書を正本とする。
- ログイン後のパスワード変更（旧パスワードを入力した更新）は本書の対象外とし、`02_password_change_screen.md` を参照する。

## 2. 画面概要

- 本機能は「ログイン前」に利用する動線である。ログイン画面の「パスワードを忘れた方」リンクを起点とする。
- 3 画面で構成する。
  - **メールアドレス入力（forgot-password）**: 登録メールアドレスを入力し、再設定用 URL の送信を依頼する。
  - **新パスワード入力（reset-password）**: メール記載 URL から遷移し、新しいパスワードを設定する。
  - **完了（reset-password-complete）**: 再設定の完了を表示し、ログイン画面へ戻る導線を提供する。
- 想定利用者は、ログインに失敗した SPEED AD クライアント担当者（営業・運用・経理）である。
- 管理者による代理パスワード再発行は本書の対象外とする。

## 3. 画面要件

### 3.1 メールアドレス入力画面（forgot-password）

- 画面タイトルと説明文を表示する。
  - タイトル「パスワードをお忘れですか？」
  - 説明「ご登録のメールアドレスを入力してください。パスワード再設定用の URL をお送りします。」
- 入力項目はメールアドレスのみ（必須、`autocomplete="email"`）。
- 送信ボタン「送信」を設ける。押下中はボタンを無効化し、スピナーと「送信中...」を表示する（二重送信防止）。
- 送信成功時はフォームを非表示にし、成功メッセージ「ご入力のメールアドレスに、パスワード再設定の手順をお送りしました。メールをご確認ください。」を表示する。
- 「ログイン画面に戻る」リンクを常時表示する。

### 3.2 新パスワード入力画面（reset-password）

- 画面タイトル「パスワードの再設定」を表示する。
- メール記載 URL からのアクセスを前提とする。URL トークンの発行・検証方式は画面外の確認事項とする。
- 入力項目は以下 2 点。いずれも `autocomplete="new-password"`、表示/非表示トグルを設ける。
  - 新しいパスワード
  - 新しいパスワード（確認）
- パスワード入力欄の直下に、強度インジケーター（弱い / 普通 / 安全 の 3 段階）とヒント文言を表示する。
- 送信ボタン「パスワードを再設定」を設ける。
- 「ログイン画面に戻る」リンクを常時表示する。

### 3.3 完了画面（reset-password-complete）

- 完了アイコン・タイトル「再設定完了」・説明文「パスワードの再設定が完了しました。」を表示する。
- 「ログイン画面へ」ボタンからログイン画面へ遷移する。

## 4. 状態と挙動

### 4.1 メールアドレス入力

- 入力値は trim 後に検証する。
- バリデーションは `input` と `submit` の双方で行う。
- 成功表示後は入力欄を再表示しない。再送したい場合はページを再読み込みする。
- 登録有無にかかわらず同じ成功メッセージを返す（メールアドレス存在有無の漏洩を避ける）。

### 4.2 新パスワード入力

- ページロード時に URL トークンをサーバー検証し、無効・期限切れの場合は専用メッセージを表示する想定。
- 入力中は `zxcvbn` によりリアルタイムに強度を算出し、バー幅・色・文言（弱い / 普通 / 安全）を更新する。
- 強度インジケーターは情報提供のみで、送信可否には直接関与しない（必須条件は §5 に従う）。
- 表示/非表示トグル押下時は、対象 input の `type` と Material Icons、`aria-label` を切り替える。
- 送信時は新パスワードとトークンをサーバーに送信し、成功応答を受けて完了画面へ遷移する。
- 送信中はボタンを無効化し、二重送信を防止する（メール入力画面と同等）。

### 4.3 完了画面

- 静的表示のみ。遷移元状態を保持しない。
- リセット成功直後のみ到達可能とする想定（直 URL 到達の扱いは §6 セキュリティで扱う）。

## 5. パスワード強度基準

- 強度基準は `02_password_change_screen.md`（ログイン後変更）と共通とする。
- **必須条件**
  - 8 文字以上。
  - 英字を 1 文字以上含む。
  - 数字を 1 文字以上含む。
- **推奨条件**
  - 記号の組み合わせ。
- UI ヒント文言（正）
  > 必須: 8 文字以上で、英字と数字を必ず含めてください。記号を組み合わせるとより安全になります。
- 確認用パスワードとの一致を検証する。
- 強度インジケーター（zxcvbn）は 3 段階表示（弱い 0-1 / 普通 2-3 / 安全 4）で補助的に提示する。送信可否は必須条件で判定する。

### 5.1 既知の実装差異

- 現行 JS 実装は「8 文字以上」と「数字 1 文字以上」のみ検証しており、**英字必須チェックが欠落**している（例: `12345678` が通る）。UI 文言との乖離として §8 で追跡する。

## 6. 非機能要件

### 6.1 セキュリティ

- リセット URL はサーバー発行の一時トークン付き URL とする。
  - 暗号論的乱数を用いる。
  - 一定時間（例: 1 時間）で失効する。
  - 一度使用したトークンは無効化する。
- 新パスワードはサーバー側で強力なハッシュ（bcrypt 等）により保存する。平文保存は禁止。
- タイミング攻撃対策として、登録有無にかかわらず応答内容・応答時間を均一化する。
- メール送信基盤（SMTP / SES 等）を介してユーザーへリセット URL を送付する。
- 完了画面への直アクセス制御は、画面外の確認事項とする。
- `autocomplete="new-password"` / `autocomplete="email"` を明示し、ブラウザ側の取扱を制御する。

### 6.2 二重送信防止

- メールアドレス入力画面: 送信中はボタンを無効化しスピナー表示する。
- 新パスワード入力画面: 送信中はボタンを無効化しスピナー表示する（現在未対応、§8 で追跡）。

### 6.3 アクセシビリティ

- 必須入力に `required` / `aria-required="true"` を付与する。
- エラー領域は `role="alert"`、`aria-describedby` で入力要素と関連付ける。
- 入力不正時は `aria-invalid="true"` を動的に付与する。
- 成功メッセージ領域は `role="status"` とし、表示時にフォーカス移動を行う。
- 強度インジケーターは `aria-live="polite"` とし、強度変化を読み上げる。
- 表示/非表示トグルは `aria-label` を状態に応じて切り替える。
- 完了画面にも通知領域（`role="status"` 相当）を設ける。

### 6.4 パフォーマンス

- ログイン前フローのため共通ヘッダー/サイドバー/フッターは読み込まない。
- `zxcvbn` は新パスワード入力画面でのみ読み込む。CDN 依存の是非は §8 で追跡する。

## 7. 画面フロー

- ログイン画面 → メールアドレス入力画面: 「パスワードを忘れた方」リンク
- メールアドレス入力画面（入力） → メールアドレス入力画面（成功表示）: 送信成功
- メールアドレス入力画面 → ログイン画面: 「ログイン画面に戻る」
- メール記載 URL → 新パスワード入力画面: ユーザーがメール本文の URL をクリック
- 新パスワード入力画面 → 完了画面: 送信成功
- 新パスワード入力画面 → ログイン画面: 「ログイン画面に戻る」
- 完了画面 → ログイン画面: 「ログイン画面へ」

## 8. 本番実装で対応すること

本書の要件を満たすために解消が必要な項目を列挙する。

| 区分 | 項目 | 内容 |
| --- | --- | --- |
| 画面外の確認事項 | 実サーバー API 接続 | メールアドレス入力画面と新パスワード入力画面をサーバー API に接続する。 |
| 画面外の確認事項 | URL トークン取得・検証 | 新パスワード入力画面は URL パラメータを参照しておらず、直 URL で誰でも到達できる状態。最優先で解消する。 |
| 画面外の確認事項 | メール送信基盤 | リセット URL をユーザーへ送付する経路が存在しない。 |
| 画面外の確認事項 | トークン有効期限・一度使用での失効 | 期限切れ専用メッセージ画面も含め未定義。 |
| バグ | 英字必須チェック | UI ヒントは「英字と数字を必ず含める」を明記するが、JS は数字のみ検証。 |
| 画面外の確認事項 | 新パスワード入力画面の二重送信防止 | 送信中のボタン無効化・スピナーが画面外の確認事項（メールアドレス入力画面のみ対応）。 |
| 仕様未確定 | 強度スコアによる送信ブロック | zxcvbn スコアは表示のみ。最低強度方針を要合意。 |
| 仕様未確定 | タイミング攻撃対策の明文化 | 実 API 実装時に応答均一化を担保する旨を固定する。 |
| セキュリティ | 完了画面の直アクセス防止 | 成功フラグなしで直リンク到達可能。 |
| UX | 再送信導線 | 成功表示後に別メールアドレスで送り直す導線がなく、リロードが必要。 |
| UX | 完了後の自動ログイン | 再設定完了後に自動ログインするかの方針未定（現状は都度ログイン）。 |
| 画面外の確認事項 | 監査ログ | リセット要求・成功/失敗のログ記録方針が未定義。 |
| 未対応 | アクセシビリティ強化 | 成功メッセージのフォーカス移動、強度インジケーターの `aria-live`、完了画面の `role="status"`。 |
| 未対応 | 多言語対応 | エラー文言は日本語ハードコード。i18n 基盤導入時に対応する。 |

## 9. 本番化要件

本セクションでは、完成済み画面を本番環境へ移す際に、画面からは直接確定できないため開発会社と確認する事項をまとめる。API・DB・ログ・権限などの詳細は、画面上の確定仕様ではなく実装時の確認観点である。

### 9.1 API 契約

以下は確認用のたたき台であり、画面上の確定仕様ではない。詳細スキーマ・パスは開発会社と合意して確定する。

| エンドポイント | 用途 | リクエスト | 主要レスポンス |
| --- | --- | --- | --- |
| `POST /api/auth/password-reset/request` | リセットメール送信依頼 | `{ email: string }` | 200 `{ ok: true }`（登録有無を問わず常に同一）／ 429 レート超過 |
| `POST /api/auth/password-reset/verify` | URL トークン検証（新パスワード入力画面ロード時） | `{ token: string }` | 200 `{ valid: true }` ／ 200 `{ valid: false, reason: "expired"\|"used"\|"invalid" }` |
| `POST /api/auth/password-reset/confirm` | 新パスワード確定 | `{ token: string, password: string }` | 200 `{ ok: true }` ／ 400 強度不足 ／ 410 トークン期限切れ・使用済み |

- 全エンドポイントは未認証で呼出可（ログイン前動線）。
- レート制限を設ける（同一 IP / 同一メールアドレスあたりの一定時間内の上限）。具体値は実装時確定。
- 全エンドポイント HTTPS 必須。

### 9.2 権限・認可

| 利用者種別 | 取扱 |
| --- | --- |
| 未ログインユーザー | 本機能の主対象。3 画面いずれも利用可 |
| ログイン中ユーザー | ログイン後パスワード変更（`02_password_change_screen.md`）への誘導が原則 |
| 管理者による代理リセット | 本書の対象外。別途管理者画面で定義 |

### 9.3 業務エラー時の挙動

| エラー種別 | ユーザー表示 | システム挙動 |
| --- | --- | --- |
| メール送信 API 失敗（5xx） | 「ただいま処理に失敗しました。時間をおいて再度お試しください。」 | 自動リトライなし。ユーザー操作に委ねる |
| レート超過（429） | 「しばらく時間をおいてから再度お試しください。」 | 自動リトライなし |
| トークン無効 / 期限切れ / 使用済み | 「URL が無効または有効期限切れです。」専用表示。リセット要求画面への再導線を提供 | リセット要求画面へのリンクを提示 |
| パスワード強度不足（400） | §5 のヒント文言を表示 | フォームを保持しユーザー修正後に再送信 |
| ネットワーク切断 | 「ネットワークに接続できません。接続を確認してから再度お試しください。」 | 自動リトライなし |

### 9.4 データ永続化

- リセットトークン管理用テーブルを新設。主項目: `token_hash`（主キー、平文トークンは保存しない）／`user_id`／`expires_at`／`used_at`／`created_at`／`requested_ip`。
- パスワード保存はユーザーテーブルへハッシュ化（bcrypt 等）して保管。平文保存禁止（§6.1 参照）。

### 9.5 ログ・監査

- 操作ログ: リセット要求、リセット完了。メールアドレスは部分伏字（`a***@example.com`）または一方向ハッシュで出力。
- 監査ログ: トークン発行・検証・使用・失効を記録。トークン値そのものはログに残さない。
- 失敗ログ: 連続失敗検知用に IP・試行回数・タイムスタンプを記録。
- 保管期間は運用方針に従う（最低 90 日を想定、開発会社と確認）。

### 9.6 想定データ量

- リセット要求: 数十件 / 日（クライアント担当者規模）。同時並行リセット要求は十数件以下を想定。
- 性能要件は §6.4 と共通の前提を用い、特別な負荷対策は不要。

### 9.7 対応ブラウザ・デバイス

- 想定対応: 最新版 Chrome / Edge / Safari / Firefox（直近 1 年以内のメジャーバージョン）。
- レスポンシブ対応: PC（横幅 ≥ 1024px）／タブレット／スマートフォン（横幅 ≥ 375px）。
- IE および非 Evergreen ブラウザは対象外。

### 9.8 受け入れ基準

- [ ] 登録メールに対して概ね 1 分以内にリセット URL が届く（メール基盤の SLA に依存）
- [ ] リセット URL は発行から 1 時間で失効する
- [ ] リセット URL は一度使用すると無効化される
- [ ] 未登録メールアドレスでも成功表示が返り、登録有無の差異が外部から判別できない
- [ ] 完了画面は成功フラグなしの直 URL アクセスでは到達できない
- [ ] 両画面で送信中ボタン無効化＋スピナー表示が動作する
- [ ] パスワード必須条件（§5）が UI ヒント文言と JS 検証で一致する（§5.1 のギャップ解消後）

## 11. 用語集

| 用語 | 定義 |
| --- | --- |
| パスワードリセット | ログイン前のユーザーが、登録メールアドレス経由で自らパスワードを再設定する動線。 |
| パスワード変更 | ログイン後のユーザーが、旧パスワードを入力したうえで新パスワードに更新する動線（本書対象外）。 |
| リセット URL | メール本文に記載される、トークン付きの新パスワード入力画面への URL。 |
| トークン | リセット URL に含まれる一時的・推測困難な識別子。ユーザーとリセット要求を紐付ける。 |
| 強度インジケーター | `zxcvbn` のスコア（0〜4）を 3 段階（弱い / 普通 / 安全）で可視化する UI。情報提供のみで送信可否には関与しない。 |
| タイミング攻撃 | 応答時間・応答内容の差から、メールアドレスの登録有無を推定する攻撃手法。 |
