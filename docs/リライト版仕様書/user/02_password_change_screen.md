## 1. 文書目的

- 本書は、ログイン後のパスワード変更画面の要件を定義する。
- 本書を正本とする。業務要件を中心にまとめ、実装詳細は扱わない。
- ログイン前のパスワードリセット（メール再設定）は本書の対象外とし、`01_password_reset_feature.md` で扱う。

## 2. 画面概要

- ログイン中のユーザーが、自身の意思でパスワードを任意のタイミングで変更するための画面。
- アカウント情報モーダル内「パスワードを変更する」導線から遷移する。
- 3 ステップのウィザード形式で進行する: 現在パスワード確認 → 新パスワード設定 → 変更完了。
- 完了表示は、ウィザード内の完了ステップと独立完了ページの 2 系統が併存する（いずれも運用中）。
- サーバー API 連携、CSRF 対策、レート制限は、画面外の確認事項として開発会社と確認する。多要素認証は今回の範囲外とする。

## 3. 想定利用者と対象範囲

### 3.1 想定利用者
- ダッシュボードにログイン済みの全ユーザー。
- 現在のパスワードを把握しており、メールリセットを介さず即時変更したいユーザー。

### 3.2 対象範囲
- パスワード変更ウィザード本体。
- パスワード変更完了画面（独立ページ）。

### 3.3 対象外
- ログイン前の「パスワードを忘れた」フロー。
- 管理者による他ユーザーのパスワードリセット。
- 初回仮パスワードからの強制変更。
- サーバーサイドのパスワード保存・ハッシュ化・履歴管理。

## 4. 画面要件

### 4.1 共通構成
- 共通ヘッダー、サイドバー、フッター、パンくずを表示する。
- 画面タイトル「パスワード変更」を表示する。
- コンテンツは中央寄せのカード領域に 1 カラムで配置する。

### 4.2 ステップ 1: 現在のパスワード確認
- 現在のパスワード入力欄を 1 つ配置する（マスク入力）。
- 「パスワードを忘れた場合」リンクを配置し、ログイン前リセット画面へ誘導する。
- アクション: `キャンセル` / `次へ`。
- 未入力時および検証失敗時はエラー文言を表示する。

### 4.3 ステップ 2: 新パスワード設定
- 新しいパスワード入力欄と確認用入力欄を配置する（各々表示/非表示切替ボタンを備える）。
- パスワード強度メーター（弱い / 普通 / 安全の 3 段階表示）を配置する。
- 強度基準の説明文を常時掲示する（§5 参照）。
- アクション: `戻る` / `クリア` / `変更する`。
- クリア操作は 2 つの入力欄と強度表示をリセットする。
- 各入力欄のエラー文言は欄直下に個別表示する。

### 4.4 ステップ 3: 変更完了（ウィザード内）
- 完了アイコン、完了見出し、案内文、`ダッシュボードに戻る` ボタンを表示する。
- ウィザード内に用意された完了ステップであり、同一ページ上で完結する場合に使用する。

### 4.5 独立完了ページ
- ステップ 2 の変更成功時に遷移する独立したページとして提供する。
- 完了アイコン、完了見出し、案内文、`ダッシュボードに戻る` リンクを表示する。
- URL が変わることで、ブックマーク・再読込にも耐える完了表示として機能する。

### 4.6 完了画面の二重実装について
- ウィザード内完了ステップと独立完了ページは、いずれも現行運用として存続する。
- 現時点ではどちらかへの一本化は行わず、両方の存在を前提とする。
- 到達経路や文言の統一性は、表示要素レベルで揃えることを前提とする。

## 5. パスワード強度基準

### 5.1 必須基準
- 8 文字以上であること。
- 英字と数字を必ず含めること。
- 上記を満たさない場合は変更を受け付けない。

### 5.2 推奨事項
- 記号を組み合わせることで強度を高めることを推奨する。
- 大文字と小文字を混在させることを推奨する。

### 5.3 強度メーターの表示
- 入力内容に応じて「弱い」「普通」「安全」の 3 段階で視覚表示する。
- 強度メーターは視覚的な目安であり、「弱い」判定であっても必須基準を満たせば保存を許容する。

### 5.4 既知の実装ギャップ
- 現在の画面実装では、英字必須のチェックが欠落しており、数字のみのパスワードが通過する不具合を抱える。
- 本書 §5.1 の必須基準を正とし、実装側の修正を要するものとして扱う。

## 6. 状態と挙動

### 6.1 初期表示
- ウィザードはステップ 1 から開始する。
- すべての入力欄は空、エラー表示は非表示とする。

### 6.2 ステップ遷移
- ステップ 1 の検証成功でステップ 2 へ進む。
- ステップ 2 で「戻る」を押下するとステップ 1 へ戻る（入力値は保持する）。
- ステップ 2 の変更成功で、独立完了ページへ遷移する。

### 6.3 入力バリデーション
- ステップ 1: 現在パスワードの空チェックと、サーバー認証を行う。
- ステップ 2: 新パスワードの空チェック、最低文字数、必須文字種、確認一致を検証する。
- エラー発生時は該当欄直下にメッセージを表示し、次ステップへ進めない。

### 6.4 認証・変更処理
- 現在パスワード検証と新パスワード適用は、いずれもサーバーへの問い合わせを想定する。
- 現在のパスワード照合は、画面外の確認事項としてサーバー側で行う前提を確認する。
- 処理中はボタンを非活性化し、二重送信を防止する。

### 6.5 キャンセル
- ステップ 1 のキャンセルはダッシュボードに戻る。
- 入力途中であっても、キャンセル押下時は確認なしで遷移する（現行仕様）。

### 6.6 サイドバー遷移ガード
- 入力欄のいずれかに値が入っている場合、サイドバー経由の遷移前に確認モーダルを表示する。
- ヘッダーリンク・ブラウザ戻る・キャンセルボタン・パスワード忘れリンクはガード対象外とする。

### 6.7 表示/非表示切替
- ステップ 2 の新・確認パスワード欄に表示切替ボタンを提供する。
- ステップ 1 の現在パスワード欄は表示切替を提供しない。

## 7. 非機能要件

### 7.1 セキュリティ
- すべてのパスワード入力欄は初期状態でマスク表示とする。
- 現在パスワード欄に対してブラウザのオートコンプリート属性を設定する。
- CSRF 対策、レート制限、アカウントロック、多要素認証は、画面からは確定できないため開発会社との確認事項とする。
- 実サーバー接続時は、HTTPS 通信・ハッシュ化保存（サーバー側）を前提とする。

### 7.2 アクセシビリティ
- 各入力欄はラベルと関連付ける。
- エラー領域と入力欄の ARIA 関連付け、強度バーの `role="progressbar"` などは、画面外の確認事項とする。
- キーボード操作で全ボタンにアクセスできること。

### 7.3 レスポンシブ
- モバイル・タブレット・デスクトップいずれも単一カラムで表示する。
- ボタン群は画面幅に応じて折返し可能とする。

## 9. 本番化要件

本セクションでは、完成済み画面を本番環境へ移す際に、画面からは直接確定できないため開発会社と確認する事項をまとめる。API・DB・ログ・権限などの詳細は、画面上の確定仕様ではなく実装時の確認観点である。

### 9.1 API 契約

以下は確認用のたたき台であり、画面上の確定仕様ではない。詳細スキーマ・パスは開発会社と合意して確定する。

| エンドポイント | 用途 | リクエスト | 主要レスポンス |
| --- | --- | --- | --- |
| `POST /api/auth/password/verify` | ステップ 1 の現在パスワード検証 | `{ currentPassword: string }` | 200 `{ ok: true }` ／ 401 不一致 ／ 429 レート超過 |
| `POST /api/auth/password/change` | ステップ 2 の新パスワード確定 | `{ currentPassword: string, newPassword: string }` | 200 `{ ok: true }` ／ 400 強度不足 ／ 401 現在パスワード不一致 ／ 409 パスワード履歴衝突（§10 画面外の確認事項連動） |

- 全エンドポイントはログイン認証済みセッション必須。CSRF トークンを付与する。
- 現在パスワード検証は連続失敗回数をカウントし、閾値超過時はレートリミット／一時ロックを返す。具体値は実装時確定。
- 全エンドポイント HTTPS 必須。

### 9.2 権限・認可

| 利用者種別 | 取扱 |
| --- | --- |
| ログイン中ユーザー | 自身のパスワードのみ変更可。主対象 |
| 未ログインユーザー | 本画面にアクセス不可。ログイン画面へリダイレクト |
| 管理者による代理変更 | 本書の対象外（§3.3 対象外） |

### 9.3 業務エラー時の挙動

| エラー種別 | ユーザー表示 | システム挙動 |
| --- | --- | --- |
| 現在パスワード不一致（401） | ステップ 1 の欄直下に「現在のパスワードが一致しません。」 | ステップ遷移せず入力継続。失敗回数カウント |
| 連続失敗ロック（429 / 423） | 「お試しの上限に達しました。時間をおいて再度お試しください。」 | ボタン非活性化。一定時間経過で解除 |
| 新パスワード強度不足（400） | §5 のヒント文言を欄直下に表示 | フォーム保持、再入力を促す |
| 確認用パスワード不一致 | 「確認用パスワードが一致しません。」 | クライアント検証で即時表示 |
| 通信失敗（5xx） | 「ただいま処理に失敗しました。時間をおいて再度お試しください。」 | 自動リトライなし |
| ネットワーク切断 | 「ネットワークに接続できません。接続を確認してから再度お試しください。」 | 自動リトライなし |

### 9.4 データ永続化

- パスワード保存はユーザーテーブルへハッシュ化（bcrypt 等）して保管。平文保存禁止（§7.1 参照）。
- 変更日時（`password_changed_at`）をユーザーテーブルに記録する。
- パスワード履歴管理（直近世代との一致禁止）は §10 画面外の確認事項で扱う。導入する場合は保存方式を開発会社と確認する。

### 9.5 ログ・監査

- 操作ログ: パスワード変更要求・変更完了。ユーザー ID を記録、パスワード値そのものは平文・ハッシュいずれも出力しない。
- 監査ログ: 現在パスワード検証失敗回数、ロック発動・解除。
- 失敗ログ: 連続失敗検知用に IP・試行回数・タイムスタンプを記録。
- 保管期間は運用方針に従う（最低 90 日を想定、開発会社と確認）。

### 9.6 対応ブラウザ・デバイス

- 想定対応: 最新版 Chrome / Edge / Safari / Firefox（直近 1 年以内のメジャーバージョン）。
- レスポンシブ対応: PC（横幅 ≥ 1024px）／タブレット／スマートフォン（横幅 ≥ 375px）。
- IE および非 Evergreen ブラウザは対象外。

### 9.7 受け入れ基準

- [ ] ステップ 1 で現在パスワードが検証 API 経由で判定される
- [ ] ステップ 2 で新パスワード必須条件（§5.1）が UI ヒント文言と JS 検証で一致する（§5.4 のギャップ解消後）
- [ ] 新パスワード確定時はサーバーに送信され、成功応答を受けて完了画面へ遷移する
- [ ] 処理中はボタンが無効化され、二重送信が発生しない（§6.4）
- [ ] サイドバー遷移ガードが、入力値がある状態でのサイドバー遷移のみ作動する（§6.6）
- [ ] 連続失敗時にロック／レート制限が機能し、ユーザー向けメッセージが表示される
- [ ] 変更成功後のセッション再発行方針（§10 画面外の確認事項）が合意・実装されるまで、最低でも変更操作のログが記録される

## 10. 画面外の確認事項

| # | 項目 | 内容 |
| --- | --- | --- |
| 1 | 実サーバー API 連携 | 現在パスワード検証および新パスワード適用を実 API に差し替える。CSRF トークン付与、エラーハンドリング、タイムアウト・リトライを整備する。 |
| 2 | 強度基準の実装整合 | 英字必須チェックの欠落を解消し、画面文言と実装を一致させる。 |
| 3 | 完了画面の一本化 | ウィザード内完了ステップと独立完了ページのいずれかへ統合する（推奨は独立ページ）。 |
| 4 | セッション再発行 | 変更成功後の旧セッション無効化・再ログイン要求を導入する。 |
| 5 | パスワード履歴管理 | 直近世代との一致禁止、現在パスワードとの同一禁止を導入する。 |
| 6 | レート制限・アカウントロック | 現在パスワード誤入力回数に応じたロックとロック解除導線を整備する。 |
| 7 | 多要素認証（MFA） | MFA 有効ユーザーの現在パスワード検証後に追加要素を要求する。 |
| 8 | アクセシビリティ拡充 | `aria-invalid` / `aria-describedby` / `role="progressbar"` の付与、フォーカス移動の整理。 |
| 9 | ナビゲーションガード拡大 | キャンセル・パスワード忘れリンク・ヘッダーリンク・ブラウザ戻るに対するガード方針を整理する。 |
| 10 | オートコンプリート属性拡充 | 新パスワード欄へ `autocomplete="new-password"` を付与する。 |
