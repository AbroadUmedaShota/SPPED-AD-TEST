## 1. 文書目的

- 本書は、グループ編集画面の要件を定義する。
- 本書を正本とする。
- 旧「本書が正」方針は破棄済み。

## 2. 画面概要

- アンケートを複数名で共同編集するための「グループ」単位で、基本情報・請求先・メンバー構成を一画面で管理する。
- 画面上部でグループを選択し、基本情報、請求先情報、メンバー管理の 3 セクションを縦並びで編集する。
- 想定利用者はグループ管理者。一般メンバーのアクセス制御は現状の対象外。
- グループデータは画面初期化時に取得し、未保存変更がある状態で切替・離脱を行う場合は確認ダイアログを表示する。

## 3. 画面要件

### 3.1 上部情報
- パンくずと画面タイトル「グループ管理」を表示する。
- 画面タイトル横に補助説明導線（ヘルプトリガ）を設ける。グループの概念・料金体系を表示する。
- グループ選択プルダウンを配置する。利用者が管理するグループ一覧から編集対象を切り替える。
- 未保存変更がある状態で選択を変更した場合は確認ダイアログを表示する。

### 3.2 グループ情報セクション
- 見出し「グループ情報」を持つ開閉可能なカード。初期は開いた状態とする。
- **グループ名**: 必須入力。空の場合はエラー表示を行う。
- **説明文**: 任意入力のテキストエリア。

### 3.3 請求先情報セクション
- 見出し「請求先情報」を持つ開閉可能なカード。初期は開いた状態とする。
- 請求先モードを 2 択のトグルボタンで切り替える。
  - **作成者情報を利用**: アカウント作成者の登録情報を読取専用で表示する。
  - **グループ専用の請求先を利用**: グループ個別の請求先を設定する。
- グループ専用モード時は以下のサブフローとする。
  - 通常時は登録内容を読取表示し、編集ボタンで編集フォームへ切り替える。
  - 編集フォーム項目: 請求先氏名、部署名、電話番号、郵便番号、住所。
  - 編集フォームは「キャンセル」「保存」の 2 ボタンを持つ。未保存時のキャンセルは確認ダイアログを表示する。
  - 保存時は成功通知を表示し、読取表示へ戻す。
- モード切替およびグループ専用請求先の保存は、画面下部の保存ボタンを有効化する（= dirty 扱い）。

### 3.4 メンバー管理セクション
- 見出し「メンバー管理」を持つ開閉可能なカード。初期は開いた状態とする。

#### 3.4.1 招待エリア
- メールアドレス入力欄とエラー表示領域。
- 役割選択プルダウン（管理者 / 一般）。
- 「招待する」ボタン。
- 入力値は形式チェック（メールアドレス形式）を行い、違反時はエラー表示する。
- 招待済メンバーの重複チェックも行う想定（現行未実装、§5 参照）。

#### 3.4.2 並び替えヘッダー
- 「権限で並び替え」「ステータスで並び替え」「並び替えをリセット」の 3 操作を提供する。
- 各並び替えボタンは昇順/降順をトグルし、インジケーター（矢印アイコン）で現在の状態を示す。
- リセット操作で元の順序へ戻す。

#### 3.4.3 メンバー一覧
- カード形式で縦並び表示する。
- カード構成: アバター / 氏名 / メールアドレス / 権限選択プルダウン / ステータスバッジ / 削除ボタン。
- ステータス種別: 加入済 / 招待中 / アドレスエラー。状態ごとに色で区別する。
- カードの非操作領域クリックで **メンバー詳細モーダル** を開く。
- 権限変更・削除・招待は保存ボタンを有効化する。

### 3.5 フッター
- 画面下部に sticky 配置する。
- 「キャンセル」ボタンと「保存」ボタンを配置する。保存ボタンは未保存変更がある場合のみ有効化する。

### 3.6 モーダル・補助導線
- **メンバー詳細モーダル**（`memberDetailModal`）: メンバーの会社名・部署名・役職・電話番号・住所などを読取専用表示する。
- **新規グループ作成モーダル**（`newGroupModal`）: 本画面からは起動しない。別フロー（ヘッダーメニュー等）で使用する関連モーダルとして参考扱い。
- **グループ管理モーダル**（`groupManagementModal`）: 同上。グループ新規作成・削除の別フローで使用する。
- **確認モーダル**（`confirmationModal`）: 未保存破棄・メンバー削除・グループ切替・内部リンク離脱などで共通利用する。

## 4. 状態と挙動

### 4.1 初期表示
- 画面表示時に以下を取得する。
  - 利用者が管理するグループ一覧
  - 既定グループの基本情報、請求先情報、メンバー一覧
- 現行実装ではハードコードされたダミーデータを使用している（§5 参照）。
- 初期は保存ボタンを無効状態（disabled）で表示する。

### 4.2 グループ切替
- プルダウンの選択変更で切替を行う。
- 未保存変更がある場合は確認ダイアログを表示し、確定時のみ切替を実行する。

### 4.3 請求先モード切替
- モード切替は即時反映し、保存ボタンを有効化する。
- グループ専用モードの編集フォームは独立した dirty フラグを持ち、編集中のキャンセルは確認ダイアログを表示する。
- 編集フォームの保存操作は画面全体の dirty フラグも有効化する。

### 4.4 メンバー操作時
- 招待時は入力クリア、一覧再描画、dirty 化を行う。
- 削除時は確認ダイアログ → 一覧から除去 → 成功通知 → dirty 化を行う。
- 権限変更（カード内プルダウン）は即時 dirty 化する。
- 並び替えとリセットは一覧を再描画する。

### 4.5 保存
- 保存時はグループ基本情報とメンバー一覧（email / role ペア）をサーバへ送信する想定とする。
- 保存成功時は成功通知を表示し、保存ボタンを無効化する。
- 保存失敗時は原因に応じたメッセージを表示し、入力値は保持する。
- **現行実装は `console.log` 出力のみのモック**（§5 参照）。

### 4.6 キャンセル / 離脱警告
- キャンセルボタン押下時、未保存変更があれば確認ダイアログを表示する。
- ブラウザ閉じ・リロード時、未保存変更があれば標準離脱警告を表示する。
- 画面内リンク遷移時も同等の未保存確認を行う。

## 5. 既知の実装ギャップ

本画面は実装上の未完成・デッドコード・データ不整合が複数存在する。改修時に優先的に対処すべき項目を示す。

### 5.1 画面とサービス層の未接続
- 共通サービス `groupService.js`（`fetchGroups` / `createGroup` / `updateGroup` / `deleteGroup` を提供）は存在するが、本画面からは **一切 import されていない**。
- 画面内部でグループ ID（`GROUP001` / `GROUP002` / `GROUP003`）および各 20 名のダミーメンバーをハードコードで生成している。
- `data/core/groups.json` とハードコード ID の体系が一致しておらず、サーバ連携時は ID マッピングとスキーマ吸収が必要。

### 5.2 保存処理が未実装
- 保存ボタン押下時の処理は `console.log` 出力と成功トースト表示のみ。
- `groupService.updateGroup()` への接続、通信中ローディング、エラー処理は未実装。
- 保存ペイロードに **請求先情報（`billingInfo` / `useGroupBilling`）が含まれていない**。請求先モード切替はボタンを有効化するが、送信対象から漏れている。
- 並び順情報も送信ペイロードに含まれない。

### 5.3 Sortable.js がデッドコード
- CDN で Sortable.js 1.15.0 を読み込み、`.sortable-ghost` / `.drag-handle` の CSS も定義しているが、初期化コード（`new Sortable(...)`）がなく、カード側にも `.drag-handle` 要素が存在しない。
- ドラッグによる手動並べ替えは動作しない。完全なデッドコードであり、撤去または実装のいずれかを選択する必要がある。

### 5.4 役割値の英/日混在バグ
- 招待フォームの役割プルダウンは英語値（`member` / `admin`）を保持する。
- メンバーカード内の役割プルダウンは日本語値（`一般` / `管理者`）を使用する。
- ダミー生成メンバーは日本語、招待直後のメンバーは英語で保持される。
- 結果として、招待直後のカード表示で `selected` 判定が一致せず、**表示上の既定値が実際の設定と乖離する**。
- 改修時は内部値を一方に統一し、表示のみ日本語ラベル化する。

### 5.5 その他の未実装・欠落
- URL クエリパラメータ（`?groupId=...`）による初期選択非対応。外部リンクから特定グループを開けない。
- データ取得失敗時のエラーハンドリング未実装。
- グループ名の空チェック未実装（HTML5 の `required` 属性のみ）。エラー表示領域は HTML に存在するが、JS 連動なし。
- 招待時のメール重複チェック未実装（モーダル側には実装あり）。
- グループ専用請求先フォームのバリデーション未実装（必須、郵便番号、電話番号）。
- アクセス権制御未実装。URL 直打ちで誰でも開ける。
- 作成者請求先とグループ請求先でキー名が異なり（`name` / `contactName`）、相互変換ロジックなし。
- 住所 2 行目の入力欄が HTML に存在せず、データモデル上も常に空文字。
- 招待直後のメンバーは詳細情報（会社名・住所等）を持たず、詳細モーダルが空欄になる。
- `innerHTML` によるカード・詳細描画が多く、サーバ連携時はエスケープ対応が必要。

### 5.6 表記の不一致
- `<title>` / `<h1>` / `data-page-id` は「グループ管理」、パンくずは「グループ編集」でラベルが揃っていない。

## 6. データモデル

### 6.1 グループ

| 項目 | 型 | 説明 |
| --- | --- | --- |
| id | string | グループ識別子 |
| name | string | グループ名 |
| description | string | 説明文 |
| useGroupBilling | boolean | グループ専用請求先を利用するか |
| billingInfo | object | グループ専用請求先（§6.2） |
| members | Member[] | 所属メンバー |

### 6.2 請求先情報

| 項目 | 型 | 備考 |

| department | string | 部署名 |
| phone | string | 電話番号 |
| postalCode | string | 郵便番号 |
| address1 | string | 住所 |
| address2 | string | 住所 2 行目（現行未使用、§5.5） |

### 6.3 メンバー

| 項目 | 型 | 備考 |
| --- | --- | --- |
| email | string | 主キー扱い |
| name | string | 氏名 |
| role | string | `管理者` / `一般`（値統一が課題、§5.4） |
| status | string | `加入済` / `招待中` / `アドレスエラー` |
| avatarUrl | string | アバター画像 URL |
| companyName | string | 会社名（詳細モーダル用） |
| departmentName | string | 部署名（同上） |
| positionName | string | 役職（同上） |
| phoneNumber | string | 電話番号（同上） |
| address | string | 住所（同上） |

### 6.4 保存ペイロード（想定）

```json
{
  "groupId": "...",
  "groupInfo": { "name": "...", "description": "..." },
  "billing": {
    "useGroupBilling": true,
    "billingInfo": { "contactName": "...", "department": "...", "phone": "...", "postalCode": "...", "address1": "..." }
  },
  "members": [
    { "email": "user0@example.com", "role": "admin" },
    { "email": "user1@example.com", "role": "member" }
  ]
}
```

- 現行モックは `groupInfo` と `members` のみ送信しており、`billing` および並び順情報が欠落している（§5.2）。

## 7. 非機能要件

### 7.1 パフォーマンス
- メンバー一覧は `innerHTML` 一括再描画で実装されている。数十件規模までは問題ないが、100 件超で仮想スクロール等の検討が必要。

### 7.2 アクセシビリティ
- セクションヘッダーの開閉状態は `aria-expanded` で表現する。
- 並び替えインジケーター・エラー表示領域に `aria-live` / `role="alert"` を付与する（現行未対応）。
- モーダルは `role="dialog"` / `aria-modal="true"` / `aria-labelledby` を備える。

### 7.3 セキュリティ
- サーバ連携時は動的表示の XSS 対策（エスケープまたは `textContent` 化）を徹底する。
- 外部アバター画像（`i.pravatar.cc`）依存はプライバシー・可用性リスクあり、本番前に差し替える。

### 7.4 国際化
- 全テキストは日本語ハードコード。役割値の英/日統一（§5.4）後に i18n 対応を検討する。

## 8. 本番化要件

本書はモックアップ作成段階の要件として作成したが、最終的に本番環境構築を担う開発会社が実装に着手するために必要な情報を本セクションに集約する。`groupService.js` がデッドコード化している（§5.1）ため、本節はその接続を前提に記述する。

### 8.1 API 契約

想定エンドポイントと入出力。詳細スキーマは実装着手時に確定する。

| エンドポイント | 用途 | リクエスト | 主要レスポンス |
| --- | --- | --- | --- |
| `GET /api/groups` | 利用者が管理するグループ一覧取得（プルダウン用） | — | 200 `{ items: Group[] }` ／ 401 未認証 |
| `GET /api/groups/{id}` | 基本情報・請求先・メンバー一覧取得 | — | 200 `Group`（§6 データモデル準拠）／ 404 未存在 |
| `PUT /api/groups/{id}` | 基本情報・請求先・メンバー更新（保存ボタン押下） | §6.4 ペイロード（`billing` 追加済み） | 200 `{ group: Group }` ／ 400 バリデーション ／ 403 権限不足 |
| `POST /api/groups/{id}/invitations` | メンバー招待 | `{ email: string, role: "admin"\|"member" }` | 201 `{ member: Member }` ／ 409 重複 ／ 400 形式エラー |
| `DELETE /api/groups/{id}/members/{email}` | メンバー削除 | — | 204 ／ 403 権限不足 ／ 404 未存在 |
| `PATCH /api/groups/{id}/members/{email}` | 役割変更 | `{ role: "admin"\|"member" }` | 200 ／ 403 権限不足 |

- 全エンドポイントはログイン認証済みセッション必須。CSRF トークンを付与する。
- 内部値は英語（`admin` / `member`）で統一し、表示のみ日本語ラベル化する（§5.4 解消）。
- 保存 API には `billing` と並び順情報を含める（§5.2 解消）。
- 招待は即時送信する前提。メール送信基盤は別途用意する（`01_password_reset_feature.md` §6.1 と同等）。

### 8.2 権限・認可

| 利用者種別 | 取扱 |
| --- | --- |
| 未ログイン | アクセス不可。ログイン画面へリダイレクト |
| 一般メンバー | 本画面アクセス不可（URL 直打ちも拒否）。§5.5 のアクセス権制御未実装を解消 |
| グループ管理者 | 所属グループの基本情報・請求先・メンバー全操作可 |
| 他グループの管理者 | 対象グループのデータは取得不可（サーバー側で二重検証） |

### 8.3 業務エラー時の挙動

| エラー種別 | ユーザー表示 | システム挙動 |
| --- | --- | --- |
| データ取得失敗（5xx） | エラー表示＋再取得導線 | §5.5 の「取得失敗時のエラーハンドリング未実装」を解消 |
| 保存バリデーションエラー（400） | 該当フィールド直下にメッセージ | 入力値保持、他セクションはそのまま |
| 招待重複（409） | 「このメールアドレスは既に招待済みです。」 | 入力クリアせず残す |
| 招待形式エラー（400） | 「メールアドレスの形式が正しくありません。」 | クライアント側でも事前検証 |
| 権限不足（403） | 「この操作を実行する権限がありません。」 | 操作を取り消し |
| メンバー削除の未存在（404） | 「対象メンバーが見つかりません。」 | 一覧を再取得 |
| ネットワーク切断 | 「ネットワークに接続できません。」 | 自動リトライなし |

### 8.4 データ永続化

- グループテーブル: §6.1 準拠（`id` / `name` / `description` / `use_group_billing` / `billing_info`（JSON または別テーブル）／ `created_by` / `updated_at`）。
- 請求先情報テーブル: §6.2 準拠。作成者請求先のキー名（`name`）とグループ請求先のキー名（`contactName`）を API 側で統一する（§5.5 解消）。`address2` を正式に追加する。
- メンバーテーブル: §6.3 準拠（`group_id` / `email` / `role` / `status` / `invited_at` / `joined_at`）。`email` は `group_id` とのペアで一意。
- 内部 `role` 値は `admin` / `member` に統一（§5.4 解消）。

### 8.5 ログ・監査

- 操作ログ: 基本情報保存・請求先モード切替・請求先保存・招待・削除・役割変更。実行ユーザー・グループ ID・対象メンバーを記録。
- 監査ログ: 請求先情報の変更、管理者権限の付与・剥奪。請求・権限の変更は別途監査ログで長期保管（要確定）。
- 失敗ログ: 権限不足・認証失敗の発生時刻・IP を記録。

### 8.6 想定データ量

- 1 グループあたりメンバー数は数十名規模を想定。100 件超で §7.1 のとおり仮想スクロールを検討。
- 1 テナントあたりのグループ数は数個〜十数個を想定。一覧取得は軽量。
- 同時編集は 1 グループあたり 1 名を想定。競合は保存時の `updated_at` 比較で検出（要検討）。

### 8.7 対応ブラウザ・デバイス

- 想定対応: 最新版 Chrome / Edge / Safari / Firefox（直近 1 年以内のメジャーバージョン）。
- レスポンシブ対応: PC（横幅 ≥ 1024px）／タブレット／スマートフォン（横幅 ≥ 375px）。
- IE および非 Evergreen ブラウザは対象外。

### 8.8 受け入れ基準

- [ ] `groupService.js` が画面から呼ばれ、ハードコードのダミーデータが撤去される（§5.1 解消）
- [ ] 保存 API に `billing`（`useGroupBilling` / `billingInfo`）と並び順情報が含まれる（§5.2 解消）
- [ ] 役割値が内部的に英語（`admin` / `member`）で統一され、招待直後とダミー生成メンバーでカード表示が一致する（§5.4 解消）
- [ ] URL クエリパラメータ（`?groupId=...`）で特定グループが初期選択される（§5.5 解消）

## 9. 用語集

| 用語 | 定義 |
| --- | --- |
| グループ | アンケートを複数名で共同編集する単位。作成者が管理者となる。 |
| グループ管理者 | メンバー招待・削除・権限変更・請求先設定ができるユーザー。 |
| 一般メンバー | アンケート編集に参加できる権限のみを持つユーザー。 |
| 作成者情報利用 | 請求先をアカウント作成者の登録情報に合わせるモード。 |
| グループ専用請求先 | グループ個別に登録する請求情報。 |
| Dirty State | 未保存変更の有無フラグ。画面全体と請求先フォームの 2 種を持つ。 |
