# プレミアムプラン登録 SPA 要件定義書

| 項目 | 値 |

| owner | product |

## 1. 文書目的

- 本書は、プレミアムプラン登録 SPA（3 ステップ式）の要件を定義する。
- 本書を正本とする。
- 決済は月末締めの請求書払いを前提とする。クレジットカード入力は本書の対象外。
- プラン紹介ページ（`premium_signup_new.html`）および解約フローは本書の対象外。

## 2. 画面概要

- 無償プラン利用者が、プレミアムプラン（月額 10,000 円税別）への申込を自己完結で実施するための 3 ステップ SPA。
- 同一ページ内でステップを切替える SPA 構成とし、ページ遷移は発生させない。
  - **Step 1: 入力** — 請求情報フォーム。
  - **Step 2: 確認** — 入力内容の表示と規約同意。
  - **Step 3: 完了** — 登録完了を通知し、ダッシュボードへ戻す。
- 想定利用者は SPEED AD クライアント担当者（主に会社の経理・管理部門）である。
- 会員情報からの自動入力、郵便番号による住所検索、離脱防止確認を備える。
- パンくずは「ダッシュボード → プレミアムプラン → 登録設定」。

## 3. 画面要件

### 3.1 ヘッダー・パンくず

- 共通ヘッダー・共通サイドバー・共通フッターを配置する（`header-placeholder` / `sidebar-placeholder` / `footer-placeholder`）。
- パンくずは 3 階層（ダッシュボード / プレミアムプラン / 登録設定）で、末尾「登録設定」が現在位置。
- 画面タイトル「プレミアムプラン登録設定」を中央に表示する。

### 3.2 ステップ進捗

- ステップインジケーター「1. 入力 → 2. 確認 → 3. 完了」を最上部に表示する。
- 各ステップ円は以下 3 状態を持つ。
  - **active**: 現在ステップ（番号表示、強調色）。
  - **completed**: 完了済ステップ（チェックアイコン、強調色）。完了済円をクリックすると対応ステップへ戻れる（Step 2 → Step 1 のみ対象）。
  - **未到達**: 番号のみ表示。
- 連結線（connector）も active 時に色付けする。

### 3.3 Step 1（入力）

- セクションタイトル「請求情報の入力」を表示する。
- 入力項目は以下の 9 件。必須項目には「必須」バッジを表示する。
  - 氏名（姓）（必須、例: 山田）
  - 氏名（名）（必須、例: 太郎）
  - 会社名（必須、例: 株式会社SpeedAd）
  - 部署名（任意、例: 営業部）
  - 役職名（任意、例: 部長）
  - 電話番号（必須、ハイフンなし、例: 0312345678）
  - 郵便番号（必須、7 桁、`maxlength="7"`）＋「住所検索」ボタン
  - 住所（必須、例: 東京都港区六本木1-2-3）
  - 建物名（任意、例: 六本木ヒルズ XX階）
- フォーム右上に **sticky「会員情報を引用」ボタン**を表示する（md 以上、会員情報が有効な場合のみ）。
- 画面下部に「確認画面へ進む」ボタンを配置する。

### 3.4 Step 2（確認）

- セクションタイトル「入力内容の確認」を表示する。
- 注記「以下の内容で登録します。よろしければ『登録を完了する』ボタンを押してください。修正する場合は『修正する』ボタンで戻ってください。」を表示する（ボタン実文言は「この内容で登録する」である点を §8 で追跡）。
- 入力内容を `dl` で読取表示する（氏名 / 会社名 / 部署名 / 役職名 / 電話番号 / 郵便番号（`〒XXX-XXXX`）/ 住所 / 建物名）。任意項目が未入力の場合は「-」を表示する。
- 料金説明カードを表示する。
  - 月額 10,000 円（税別）。
  - 月末締めで計算し、翌月第 1 営業日に管理画面「ご請求内容一覧」からダウンロード可。
  - 課金開始日情報（§4.6）をユーザー区分に応じて挿入する。
- 規約同意チェックボックス（`consent-checkbox`）を設ける。ラベルから以下にリンクする。
  - 利用規約（`terms-of-service.html`、新規タブ）
  - プライバシーポリシー（外部 URL、新規タブ）
- アクションは「修正する」（Step 1 に戻る）と「この内容で登録する」（送信）の 2 つ。チェック ON で送信ボタンが活性化する。

### 3.5 Step 3（完了）

- 完了ビュー「Welcome to Premium」を表示する。バッジアイコンとパーティクル演出を伴う装飾的な表現とする。
- 説明文「プレミアムプランへの登録が完了しました。あなただけの特別な体験が、今ここから始まります。」を表示する。
- 「ダッシュボードへ戻る」ボタン（`index.html` へのリンク）を配置する。

### 3.6 モーダル群

- **会員情報確認モーダル（`account-check-modal`）**: 画面初期化時、会員情報が十分に揃っている場合に表示する。
  - 項目: 氏名／会社名／部署名／役職名／電話番号／郵便番号／住所／建物名／メール。
  - アクション: 「はい、この情報を使って登録へ進む」（自動入力 → Step 2 へ進む）／「いいえ、情報を入力する」（モーダルを閉じ Step 1 へ）。
- **離脱防止モーダル（`prevent-leave-modal`）**: Step 1 / Step 2 で未保存変更がある状態でページ離脱・戻る操作を検知したとき表示する。
  - 文言: 「入力中の情報があります。このページを離れると、入力中の情報が失われますがよろしいですか？」
  - アクション: 「はい、ページを離れる」（遷移を許可）／「いいえ、このページに留まる」（同画面に留まる）。
- **エラーモーダル（`error-modal`）**: 登録処理失敗時に表示する。既定文言「登録処理中にエラーが発生しました。もう一度お試しください。」。アクション「もう一度試す」で閉じ、Step 2 に留まる。オーバーレイクリックでも閉じる。

## 4. 状態と挙動

### 4.1 ステップ遷移

- 初期表示ロード時に会員情報を評価する（§4.4）。
  - プレミアム会員判定時は Step 3 へ直接遷移する（重複登録防止）。
  - 会員情報が有効な場合は会員情報確認モーダルを表示する。
  - それ以外は Step 1 を表示する。
- Step 1 → Step 2 は「確認画面へ進む」押下 + Step 1 全バリデーション通過が条件。
- Step 2 → Step 1 は「修正する」押下またはステップ 1 円クリックで戻る。Step 1 へ戻るたびに入力値は保持するが、規約同意チェックおよび送信ボタン活性化状態はリセットする。
- Step 2 → Step 3 は「この内容で登録する」押下 + 送信成功が条件。
- ステップ切替時は画面上端へスムーズスクロールする。

### 4.2 バリデーション

- 各入力は以下 2 契機で検証する。
  - フィールドの `blur` 時（リアルタイム検証）。
  - 「確認画面へ進む」押下時（全項目を一括検証、`validateForm`）。
- 検証通過時はフィールドに緑のチェックアイコン（`check_circle`）を表示し、枠線を緑に変える。
- 違反時は該当項目直下にエラーメッセージを表示し、枠線を赤に変える。
- 詳細ルールは §5 に記載する。

### 4.3 住所検索

- 「住所検索」ボタン押下で `zip` 値から数字のみを抽出し、外部 API に照会する。
  - 現行実装は `https://zipcloud.ibsnet.co.jp/api/search` を呼出。本番では要確定（§8）。
- 7 桁未満の場合は API を呼ばず、「郵便番号を7桁で入力してください。」を表示する。
- 検索中はボタンを「検索中...」に差替え、二重押下を防ぐため `disabled` にする。
- 成功時は `address1 + address2 + address3` を結合して住所欄に自動入力し、建物名欄へフォーカス移動する。
- 該当なしの場合は「該当する住所が見つかりませんでした。」を表示する。
- API 失敗・ネットワーク切断時は「住所の取得に失敗しました。」を表示する。

### 4.4 会員情報引用

- `localStorage` の `simulationUserData` を初期読込し、未保存の場合はダミーデータで代替する（開発用。本番では §8 で要整備）。
- 会員情報の有効性は「姓」「名」「会社名」の 3 点が埋まっていることで判定する。
- 会員情報確認モーダルから「この情報を使って登録へ進む」を選んだ場合、フォームに自動入力したうえで全項目を検証し、全項目通過なら Step 2 に遷移する。通過しないときは Step 1 に留まる。
- Sticky の「会員情報を引用」ボタンは Step 1 表示中のみ表示する（Step 2 / Step 3 では非表示）。Sticky から押した場合は自動入力のみ行い、Step 1 に留まる（自動遷移しない）。

### 4.5 離脱防止

- Step 1 または Step 2 に滞在中で、かつ未保存変更フラグ `hasUnsavedChanges` が立っているときに作動する。
- 契機は以下の 3 系統。
  - `beforeunload`（ブラウザ閉じ・リロード・URL 直打ち）— ブラウザ標準ダイアログを表示する。
  - ページ内 `<a>` タグクリック — 独自の離脱防止モーダルを表示する。
  - `popstate`（戻る/進む） — 独自の離脱防止モーダルを表示する。
- 「はい、ページを離れる」押下時は、対象 URL（リンククリック時のみ）へ遷移する。
- Step 3 表示中、および「修正する」で Step 1 へ戻った直後の規約チェックリセット時は、未保存変更フラグを落とす。

### 4.6 登録送信

- 「この内容で登録する」押下で以下を行う。
  - 送信ボタンを `disabled` 化し、スピナー＋「処理中...」に差替え（二重送信防止）。
  - 登録ペイロード（§9）をサーバー送信する。
- 成功時は Step 3 に遷移する。
- 失敗時はエラーモーダル（§3.6）を表示し、Step 2 に留まる。送信ボタンは再度有効化する。
- **課金開始日メッセージ** を Step 2 料金カード内に動的挿入する。
  - `is_rejoining_user = true`（再加入者）: 当日契約開始、当月分満額請求。契約期間「本日〜当月末」と注記を表示する。
  - `is_rejoining_user = false`（新規）: 初月無料、翌月 1 日課金開始。翌月 1 日を動的表示する。
- **現行実装は `setTimeout` による成功モック**。実 API 接続・業務エラー分岐は §8 で追跡する。

## 6. 非機能要件

### 6.1 レスポンシブ

- PC（≥ 1024px）／タブレット／スマートフォン（≥ 375px）で崩れなく利用できること。
- Sticky「会員情報を引用」ボタンは md 以上でのみ表示する（モバイルでは非表示）。

### 6.2 アクセシビリティ

- 必須フィールドに `required` / `aria-required="true"` を付与する（現行未対応、§8）。
- エラー領域に `role="alert"`、`aria-describedby` で入力要素と関連付ける（現行未対応、§8）。
- モーダルに `role="dialog"` / `aria-modal="true"` / `aria-labelledby` / フォーカストラップ / Escape 閉じを備える（現行未対応、§8）。
- ステップインジケーターは現在ステップを `aria-current="step"` で示す（現行未対応、§8）。

### 6.3 セキュリティ

- 本画面はログイン済ユーザーのみアクセス可能とする。未ログイン時はログイン画面へリダイレクトする（現行未対応、§8）。
- 既にプレミアム会員のユーザーは Step 3 へ直接遷移させ、重複登録を防ぐ（現行は `is_premium_member` フラグで判定）。
- 登録ペイロードは HTTPS 必須、CSRF トークンを付与する。
- 動的 DOM 挿入（`innerHTML`、§4.6 の課金開始日メッセージ等）はサーバー連携時に固定文言化し、ユーザー入力を混入させない。

### 6.4 パフォーマンス

- 本画面は SPA のためページ遷移が発生しない。ステップ切替は DOM の `hidden` クラス切替のみで完結する。

## 7. 画面フロー

- プラン紹介ページ（`premium_signup_new.html`）「申し込みに進む」CTA → 本画面 Step 1。
- ダッシュボード（`index.html`）→（別導線があれば）本画面 Step 1。※別導線の有無は要確定（§8）。
- Step 1 入力 → Step 2 確認 → Step 3 完了 → ダッシュボード（`index.html`）。
- 既にプレミアム会員: 本画面ロード時に Step 3 へ自動遷移。
- ページ離脱（ブラウザ閉じ・リロード・戻る・内部リンク）: Step 1 / Step 2 かつ未保存変更ありの場合は離脱防止モーダルで確認。

## 9. データモデル

### 9.1 登録ペイロード（想定）

```json
{
  "billing": {
    "lastName": "山田",
    "firstName": "太郎",
    "companyName": "株式会社SpeedAd",
    "department": "営業部",
    "title": "部長",
    "phone": "0312345678",
    "zip": "1060032",
    "address": "東京都港区六本木1-2-3",
    "building": "六本木ヒルズ 33F"
  },
  "consent": {
    "termsOfService": true,
    "privacyPolicy": true,
    "consentedAt": "2026-04-24T10:00:00+09:00"
  }
}
```

- 任意項目（`department` / `title` / `building`）は未入力時に空文字で送信する想定。
- 規約同意はチェックボックス 1 つで利用規約・プライバシーポリシーの両方に同意した扱いとする。内部的には両フラグを `true` で記録する。

### 9.2 課金開始日情報（クライアント算出）

| ユーザー区分 | 契約開始日 | 初月課金 | 課金開始月 |
| --- | --- | --- | --- |
| 新規（`is_rejoining_user=false`） | 申込日 | 無料 | 翌月 1 日 |
| 再加入（`is_rejoining_user=true`） | 申込日 | 満額 | 申込月 |

- 再加入者判定ロジックは本書の対象外。サーバー側で判定して返却する想定（要確定）。
- クライアントは受領したフラグに従いメッセージ表示のみを行う。

## 10. 本番化要件

本書はモックアップ作成段階の要件として作成したが、最終的に本番環境構築を担う開発会社が実装に着手するために必要な情報を本セクションに集約する。

### 10.1 API 契約

想定エンドポイントと入出力。詳細スキーマは実装着手時に確定する。

| エンドポイント | 用途 | リクエスト | 主要レスポンス |
| --- | --- | --- | --- |
| `GET /api/users/me/profile` | 会員情報取得（引用モーダル表示用） | — | 200 `{ lastName, firstName, companyName, phone, zip, address, building, departmentName, positionName, email, is_premium_member, is_rejoining_user }` ／ 401 未認証 |
| `POST /api/premium/register` | プレミアムプラン登録 | §9.1 ペイロード | 200 `{ ok: true, contractId, billingStartAt }` ／ 400 バリデーション／ 409 既にプレミアム会員／ 5xx サーバエラー |
| `GET /api/address/zip/{code}` | 郵便番号住所検索（自社プロキシ想定） | — | 200 `{ prefecture, city, town }` ／ 404 該当なし |

- 登録 API は HTTPS 必須、ログイン認証済セッション必須、CSRF トークン付与。
- 二重送信防止のため `Idempotency-Key` ヘッダー付与を推奨（要確定）。
- 郵便番号検索は外部公開 API 直接呼出ではなく、自社プロキシ経由を原則とする（利用規約・負荷・可用性面）。

### 10.2 権限・認可

### 10.3 業務エラー時の挙動

| エラー種別 | ユーザー表示 | システム挙動 |
| --- | --- | --- |
| 登録バリデーションエラー（400） | Step 2 でエラーモーダル＋詳細文言。修正導線を提示 | 入力値保持。Step 1 へ戻れる |
| 既にプレミアム会員（409） | エラーモーダル「既にプレミアムプランに加入済みです。」 | ダッシュボードへ戻る導線を提示 |
| 住所検索 該当なし（404） | 郵便番号欄直下に「該当する住所が見つかりませんでした。」 | 住所欄は手入力で続行 |
| 住所検索 通信失敗（5xx・ネットワーク） | 郵便番号欄直下に「住所の取得に失敗しました。」 | 住所欄は手入力で続行 |
| 登録 通信失敗（5xx） | エラーモーダル「登録処理中にエラーが発生しました。もう一度お試しください。」 | 自動リトライなし。入力値保持、Step 2 に留まる |
| ネットワーク切断 | エラーモーダル「ネットワークに接続できません。」 | 自動リトライなし |

### 10.4 データ永続化

- 契約テーブル: `contract_id` / `user_id` / `plan`（premium）／ `status`（active/cancelled 等）／ `billing_start_at` / `contracted_at` / `created_at`。
- 請求先テーブル: §9.1 `billing` 準拠（`contract_id` との 1:1 または N:1）。
- 規約同意ログ: `user_id` / `contract_id` / `terms_version` / `privacy_version` / `consented_at` / `consent_ip`。法的エビデンスとして別テーブルで保管。
- 登録ペイロードは個人情報を含むためアプリケーションログには出力しない（§10.5）。

### 10.5 ログ・監査

- 操作ログ: 登録要求・登録完了・登録失敗。実行ユーザー ID・契約 ID・結果コードを記録。
- 監査ログ: 規約同意（利用規約バージョン・プライバシーポリシーバージョン・同意日時・IP）を長期保管。
- 失敗ログ: 登録 API の 4xx / 5xx 発生時刻・原因コードを記録。個人情報は出力しない。
- 保管期間は運用方針に従う（最長 1 年を想定、要確定）。

### 10.6 想定データ量

- 登録要求: 数件〜十数件 / 日を想定。同時並行登録は数件以下。
- 性能要件は特別な対策不要。住所検索 API の外部呼出は同時数件程度を想定。

### 10.7 対応ブラウザ・デバイス

- 想定対応: 最新版 Chrome / Edge / Safari / Firefox（直近 1 年以内のメジャーバージョン）。
- レスポンシブ対応: PC（横幅 ≥ 1024px）／タブレット／スマートフォン（横幅 ≥ 375px）。
- IE および非 Evergreen ブラウザは対象外。
