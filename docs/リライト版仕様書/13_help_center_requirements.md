## 1. 文書目的

- 本書は、ダッシュボード配下ヘルプセンターの要件を定義する。
- 本書を正本とする。
- 旧「本書が正」方針は破棄済み。
- `support.speed-ad.com` 配下の公開サポートサイトは別スコープ（§8）。

## 2. 画面概要

- SPEED AD ダッシュボード利用者（広告運用担当・管理者）の疑問を自己解決させ、問い合わせ件数を削減する機能。
- 2画面構成：
  - **トップ画面**（`help.html`）: ヒーロー検索、人気キーワード、おすすめ記事、Bento カテゴリグリッド、CTA。
  - **コンテンツ画面**（`help-content.html`）: URL クエリに応じてカテゴリ一覧・記事詳細・検索結果のいずれかを描画する単一ページ。
- データソースは `data/help_articles.json`（唯一）。クライアント完結でサーバー処理は行わない。
- 対応言語は日本語・英語。サイト内切替 UI で言語変更する。

## 3. 画面要件

### 3.1 トップ画面（`help.html`）

#### 3.1.1 ヒーロー検索フォーム
- 検索フォームをヒーローに配置する。送信先は `help-content.html?search=<keyword>`。
- キーワードは URL エンコードして渡す。
- サーバー送信は行わないため、サーバー化する場合のレート制限方針は別途定義する。

#### 3.1.2 検索サジェスト
- 入力中に逐次サジェストを表示する。
- **最小入力文字数**: 2 文字。
- **最大表示件数**: 6 件。
- **マッチ方式**: 記事タイトル（`question`）と本文（`answer`）の連結に対し、空白区切りトークンの AND 部分一致（小文字化のみ）を行う。
- 件数超過分は非表示とし、ランキング規則（タイトル前方一致優先等）は定義しない（JSON 定義順）。
- **0 件時**: 「`{keyword}` の記事が見つかりません。Enter キーで全文検索できます。」を表示し、ドロップダウンは開いたままにする。
- キーボード操作: 上下矢印・Home・End で候補選択、Enter で記事詳細へ遷移、Escape で閉じる。
- アクセシビリティ: ARIA `combobox` / `listbox` パターンで実装し、件数変化を `aria-live` で通知する。

#### 3.1.3 スティッキー検索バー
- ヒーローがビューポート上端を通過した時点で、画面上部に縮小版検索バーを固定表示する。
- 出現・消失はフェードで制御する（遷移 250ms）。
- SP / Tablet / Desktop で同一挙動とする。

#### 3.1.4 人気キーワードチップ
- ヒーロー下部に最大 5 件のチップを表示する。
- クリックで当該キーワードの検索結果ページへ遷移する。
- 現行は静的定義リスト。PV・検索ログからの動的算出は将来検討。

#### 3.1.5 おすすめ記事
- `isFeatured=true` の記事から最大 6 件を表示する。
- 並び順は JSON 定義順（`order` は現行未使用）。
- 6 件を超える `isFeatured` 記事がある場合は超過分を非表示とし、月次棚卸しで整理する運用とする。
- `published=true` のみ配信対象（§5.3）。
- レスポンシブ: SP 1 列 / Tablet 2 列 / Desktop 3 列。

#### 3.1.6 Bento カテゴリグリッド
- 全カテゴリを Bento 形式（異なるサイズのカードの格子レイアウト）で表示する。
- カテゴリ順は固定順で定義する。
- 各カードに記事プレビュー最大 2 件を表示する。**公開記事 0 件のカテゴリは非表示**。
- グリッド末尾に FAQ 専用カードを挿入し、`faq.html` へ誘導する。

#### 3.1.7 CTA セクション
- ページ下部に「お探しの答えは見つかりましたか？」セクションを配置する。
- リンク:
  - 「お問い合わせ（不具合報告・ご質問）」→ `bug-report.html`
  - 「よくある質問を見る」→ `faq.html`
- `bug-report.html` は不具合・一般問い合わせ兼用で、問い合わせ種別選択を必須とする。
- 「回答には通常 ○ 営業日いただいております」文言を表示する（日数は運用側定義）。

### 3.2 コンテンツ画面（`help-content.html`）

#### 3.2.1 URL クエリによる表示モード切替
優先順位（上が優先）:
1. `?search=<keyword>`: 検索結果一覧
2. `?article=<articleId>`: 記事詳細
3. `?category=<categoryId>`: カテゴリ内記事一覧
4. いずれも無い、または `search` が trim 後空: 404 相当の画面

#### 3.2.2 検索結果
- 空白（半角・全角）分割の AND 検索。連続空白は単一区切り。
- 対象はタイトルと本文の連結文字列。
- クォート囲みは未対応（通常トークン扱い）。
- ヒット箇所は `<mark>` でハイライトする（§5.1 の手順必須）。
- **0 件時**:
  - 「該当する記事は見つかりませんでした」を表示する。
  - 次アクション導線として、人気キーワードチップ・FAQ リンク・問い合わせリンクを提示する。

#### 3.2.3 カテゴリ内の絞り込み・ソート
- キーワード絞り込みはタイトル + サマリ（無ければ本文）の部分一致。
- ソート順:
  - `featured`（既定）: `isFeatured` 降順 → `updatedAt` 降順 → タイトル昇順 → 記事 ID 昇順。
  - `newest`: `updatedAt` 降順。
  - `title`: ロケール準拠のタイトル昇順。空タイトルは末尾。
- 公開記事が無い場合は「このカテゴリにはまだ記事がありません」を表示する。
- `published=true` のみ配信対象（§5.3）。

#### 3.2.4 パンくず
- 先頭に「ダッシュボード → ヘルプセンター」を常に表示する。
- モード別の末尾要素:
  - 検索結果: `検索結果: "<keyword>"`（リンクなし）
  - カテゴリ一覧: カテゴリ名（リンク）
  - 記事詳細: カテゴリ名（リンク）→ 記事タイトル（リンクなし）
- 生成失敗時は最終階層を「...」または空文字で代替し、描画は継続する。

#### 3.2.5 記事詳細
- 記事ヘッダーまたはメタ情報欄に `articleId` を表示し、サポート照合を可能とする。
- 記事本文はプレーンテキスト扱い。HTML タグを含む場合もエスケープしたうえで表示する（§5.1）。
- おすすめ記事セクション: `isFeatured=true` から現在記事を除外して最大 3 件を表示する。除外後 0 件時はセクション非表示。

#### 3.2.6 フィードバック UI
- 「この記事は参考になりましたか？」セクションに「はい」「いいえ」の 2 ボタンを配置する。
- **保存方針**: 投票は端末内の `localStorage` にのみ保存する。サーバー送信は行わない。
- UI に「端末内のみ保存。外部に送信されません。」旨を明示する。
- 投票済み（期限内）はボタンを表示せず「ご協力ありがとうございました。」のみ表示する。
- **「いいえ」投票後**: 「投票は記録されません。問い合わせはこちら（記事 ID を記載ください）」と、`bug-report.html` / `faq.html` への誘導を表示する。
- 複数タブ対応: `storage` イベント購読により、他タブでの投票を反映する。
- 容量超過時（`QuotaExceededError`）: コンソール警告 + 投票ボタン非表示 + 「フィードバックを保存できませんでした。」を表示する。
- 投票取消 UI は現時点対象外。

## 4. データ要件

### 4.1 データソース
- `data/help_articles.json` を唯一のデータソースとする。
- 構造: `{ categories: [{ id, name, questions: [{ id, question, answer, summary, isFeatured, updatedAt, order, videoUrl, published }] }] }`。
- 主要フィールド:
  - `question` / `answer` / `summary`: ローカライズ対象（記事タイトル・本文・要約）。
  - `isFeatured`: おすすめ対象（§3.1.5 / §3.2.5）。
  - `published`: 配信制御（§5.3、配信物からは除去）。
  - `order` / `updatedAt`: 並び順・更新日時。

### 4.2 欠落時の既定値
| フィールド | 既定値 |
| --- | --- |
| `isFeatured` | `false` |
| `updatedAt` | エポック（最古扱い） |
| `order` | 末尾 |
| `published` | `false`（配信対象外） |

### 4.3 想定データ規模
- 記事 200 件 / 合計 500 KB を上限の目安とする。
- 超過時はカテゴリ分割または遅延ロードを検討する。

## 5. セキュリティ要件（交渉不可）

### 5.1 XSS 対策（検索ハイライト手順）

検索ヒット箇所に `<mark>` を挿入する実装は、順序を誤ると XSS を招く。実装順序は以下に固定する（**順序変更禁止**）。

1. **表示対象テキスト（タイトル・本文）を HTML エスケープする**。`&`, `<`, `>`, `"`, `'` を対象とし、`escapeHtml()` を用いる。ユーザー入力や記事本文に `<script>` が含まれていても文字列として無害化することが目的。
2. **エスケープ済み文字列に対して、各トークンの正規表現でマッチ位置を特定する**。トークン側も `escapeRegex()` で正規表現特殊文字を無害化しておく。
3. **マッチ位置に `<mark>$&</mark>` を `String.replace()` で挿入する**。この `<mark>` は開発者が書いた固定タグなので HTML として解釈されてよい。
4. **生成済み HTML 文字列を `innerHTML` に代入する**。未エスケープ文字列の `innerHTML` への直接代入は禁止。

この順序（escapeHtml → `<mark>` 挿入）を変更する場合は、改訂履歴に変更箇所・変更理由・代替方式の安全性根拠を明記したうえで承認を得る。

### 5.2 URL パラメータのバリデーション
- `article` / `category` は許可リストパターン `^[A-Za-z0-9_-]{1,64}$` に一致する場合のみ受け付ける。不一致は 404。
- `article` / `category` が JSON 内の既知 ID に一致しない場合は 404。
- URL に組み込む `href` は URL コンストラクタで同一オリジン検証する。`javascript:` スキームは無効化する。
- `search` は長さ上限 200 字（UTF-16 code units）、制御文字（U+0000〜U+001F、U+007F〜U+009F）を除去する。

### 5.3 `published=true` のみ配信
- `published=false` または未定義の記事は全モードで除外する。
- **公開ビルドには `published=true` 記事のみを焼き込み、成果物 JSON から `published` フィールド自体を除去する**。
  - 理由: JSON に残存すれば HTTP 越しに全文取得されうる。フラグ名の存在自体が「公開保留記事あり」の情報漏洩となる。成果物から除去することを一次防御とし、アプリ側フィルタは二重防御とする。
- CI 検査で以下を必須とする:
  - `published=false` 記事が成果物に混入していないこと。
  - 成果物 JSON に `published` キーが一切存在しないこと（JSON Schema + grep の両検査）。
- `help_articles.json` はダッシュボードログイン済みセッションのみに配信する。未認証配信は禁止。

### 5.4 公開範囲・認証
- ダッシュボードログイン済みユーザーのみアクセス可能。未認証時はログイン画面へリダイレクトする。
- 社内運用手順や契約情報を含みうるため、公開ドメインへ露出させない。
- CSP（`default-src 'self'` を基本、外部は明示許可）、HSTS（`max-age=31536000; includeSubDomains`）をデプロイ時に付与する。

## 6. フィードバック localStorage スキーマ

### 6.1 キー名前空間
- キー名: `sa.help.feedback.v1`。
- 接頭辞は「プロダクト名 + 機能名 + バージョン」。他機能との衝突・上書きを避けるため。
- スキーマ更新時は `v2` に上げて共存可能とする。

### 6.2 スキーマ
```
{
  [articleId]: {
    v: 1,
    vote: "yes" | "no",
    at: "<ISO8601 UTC>"
  }
}
```

### 6.3 有効期限
- **180 日**（カレンダー日、UTC 基準）。
- 期限切れは「未投票」扱いに戻す。
- 記事更新後は「参考になった」判断が古くなるため、永久保存を避けて記事改善サイクルに追従させる。

### 6.4 バリデーションと移行
- 読み込み時は JSON パース後、`v` の存在・`vote` の値（`yes` / `no`）・`at` のフォーマットをバリデーションする。
- `at` に未来日時がある場合は端末時計改竄の可能性があるため、当該エントリは「未投票」扱いとする。
- 旧スキーマ `{ [articleId]: { vote } }`（旧キー `helpArticleFeedback`）からの移行は 1 回限りで実施する。`at` 欠落時は移行実行時刻で補完し、180 日カウントを起算する。

### 6.5 PII ポリシー
- 投票データに PII・自由記述テキストを含めない。
- サーバー送信化を行う場合は、利用目的公表・法的根拠・保存期間・削除請求・越境移転・PII 伏字化の整備を前提条件とする。

## 7. 多言語対応

- 対応言語: 日本語（ja）・英語（en）。
- サイト内の言語切替 UI で言語を変更する。
- ローカライズ対象: `question` / `answer` / `summary` / カテゴリ名。
- 非ローカライズ: `isFeatured` / `updatedAt` / `order` / `published`。
- **フォールバック順**: 現在言語 → 日本語 → キー名。
- 未翻訳時は日本語で代替表示し、コンソールに `[i18n warn] missing translation: {lang}/{articleId}` を出力する。
- 言語切替時はリアルタイムに再レンダリングする。`localStorage`（§6）を再読込して投票済み UI を保持し、スクロール位置・展開状態・検索結果はリセットする。

## 8. サポートサイト（`support.speed-ad.com`）分離計画との関係

- `support.speed-ad.com` 配下（`/faq/`, `/help/`, `/tutorial/`）はダッシュボード配下ヘルプセンターとは**別スコープ**（SPDAD2026-73 由来）。
- 本書の対象はダッシュボード配下のみ。公開サポートサイト側は `21_support_site_requirements` 等の別書を正本とする。
- 両者の関係:
  - ダッシュボード配下: 認証済みユーザー向け、運用手順・契約情報を含みうるため非公開。
  - サポートサイト配下: 公開ドメイン、外部検索エンジンからの流入を想定しないが、`robots` の `noindex, nofollow` を付与する運用。
- 共通アセット（CSS / JS / 画像）の運用、バージョン付きファイル名によるキャッシュ運用、CSP / HSTS などはサポートサイト側要件で規定する。
- 将来的にダッシュボード内ヘルプをサポートサイトへ移設する場合は、本書のセキュリティ要件（§5）・フィードバック仕様（§6）・多言語（§7）を移設先に適用する。

## 9. 非機能要件

### 9.1 パフォーマンス
- 主要操作（検索遷移・サジェスト表示・カテゴリ/記事切替・ソート・絞り込み）は p95 で 1 秒以内。
- 計測条件: データ 200 件、Moto G4 相当、4G（12 Mbps / 70 ms RTT）、キャッシュなし、各操作 5 回以上（冒頭 1 回除外）。

### 9.2 レスポンシブ・アクセシビリティ
- SP / Tablet / Desktop で表示崩れなし。
- WCAG 2.1 AA を目標。axe-core で critical / serious 違反ゼロを合格基準とする。
- キーボードのみで全操作可能。コントラスト比 4.5:1 以上。
- 検索サジェストは ARIA `combobox` / `listbox` パターン、`aria-live` で件数変化を通知する。

### 9.3 エラーハンドリング
- JSON 読み込み失敗時は「コンテンツの読み込みに失敗しました。」を表示する。
- エラー UI にリトライボタン、問い合わせ導線、エラー ID 表示 + コピーボタンを設ける。
- エラー ID 書式: `YYYYMMDD-HHMMSS-XXXX`（UTC、XXXX は大文字英数字 4 桁）。衝突は照合用途のため許容。
- URL パラメータが既知 ID に一致しない場合は 404 にフォールバックする。

### 9.4 対応環境
- 対応ブラウザ: Chrome / Firefox / Safari / Edge 最新 2 バージョン。
- 同時アクセス数は CDN 帯域依存。

### 9.5 ログ方針
- サーバー送信は行わず、`console.debug()` のみ。
- サーバー収集化時は §6.5 の前提条件を満たす。
- 対象: 検索実行・記事閲覧・フィードバック操作。投票者識別情報・セッション ID は出力しない。

## 10. Definition of Done

**機能要件**:
- [ ] サジェスト 2 文字以上で動作、0 件メッセージ表示
- [ ] スティッキーバーのフェード動作（250ms）
- [ ] おすすめ記事が `isFeatured=true` で最大 6 件
- [ ] Bento グリッドで公開記事 0 件カテゴリが非表示
- [ ] URL バリデーション（許可リスト・同一オリジン）
- [ ] `search` 200 字制限・制御文字除去
- [ ] XSS 対策手順（§5.1）をコードレビューで確認
- [ ] 記事詳細に記事 ID 表示
- [ ] フィードバック（`storage` 購読・Quota 対応・180 日期限・「いいえ」後誘導）
- [ ] localStorage キーが `sa.help.feedback.v1` に移行
- [ ] 成果物 JSON に `published=false` 記事混入なし、`published` キー自体が除去されていることを CI で検査
- [ ] `help_articles.json` が認証済みのみに配信

**非機能要件**:
- [ ] axe-core で critical / serious 違反ゼロ
- [ ] p95 1 秒以内を §9.1 条件で確認
- [ ] SP / Tablet / Desktop で表示崩れなし
- [ ] XSS ペイロード（`<script>alert(1)</script>` 等）で文字列として表示されることを確認
