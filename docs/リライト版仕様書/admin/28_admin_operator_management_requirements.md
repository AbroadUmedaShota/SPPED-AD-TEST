## 1. 文書目的

- SPEED AD の名刺データ化業務に従事するオペレーター（社内 アブロードスタッフ・社外委託先含む）のアカウントを管理する画面の要件を定義する。
- 既存モックの UI/UX に依存せず、`docs/画面設計/仕様/admin/` 配下の正本へ統合するためのリライト版として整理する。
- オペレーターの稼働実績・KPI 等の分析機能は本書の対象外とし、AD-10 実績管理の責務とする。
- 業務オペレーター向けの作業画面（BY-211_オペレーター入力画面・BY-221_照合画面・BY-231 業務用オペレーター管理）の機能は本書の対象外とする。

---

## 2. 画面概要

### 2.1 オペレーター管理一覧（AD-09）

管理者がオペレーターアカウントの検索・絞り込み・一覧閲覧を行う起点となる画面。本画面が AD 配下のオペレーター管理の単一エントリポイントであり、個別オペレーターの詳細確認と通常のアカウント操作は 2.2 のオペレーター詳細モーダルで行う。グループ（所属会社）の作成・編集は 2.3 のグループ管理モーダルで行う。実績指標の閲覧・分析は AD-10 実績管理へ遷移する。

### 2.2 オペレーター詳細モーダル

特定オペレーターの基本情報・所属・権限・対応可能言語・パスワード管理・操作ログ・ログイン履歴を確認し、通常のアカウント操作を行うモーダル。
- AD-09 の一覧行クリックで同画面内にオーバーレイ表示し、閉じると一覧の検索条件・ページ・スクロール位置を保持して戻る。
- モーダル内はタブまたは折りたたみセクションで領域を切り替え、AD-09 内で扱う機能へアクセスできる構成とする。
- 基本情報エリアにはオペレーター ID・氏名・メールアドレス・所属グループ・権限・対応可能言語・ステータスを配置する。
- パスワード管理エリアにはパスワード初期化メール送信と、管理者によるパスワード直接設定（Lv4 限定）を配置する。
- 履歴エリアにはログイン履歴、当該オペレーターに対する管理者操作ログ、最終ログイン日時を配置する。
- 実績情報は本モーダル内に表示せず、「AD-10 で実績を見る」リンクから AD-10 実績管理（当該オペレーターでフィルタした状態）へ遷移する。

### 2.3 グループ管理モーダル

オペレーターの所属グループ（会社単位）を作成・編集・削除し、メンバーを追加・削除するモーダル。
- AD-09 上部の「グループ管理」ボタンから開く。
- グループは「オレンジグループ」「アブロード本体」「外注先 A 社」のような所属会社/法人単位を指す。
- グループ作成・編集タブ構成で、グループ名の入力およびメンバー（オペレーター）の追加・削除を行う。

## 3. 機能要件

### 3.1 一覧・検索

#### 3.1.1 オペレーター一覧表示

- ページネーション付きのテーブル形式でオペレーターアカウントを一覧表示する。
- 1ページあたりの表示件数は 25 / 50 / 100 件から切り替え可能とする。
- 各行に表示する情報: 選択チェックボックス・氏名・メールアドレス・権限・所属グループ・最終ログイン日時・ステータス・行操作（詳細 / 削除）。
- ステータス（有効 / 無効 / 招待中）はラベル（色付きバッジ）で視覚的に区別する。
- 招待中の仮アカウントも本一覧に混在表示する（招待管理エリアを別建てしない）。
- 論理削除済みアカウントは通常一覧から除外する。
- 実績指標（総処理数・エラー率等）は本画面に表示しない。AD-10 実績管理で確認する。

#### 3.1.2 検索・フィルタリング

- 以下の条件で絞り込みを行う。複数条件の AND 検索とする。
  - キーワード（氏名・メールアドレス・オペレーター ID の部分一致）
  - 権限（Lv1 オペレーター / Lv2 オペレーター管理者 / Lv3 アブロードスタッフ / Lv4 アブロードマネージャー の複数チェック）
  - 所属グループ（単一選択）
  - 対応可能言語（単一選択）
  - ステータス（有効 / 無効 / 招待中 の複数チェック）
- 検索条件のリセット機能を提供する。
- Lv2 がアクセスした場合は、フィルタ・検索条件にかかわらず**自グループ内のオペレーターのみ**を返却する（§4.4 参照）。

#### 3.1.3 ソート

- 以下の列でソート可能とする。昇順 / 降順 / 解除の3状態を切り替える。
  - 氏名 / メール / 権限 / 所属グループ / 最終ログイン日時 / ステータス
- 総処理数によるソートは AD-09 では提供しない（AD-10 で提供）。

### 3.2 オペレーター操作

#### 3.2.1 詳細表示

- 一覧の行から対象オペレーターを選択し、同画面上にオペレーター詳細モーダルを開く。別画面遷移は伴わない。
- モーダルを閉じると一覧の検索条件・ページ・スクロール位置を保持したまま戻る。
- モーダル表示中もブラウザの戻る/進む操作で開閉できる URL 同期（クエリパラメータ等）を提供する。

#### 3.2.2 基本情報の保存

- オペレーター詳細モーダルの「変更を保存」押下時、基本情報・対応可能言語の変更をまとめて API 送信する。
- 基本情報は氏名・メール・所属グループ・権限・ステータス・対応可能言語を対象とする。
- 保存時はサーバー側で入力値、権限、競合状態、テナント分離（§4.4）を再検証し、成功時に管理者操作ログへ差分を記録する。
- 権限変更は Lv4 のみ実行できる。詳細は §7.2 を参照。
- 所属グループ変更は Lv4 のみ実行できる。Lv3 は閲覧のみ、Lv2 は他グループへの移動を伴うため不可とする。
- 処理実績・操作ログ・ログイン履歴に紐づく所属グループは、各レコード発生時点の `group_id` をスナップショットとして保持する。オペレーターの現在の所属を変更しても、過去のレコードに記録された所属は変わらない。
- AD-07 / AD-08 / AD-10 等での所属グループ別集計は、本節の方針に従い各レコード発生時点の `group_id` に基づいて行う。

#### 3.2.3 ステータス管理（有効化・無効化）

- 「有効」のオペレーターを「無効」に変更する。「無効」のオペレーターを「有効」に戻す。
- 退職・契約終了・長期休暇など、稼働させないケースは全て「無効化」で扱う。退職と休止を別ステータスとして区別しない。
- 実行前に確認ダイアログを表示する。
- 実行結果は管理者操作ログに記録する（操作者・日時）。
- ステータス変更はサーバー側で現在状態と必要権限を再検証し、UI の表示/非表示だけに依存しない。
- 無効化時は対象アカウントの既存セッションと refresh token を失効させ、再有効化後も再認証を必須とする。

#### 3.2.4 オペレーター削除

- 無効化済みオペレーターに対してのみ削除操作を提供する。有効・招待中オペレーターへの削除は不可とする。
- オペレーター削除は Lv4 のみ実行できる。
- 削除は物理削除ではなく論理削除（deleted_at）とし、過去の処理実績・操作ログ・ログイン履歴との外部キーは維持する。
- 実行前に「この操作は取り消せません」の確認ダイアログを表示し、オペレーター氏名の入力による二重確認を求める。
- 実行結果は管理者操作ログに記録する。
- 削除可否は ステータスだけでなく、サーバー側の整合条件で判定する。
- 削除実行時は対象アカウントの既存セッション、未使用トークンをサーバー側で失効する。
- 論理削除済みオペレーターは通常一覧から除外する。復元操作は AD-09 では提供しない。

#### 3.2.5 パスワード初期化メール送信

- 対象オペレーターのメールアドレス宛にパスワード再設定メールを送信する。
- Lv4 / Lv3 が実行できる。Lv2 は自グループ内の Lv1 に対してのみ実行できる。
- 送信完了後に管理者操作ログおよびメール送信履歴に記録する。

#### 3.2.6 管理者によるパスワード直接設定

- 管理者が新パスワードを直接入力して即時確定する操作を提供する。
- **Lv4 のみ実行できる。** Lv3 / Lv2 は実行不可。
- 用途は「リセットメールが届かないオペレーターへの緊急対応」に限定する。
- 実行時は理由入力を必須とする。
- 新パスワードはサーバー側で複雑度要件を再検証する。
- 設定完了後、対象オペレーター本人へ「パスワードが管理者により変更されました」通知メールを送信する（本人検知用）。
- 実行結果は管理者操作ログに記録する。記録項目は操作者 ID・対象オペレーター ID・操作日時・理由とし、**パスワード値そのものは記録しない**。
- 設定実行時は対象アカウントの既存セッションをサーバー側で失効する。

#### 3.2.7 自己操作の禁止

- 管理者は自分自身（操作者本人）のアカウントを対象として、以下の操作を AD-09 上で実行できない。
  - 権限変更・所属グループ変更・ステータス変更（有効化・無効化）・オペレーター削除・パスワード直接設定
- API 側で `target.operator_id == request.user.operator_id` の場合に 403 を返す。
- アブロードマネージャー（Lv4）の在籍数は1名以上を維持する。最後の Lv4 を無効化または論理削除する操作は、自己操作禁止に加えてサーバー側で在籍数チェックを行い拒否する。
- 一括操作（権限変更・所属変更・ステータス変更・削除）の対象に自分自身が含まれる場合、当該対象のみを `failed[]` に分類して `self_target` 理由で返却し、他の対象は処理を続行する。
- 上記制限は §7.2 のロール別権限・§4.4 のテナント分離より優先して適用する。

### 3.3 招待管理

#### 3.3.1 新規オペレーター招待

- メールアドレス・氏名・所属グループ・権限を入力して招待メールを送信する。
- 招待メールには本登録用の有効期限付き URL（72時間有効）を含める。
- 招待と同時に仮アカウントレコードを生成し、「招待中」ステータスで管理する。
- 招待状態は「招待中 / 登録済み / 期限切れ / 取消済み」で管理する。
- 招待発行前に、入力されたメールアドレスに対応する**未削除のオペレーターアカウント**（ステータスが有効・無効・招待中のいずれか）が存在するかを確認する。存在する場合は新規招待を作成せず、既存アカウントの再有効化または招待再送の導線を案内する。論理削除済みアカウントのみが存在する場合は、同一メールアドレスでの新規招待を許可する（退職後の再雇用ケースを想定）。
- 同一メールアドレスの未登録招待（招待中・期限切れ）が存在する場合、新規招待は作成せず、既存招待の再送導線へ誘導する。
- 招待 URL から本登録（パスワード設定）が完了した時点で仮アカウントを正規アカウントへ昇格し、招待状態を「登録済み」、アカウントステータスを「有効」に変更する。
- 招待 URL はワンタイム性を持ち、登録完了後または失効後の再利用を許可しない。
- 招待は Lv4 のみ実行できる。Lv2/Lv3 は招待実行不可とする。

#### 3.3.2 招待中アカウントの操作

- 招待中の仮アカウントは AD-09 一覧に通常表示する。
- 招待中の行に対しては、招待再送・招待キャンセル・有効期限の確認が可能とする。
- 72時間経過した招待は自動的に「期限切れ」へ遷移する。
- 招待再送は Lv4 / Lv3 が実行できる。再送時は旧トークンを即時失効し、新しいトークンと有効期限を発行する。
- 招待キャンセルは Lv4 のみ実行できる。取消済みの旧 URL は利用できない。
- 登録済みの招待は管理用一覧の通常表示から除外し、監査用途の履歴としてのみ参照する。

### 3.4 一括操作

#### 3.4.1 一括選択

- 一覧テーブルのチェックボックスで複数オペレーターを選択できる。
- 全選択 / 全解除を提供する。
- 1件以上選択された状態で「一括操作」エリアが表示される。
- 一括操作の対象範囲は現在の検索・フィルタ条件に合致する全件ではなく、**ユーザーがチェックボックスで明示的に選択した行のみ**とする。

#### 3.4.2 一括権限変更

- 選択されたオペレーターに新しい権限を一括適用する。
- Lv4 のみ実行できる。
- 実行前に「対象件数・変更後権限」を表示する確認ダイアログを出す。
- 実行結果は対象件数分の管理者操作ログを記録する。

#### 3.4.3 一括所属変更

- 選択されたオペレーターを別グループへ一括移動する。
- Lv4 のみ実行できる。
- 実行前に「対象件数・変更後グループ」を表示する確認ダイアログを出す。
- 実行結果は対象件数分の管理者操作ログを記録する。

#### 3.4.4 一括ステータス変更

- 選択されたオペレーターのステータス（有効 / 無効）を一括変更する。
- Lv4 / Lv3 が実行できる。Lv2 は自グループ内の Lv1 のみが選択された場合に限り実行できる。
- 実行前に「対象件数・変更後ステータス」を表示する確認ダイアログを出す。
- 無効化に伴う既存セッション失効は §3.2.3 と同様に行う。
- 実行結果は対象件数分の管理者操作ログを記録する。

#### 3.4.5 一括削除

- 選択された無効化済みオペレーターを一括で論理削除する。
- Lv4 のみ実行できる。
- 有効・招待中のオペレーターが選択に含まれている場合、API 側で拒否し、対象外件数を表示する。
- 実行前に「対象件数」を表示し、文字列入力（例: 「削除」と入力）による二重確認を求める。
- 実行結果は対象件数分の管理者操作ログを記録する。

---

### 3.5 グループ管理

#### 3.5.1 グループ作成

- グループ管理モーダルから新規グループを作成する。
- 入力項目: グループ名のみ。
- 同名グループの重複を禁止し、サーバー側で大文字小文字を問わず一意性を検証する。
- グループ作成は Lv4 のみ実行できる。
- 実行結果は管理者操作ログに記録する。

#### 3.5.2 グループ編集

- 既存グループを選択してグループ名を変更できる。
- グループ編集は Lv4 のみ実行できる。
- 名称変更時、所属メンバーの権限・状態は変更しない。
- 実行結果は管理者操作ログに記録する。

#### 3.5.3 メンバー追加・削除

- 選択中のグループに対し、未所属のオペレーターを追加・既所属のオペレーターを削除（未所属化）できる。
- 操作は Lv4 のみ実行できる。
- メンバー削除は §3.2.4 のオペレーター削除とは異なり、グループからの離脱のみを意味する（オペレーター自体は削除されない）。
- 実行結果は管理者操作ログに記録する。

#### 3.5.4 グループ削除

- 既存グループを削除する。
- グループ削除は Lv4 のみ実行できる。
- **削除前提条件**: 当該グループに所属するメンバー（有効・無効・招待中を問わず）が0名でない限り、削除を拒否する。事前に全メンバーを別グループへ移動するか未所属化する必要がある。
- 実行前に「この操作は取り消せません」の確認ダイアログを表示する。
- 実行結果は管理者操作ログに記録する。

---

### 3.6 履歴・ログ

#### 3.6.1 更新ログ（画面横断・サイドバー）

- AD-09 上で実行された全管理者操作を時系列でタイムライン表示する。
- 表示項目: 操作種別・操作者・対象オペレーター名・メッセージ・日時。
- 種別: 新規招待・招待再送・招待キャンセル・権限変更・所属変更・ステータス変更・削除・パスワード初期化メール送信・パスワード直接設定・グループ作成・グループ編集・グループ削除・メンバー追加・メンバー削除・一括操作。
- 種別フィルタを提供する。
- Lv2 がアクセスした場合は、自グループに関わる操作ログのみ表示する。
- ログ自体の編集・削除は不可とする。

#### 3.6.2 操作ログ（オペレーター詳細モーダル内）

- オペレーター詳細モーダル内に、そのオペレーターに対して管理者が行った操作の履歴を表示する。
- 表示項目: 操作日時・操作種別・操作者（管理者名）・変更前の値・変更後の値・理由（入力された場合）。
- 操作種別と期間で絞り込めるフィルタを提供する。
- ログ自体の編集・削除は不可とする。
- §3.6.1 と同一の管理者操作ログテーブルを対象オペレーター ID で絞り込んで取得する。

#### 3.6.3 ログイン履歴

- オペレーター詳細モーダルに当該オペレーターのログイン履歴を表示する。
- 表示項目: ログイン日時・IPアドレス・デバイス種別（デスクトップ / モバイル）・ブラウザ・ログイン結果（成功 / 失敗）。
- 最新50件を表示し、それ以上はページネーションまたは期間フィルタで参照できる。

### 3.7 他画面連携

#### 3.7.1 AD-10 実績管理への導線

- オペレーター詳細モーダルに「AD-10 で実績を見る」リンクを設置する。
- リンク先は AD-10 実績管理画面を当該オペレーターでフィルタした状態で開く。
- AD-09 内には実績指標（総処理数・エラー率・報酬額・納期遵守率・グラフ等）を表示しない。

## 4. 状態と挙動

### 4.1 ステータスに応じた操作制限

AD-09 では以下の状態を扱う。

- ステータス: 有効 / 無効 / 招待中
- 招待状態: 招待中 / 登録済み / 期限切れ / 取消済み（招待中ステータスの内訳）
- 削除状態: 未削除 / 論理削除済み

| ステータス | 利用可能な操作                                                       | 不可の操作                            |
| ---------- | -------------------------------------------------------------------- | ------------------------------------- |
| 有効       | 詳細表示・無効化・権限変更・所属変更・パスワード初期化メール・パスワード直接設定 | 削除                                  |
| 無効       | 詳細表示・有効化・権限変更・所属変更・削除・パスワード初期化メール・パスワード直接設定 | -                                     |
| 招待中     | 詳細表示・招待再送・招待キャンセル                                  | 権限変更・所属変更・有効化・無効化・削除・パスワード操作（本登録完了前のため） |

上表はステータス上の操作可否を示す。実行可否は §7.2 のロール別権限と §4.4 のテナント分離も合わせて判定する。

ステータスはサーバー側で再検証し、直接 URL アクセスや API 直叩きによる不可操作を拒否する。

### 4.2 招待状態に応じた操作制限

| 招待状態 | 利用可能な操作 | 不可の操作 |
| -------- | -------------- | ---------- |
| 招待中   | 招待再送・招待キャンセル・履歴参照 | 新規招待の重複作成 |
| 登録済み | 監査用途の履歴参照 | 招待再送・招待キャンセル |
| 期限切れ | 招待再送・履歴参照 | 旧URLの利用・同一メールでの新規招待重複作成 |
| 取消済み | 履歴参照 | 招待再送・旧URLの利用 |

同一メールアドレスに招待中または期限切れの未登録招待が存在する場合は、新規招待ではなく既存招待の再送またはキャンセルを案内する。

### 4.3 削除状態に応じた操作制限

| 削除状態     | 一覧表示                                     | 利用可能な操作 | 不可の操作 |
| ------------ | -------------------------------------------- | -------------- | ---------- |
| 未削除       | 通常表示                                     | ステータスに応じた操作 | なし |
| 論理削除済み | 通常一覧から除外し、監査用途の履歴でのみ参照 | 監査ログ参照   | 復元・編集・パスワード操作・グループ移動 |

論理削除済みオペレーターを通常一覧へ戻す復元操作は AD-09 では提供しない。

### 4.4 テナント分離（グループ越境制限）

AD-09 ではグループ（所属会社/法人）をテナント単位として扱い、Lv2 のアクセス範囲をグループ内に限定する。

| ロール | アクセス可能なグループ範囲 | 補足 |
| ------ | -------------------------- | ---- |
| Lv4 アブロードマネージャー | 全グループ | 制限なし |
| Lv3 アブロードスタッフ | 全グループ | 制限なし |
| Lv2 オペレーター管理者 | **自グループのみ** | 一覧取得・詳細取得・操作の全てで自グループ外のオペレーターを返却・操作不可 |
| Lv1 オペレーター | アクセス不可 | AD-09 自体にアクセスできない |

Lv2 がアクセスした場合のサーバー側挙動:
- 一覧取得 API は `WHERE group_id = request.user.group_id` を強制する。
- 詳細取得 API は対象オペレーターの group_id が一致しない場合、404 を返す（存在を漏らさない）。
- 更新・削除 API は対象オペレーターの group_id が一致しない場合、403 を返す。
- グループ管理モーダルは Lv2 では使用不可とする（グループ作成・編集・削除は全て Lv4 のみ）。

権限変更・所属グループ変更そのものを Lv2 が実行できる経路は §7.2 に従い提供しない。本節 §4.4 はテナント越境ガード（Lv2 が自グループ外のオペレーターへアクセス・操作することの防止）を主目的とし、Lv2 自身の操作許可リストは §7.2 を一次情報源とする。

> **注記**: 本節および §7.2 の Lv2 アクセス権限はレビュー時に再確認する暫定値とする。最終的に Lv2 のアクセスを認めない結論となった場合、本節の Lv2 関連記述は削除する。

## 6. 非機能要件

- 一覧の検索レスポンスは2秒以内を目標とする。
- ページネーションは1ページ最大100件とし、全件数を表示する。
- 更新・削除・送信など副作用を伴う API は Idempotency-Key により多重実行を防止する。
- 一括操作は同期的に処理する場合、最大100件までを上限とする。それを超える場合はバックグラウンド処理とし、完了通知を行う。

### 7.2 権限・認可

AD-09 で扱うロールは以下とする。

| ロール                              | AD-09 での扱い                                                |
| ----------------------------------- | ------------------------------------------------------------- |
| Lv4 アブロードマネージャー（authority=4） | フルアクセス、全グループ横断                                   |
| Lv3 アブロードスタッフ（authority=3）  | 閲覧と日常運用、全グループ横断。発行・権限変更・所属変更・パスワード直接設定・グループ管理は不可 |
| Lv2 オペレーター管理者（authority=2） | 自グループ内の Lv1 のみアクセス可（テナント分離）。詳細は §4.4 |
| Lv1 オペレーター（authority=1）     | AD-09 アクセス不可                                            |

> **注記**: Lv2 のアクセスはレビュー時に再確認する暫定値。最終的に Lv2 を AD-09 スコープ外とする場合は本表とテナント分離節を改訂する。

| 操作                                            | Lv4 | Lv3 | Lv2 | 備考 |
| ----------------------------------------------- | --- | --- | --- | ---- |
| オペレーター一覧・詳細の閲覧                    | 可  | 可  | 自グループ内のみ | Lv2 は API レベルで他グループを返却しない |
| 基本情報（氏名・メール）の更新                  | 可  | 可  | 自グループ内のみ |  |
| 対応可能言語の更新                              | 可  | 可  | 自グループ内のみ |  |
| 権限変更                                        | 可  | 不可 | 不可 | 自己昇格防止のため Lv2 不可 |
| 所属グループ変更                                | 可  | 不可 | 不可 | グループ越境を伴うため Lv2 不可 |
| ステータス変更（有効化・無効化）                | 可  | 可  | 自グループ内のみ | Lv2 は配下 Lv1 のみ |
| オペレーター削除（論理削除）                    | 可  | 不可 | 不可 | 不可逆操作のため Lv4 のみ |
| 新規オペレーター招待                            | 可  | 不可 | 不可 | アカウント発行に関わるため |
| 招待再送                                        | 可  | 可  | 自グループ内のみ | 旧トークン失効・ログ必須 |
| 招待キャンセル                                  | 可  | 不可 | 不可 | アカウント発行フローの停止に関わるため |
| パスワード初期化メール送信                      | 可  | 可  | 自グループ内のみ |  |
| パスワードの管理者直接設定                      | 可  | 不可 | 不可 | なりすましリスクのため Lv4 限定。理由入力・本人通知・操作ログ必須 |
| 一括権限変更                                    | 可  | 不可 | 不可 |  |
| 一括所属変更                                    | 可  | 不可 | 不可 |  |
| 一括ステータス変更                              | 可  | 可  | 自グループ内 Lv1 のみ |  |
| 一括削除                                        | 可  | 不可 | 不可 | 二重確認必須 |
| グループ作成・編集・削除                        | 可  | 不可 | 不可 | グループ削除はメンバー残存時拒否 |
| メンバー追加・削除                              | 可  | 不可 | 不可 |  |
| 更新ログ（横断）の閲覧                          | 可  | 可  | 自グループ操作のみ表示 |  |
| 操作ログ（個別）の閲覧                          | 可  | 可  | 自グループ内のみ |  |
| 管理者操作ログの削除・編集                      | 不可 | 不可 | 不可 | 全管理者に対して禁止 |

権限判定は画面表示だけでなく API 側でも再検証し、直接 URL アクセスや API 直叩きによる操作を拒否する。テナント分離も API 側で強制する。

### 7.3 データ永続化

- オペレーターテーブルに論理削除フラグ（`deleted_at`）を持つ。
- 管理者操作ログは専用テーブルで管理し、外部キー制約でオペレーターテーブル・操作者（管理者）テーブルと紐付ける。
- 招待レコードはトークン・有効期限・ステータス・失効日時を持つ専用テーブルで管理する。
- 招待再送時は旧トークンを失効済みとして保存し、最新トークンのみ有効にする。
- グループは専用テーブルで管理し、オペレーターテーブルから group_id で参照する。
- 対応可能言語は専用マスタテーブル（id, name, sort_order, deleted_at）で管理し、オペレーターには中間テーブルで多対多関連を持つ。言語マスタの管理画面は AD-09 では提供せず、開発側で直接更新する。

### 7.4 ログ・監査

- 以下の操作はすべて管理者操作ログテーブルに記録する。
  - 新規招待・招待再送・招待キャンセル
  - 権限変更・所属変更・ステータス変更
  - パスワード初期化メール送信・パスワード直接設定
  - オペレーター削除
  - グループ作成・編集・削除・メンバー追加・削除
  - 一括操作（操作種別ごとに対象件数分のレコードを生成）
- ログの記録項目: 操作種別・対象オペレーター ID・操作者 ID・操作日時・変更前値・変更後値・理由（入力された場合）。
- パスワード直接設定では、変更前後のパスワード値そのものは記録しない（メタ情報のみ）。
- 管理者操作ログは改ざん不可とし、編集・削除不可とする。
- 管理者操作ログの保持期間は、監査要件と個人情報保護方針に基づき本番化前に確定する。

### 7.5 想定データ量

- 管理対象オペレーター数: 数百〜数千件規模を想定。
- グループ数: 数十件規模を想定。
- ログイン履歴の保持期間: 直近1年分を表示対象とし、それ以上はアーカイブ。

### 7.6 受け入れ基準

- [ ] 検索条件の AND 組み合わせで正しく絞り込まれる
- [ ] 一覧テーブルは氏名・メール・権限・所属グループ・最終ログイン日時・ステータス・操作の7列で表示される（総処理数列が存在しない）
- [ ] ソート対象に総処理数が含まれない
- [ ] 招待中アカウントは AD-09 一覧に通常表示される
- [ ] 招待再送後に旧 URL が失効し、新 URL のみ利用できる
- [ ] 同一メールの未登録招待がある場合、新規招待は作成されず再送またはキャンセル導線が表示される
- [ ] 新規招待・招待キャンセル・権限変更・所属変更・オペレーター削除・パスワード直接設定・グループ管理は Lv4 のみ実行できる
- [ ] パスワード直接設定実行時に本人へ通知メールが送信され、操作ログには値そのものが記録されない
- [ ] パスワード初期化メール送信・招待再送・ステータス変更は Lv4/Lv3 が実行でき、Lv2 は自グループ内のオペレーターに対してのみ実行できる
- [ ] Lv2 が AD-09 にアクセスすると自グループ外のオペレーターは API レベルで返却されない
- [ ] Lv2 がオペレーターの権限を Lv2 以上に変更しようとすると API が拒否する
- [ ] Lv2 がオペレーターの所属グループを変更しようとすると API が拒否する
- [ ] 無効化以外のステータスのオペレーターに削除を実行すると API が拒否する
- [ ] 削除はオペレーター氏名の入力による二重確認、一括削除は文字列入力による二重確認が必要になる
- [ ] 無効化時に既存セッションが失効し、再有効化後も再認証が必要になる
- [ ] 論理削除済みオペレーターは通常一覧から除外され、復元操作は AD-09 で提供されない
- [ ] グループにメンバーが残存している場合、グループ削除が API レベルで拒否される
- [ ] グループ名の重複は大文字小文字を問わず作成・編集時に拒否される
- [ ] 更新ログ（サイドバー）と操作ログ（詳細モーダル）が同一データソースから取得され、フィルタ条件のみが異なる
- [ ] 対応可能言語の選択肢は DB マスタから取得される（ハードコードされていない）
- [ ] AD-09 上にオペレーターの実績指標が一切表示されず、AD-10 への導線リンクのみ存在する
- [ ] 全管理者操作がログに記録され、変更前後の値を確認できる
- [ ] 管理者操作ログが管理者自身でも削除・編集できない
- [ ] 管理者が自分自身に対して権限変更・所属変更・ステータス変更・削除・パスワード直接設定を実行しようとすると API が拒否する
- [ ] 最後のアブロードマネージャー（Lv4）を無効化または削除しようとすると API が拒否する
- [ ] 一括操作の対象に自分自身が含まれた場合、当該対象のみ `failed[]` に `self_target` 理由で返却され、他は処理が続行される
- [ ] 招待発行時、入力メールに対応する未削除アカウント（有効/無効/招待中）が存在すれば新規招待が作成されず、再有効化または招待再送導線が案内される
- [ ] 論理削除済みアカウントのみが存在するメールアドレスに対する新規招待は許可される
- [ ] オペレーターの所属グループを変更しても、過去の処理実績・操作ログ・ログイン履歴に記録された `group_id` は変わらない
- [ ] AD-07 / AD-08 / AD-10 の所属グループ別集計は、各レコード発生時点の `group_id` に基づいて行われる

---

## 8. 範囲外

本書の対象に含めないもの。

- オペレーターの稼働実績・KPI・報酬計算・グラフ表示（AD-10 実績管理の責務）
- オペレーター向け作業画面（BY-211/BY-212/BY-213 入力画面、BY-221/BY-222/BY-223 照合画面）
- オペレーター作業画面側の管理機能（BY-231 業務用オペレーター管理は本書のスコープ統合により本番化時に廃止または機能限定とする方針。詳細は本番化前に確定）
- 対応可能言語マスタの管理画面（DB 直接更新で運用）
- グループ属性（契約形態・連絡先・SLA 等）の管理。本書では「グループ名」のみを管理し、追加属性は本番化前に確定する。
- データ入力タスク・照合タスクの割り当て管理（AD-07 / AD-08 / AD-14 の責務）
