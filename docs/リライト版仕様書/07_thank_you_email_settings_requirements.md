## 1. 文書目的

- 本書は、お礼メール設定画面および送信実行画面の要件を定義する。
- 本書を正本とする。
- 旧「本書が正」方針は破棄済み。

## 2. 画面概要

- アンケート単位でお礼メールの送信方法、メール内容（テンプレート／件名／本文／差し込み変数）、送信対象者、クーポン適用を設定し、送信実行まで行う画面。
- 想定利用者はアンケート運用管理者（広告運用担当・管理者）。会期終了後に受信者リストを確認し、送信を実行する運用を想定する。
- 画面は左側の設定・プレビュー・対象者リストエリアと、右側の料金見積サマリーエリアで構成する。
- アンケート基本情報、現在の設定値、受信者データ、最新の見積、クーポン適用結果をサーバーから取得して初期表示する。
- 画面は会期進行段階に応じた 5 つの状態を持ち、UI が段階的に切り替わる（§4）。

## 3. 画面要件

### 3.1 上部情報
- パンくずと画面タイトル「アンケート『[名]』のお礼メール設定」を表示する。
- アンケート基本情報を表示する: アンケート名、アンケートID、会期期間。
- 送信期間を過ぎている場合は上部に期限切れバナーを表示する。

### 3.2 左カラム
- **送信設定カード**: 送信方法を単一選択する。
  - `手動送信` / `自動送信` / `送信しない` の 3 択。既定値は「手動送信」。
- **メール内容設定カード**
  - テンプレート選択: 標準／特別などの定型文を選択できる。選択で件名と本文が置換される。
  - 件名入力: 最大 255 文字。文字数カウンタを表示する。
  - 本文入力: 複数行入力。
  - 差し込み変数パネル: 開閉可能で、変数バッジをクリックすると直前にフォーカスしていた入力欄のキャレット位置に挿入する。
- **リアルタイムプレビューカード**: 件名と本文に差し込み変数を解決した結果を表示する。未設定変数がある場合は警告を表示する。
- **送信対象者リストカード**
  - 受信者一覧（会社名／部署／氏名／メールアドレス／ステータス）を表形式で表示する。
  - 行ごとのチェックボックスで送信対象の有効／無効を切り替え、ヘッダで全選択／全解除する。
  - ステータスは `送信待ち` / `送信済み` / `失敗` / `除外` / `データ化中` を扱う。
  - ページネーション: 1 ページあたりの件数（20／50／100）、現在範囲、ページ送りを提供する。
  - 全体ステータスメッセージや件数条件メッセージを状態に応じて表示する。
- **アクションボタン群**: `設定を保存` / `お礼メールを送信` を配置する。送信ボタンは送信可能状態でのみ活性化する。

### 3.3 右カラム
- 料金見積もりサマリーを表示する。
  - ご請求見込み金額（税込）
  - 送信対象者数
  - 内訳（対象者数 × 単価、無料枠適用、クーポン割引、消費税、合計）
  - 無料枠適用通数・課金対象通数
  - 料金注記（「100 通まで無料」「101 通目以降 ¥1/通」「税込」）
- クーポン適用エリアを表示する。
  - 未適用時はコード入力と検証導線。
  - 適用中はクーポン内容と解除導線。他画面由来の共有クーポンは識別表示する。

### 3.4 モーダル・補助導線
- 送信確認モーダル（対象者数・合計金額を再掲、キャンセル／送信実行）
- 期限切れバナー
- 共通の未保存離脱確認モーダル（変更がある場合）

## 4. 状態と挙動

### 4.1 画面状態

画面は以下 5 状態を持ち、会期進行に応じて遷移する。現行モックではシナリオセレクタによる手動切替のみ提供し、状態遷移の自動判定ロジックは未実装（§10）。

| 状態 | 条件 | 主な振る舞い |
|------|------|------------|
| `before_or_during_event` | 会期前・会期中 | 設定は編集可。対象者リストは非表示または「会期終了後に表示」メッセージ。送信ボタンは非活性。 |
| `after_event_processing` | 会期終了後、名刺データ化処理中 | 対象者リストに暫定件数を表示。全体ステータスに「一部未確定」旨のメッセージ。送信ボタンは非活性。 |
| `after_event_ready` | データ化完了、送信待機 | 対象者リスト確定。未設定変数があれば警告表示。送信ボタン活性。 |
| `after_event_sent` | 送信完了 | 対象者リストに送信結果（送信済み／失敗／除外）を行背景色で表示。入力欄は非活性。 |
| `period_expired` | 送信期限経過 | 期限切れバナー表示。対象者リストはグレーアウト。送信ボタン非活性。 |

### 4.2 初期表示
- 画面表示時に以下を取得する。
  - アンケート基本情報
  - 既存のお礼メール設定（送信方法、テンプレート、件名、本文、対象者選択状態）
  - 受信者データ
  - 適用中クーポン
  - 現在の見積
- 取得失敗時はエラーメッセージと再試行導線を表示する。
- クーポンは画面間共有キー（`sharedCoupon_{surveyId}`）を参照して復元する（§6）。

### 4.3 入力変更時
- 送信方法、テンプレート、件名、本文、対象者選択、クーポン変更時はプレビューと見積を再計算する。
- テンプレート変更は件名・本文を上書きする。既入力内容の扱いは設計課題とする。
- 差し込み変数挿入は直前フォーカス先（件名または本文）のキャレット位置に挿入する。

### 4.4 送信対象者操作
- 行チェックボックスで送信対象の有効／無効を切替可能。対象者数の変動は右カラムの見積に即時反映する。
- 全選択チェックで現ページ／全件の一括切替を行う。
- ページ送り・件数変更で一覧を再描画する。

### 4.5 クーポン
- クーポンコード検証はサーバーに問い合わせる。
- 成功時は割引内容とメッセージを返し、見積に反映する。
- 失敗時は入力欄に紐づくエラーを表示する。
- 解除時は確認後にサーバー側の適用状態を更新する。
- 名刺データ化設定画面と同じ共有キーでクーポンを共有する（§6）。

### 4.6 保存
- 保存時は画面を一時的に操作不可とし、二重送信を防止する。
- `送信しない` 選択時は「お礼メール機能無効」として保存する。
- 保存成功時は最新状態を反映する。
- 保存失敗時は原因に応じたメッセージを表示し、入力値は保持する。

### 4.7 送信実行
- 送信ボタン押下で送信確認モーダルを表示する。対象者数と合計金額を再提示する。
- 送信実行時は画面全体を操作不可とし、完了まで進捗を表示する。
- 送信結果（送信済み／失敗の件数）を行ごとのステータスに反映し、画面状態を `after_event_sent` に遷移する。
- 送信失敗時は原因メッセージと再試行導線を提供する。
- 実サーバへの送信は現行モック未実装（§10）。

## 5. 料金計算・送信条件

### 5.1 料金計算入力
- 送信対象者数（チェック有効件数）
- 適用クーポン
- 無料枠・単価・税率は固定値として扱う

### 5.2 料金計算ルール
- **無料枠**: 100 通まで無料。
- **単価**: 101 通目以降 ¥1／通。
- **消費税**: 税込 10%。
- **合計（税込）** = max(0, 対象者数 − 100) × ¥1 × 1.10 − クーポン割引。

### 5.3 見積結果表示項目
- 合計金額（税込）
- 送信対象者数
- 無料枠適用通数
- 課金対象通数（対象者数 − 無料枠適用通数）
- クーポン割引額
- 消費税額
- 内訳の注記

### 5.4 送信条件
- 画面状態が `after_event_ready` であること。
- 送信対象者が 1 名以上有効であること。
- 送信方法が `送信しない` でないこと。

## 6. 連携要件（参考）

### 6.1 初期表示に必要な情報
- アンケート基本情報（名称・ID・会期期間）
- 画面状態の判定材料（会期状態・データ化進捗・送信履歴・送信期限）
- 既存のお礼メール設定（送信方法、テンプレート、件名、本文）
- 受信者データ（会社名・部署・氏名・メールアドレス・ステータス）
- クーポン適用状態
- 見積結果

### 6.2 保存時に必要な情報
- 送信方法、テンプレート、件名、本文、送信対象者選択状態、クーポン適用情報、競合制御情報

### 6.3 送信実行時に必要な情報
- 送信対象者リスト、件名、本文、差し込み変数解決結果、見積確定額

### 6.4 クーポン検証・共有
- クーポンコード、適用可否、割引内容、メッセージ、適用後見積。
- 名刺データ化設定画面とクライアント側共有キー（`sharedCoupon_{surveyId}`）でクーポンを共有する。両画面で同じクーポンを表示・解除できる。

### 6.5 差し込み変数
- 利用可能な差し込み変数: `{{会社名}}` / `{{部署名}}` / `{{役職}}` / `{{氏名}}` / `{{アンケート名}}`。
- 件名・本文のいずれにも挿入可能。
- プレビューでは受信者情報とアンケート情報で解決する。

### 6.6 設計上の論点
- エンドポイント命名、HTTP メソッド、競合制御方式、エラー応答形式
- 画面状態判定ロジックをサーバ側 / クライアント側のどちらに寄せるか
- 送信期限の管理・自動遷移の責務分担
- 差出人・返信先・対象者フィルタ条件の仕様化（現行未実装）
- テンプレート切替時の既入力保持ポリシー

## 7. 失敗系要件

- **バリデーションエラー**: 件名未入力／超過、対象者 0 名、本文空
- **競合更新**: 他ユーザー更新を検知し、再読込または再取得導線を表示
- **認可エラー**: 編集・送信権限がない場合は操作不可とし、理由を表示
- **一時障害**: 取得・保存・クーポン検証・送信失敗時に再試行可能とする
- **送信部分失敗**: 一部宛先のみ失敗した場合、失敗分を識別可能にし再送導線を提供する

## 8. 非機能要件

- 入力変更に対するプレビュー・見積更新は利用者を待たせない応答時間で行う。
- 二重送信（保存・送信）を防止する。
- 画面再読込後も保存済み設定と送信履歴が正しく再現される。
- 大量受信者（数百件以上）でも表描画・ページネーションが安定して動作する。
- 送信確認モーダルには A11y 属性（`role="dialog"` / `aria-modal` / `aria-labelledby` / フォーカストラップ）を付与する。

## 9. 手動確認観点

- 初期表示で既存設定・受信者リスト・見積が正しく表示される。
- 5 つの画面状態それぞれで UI 表示が正しく切り替わる。
- 送信方法切替でプレビュー・見積・保存結果が整合する。
- テンプレート切替、件名・本文編集、差し込み変数挿入、プレビュー反映が正しく動作する。
- 対象者リストのチェック切替・全選択・ページ送り・件数変更で見積が正しく更新される。
- クーポン適用／解除が正しく反映され、名刺データ化設定画面と共有される。
- 保存成功／失敗、競合更新、認可エラーが正しく表示される。
- 送信確認モーダル・送信実行・結果反映・期限切れバナーが正しく動作する。

## 10. 現行モックとの差分（要整合）

現行モックは以下の点で正式要件から未達のため、本番実装時に解消する:

- **画面状態の自動遷移未実装** — 現行はシナリオセレクタによる手動切替のみ。会期・データ化進捗・送信期限からの自動判定が必要
- **送信期限管理未実装** — 期限日の管理・通知・自動遷移ロジックなし
- **実サーバ保存がモック** — `console.log` + 遅延のみ、実サーバ送信なし
- **実メール送信がモック** — ランダム成否分布を返すのみ、実送信基盤への接続なし
- **受信者データがモック** — 固定件数の合成データ、`surveyId` に応じた実データ取得なし
- **差出人・返信先設定未実装** — HTML に項目がなく、送信者アドレスを画面から指定できない
- **対象者フィルタ未実装** — 送信対象を条件（属性・セグメント）で絞り込む UI がない
- **クーポン検証がモック** — 名刺データ化設定画面と共通の限定コードのみ
- **クーポン共有キーがクライアント localStorage** — サーバ側管理へ移行
- **デバッグ用シナリオセレクタが本番画面に残存** — 本番リリース前に撤去
- **楽観ロック未実装** — 他ユーザー更新との競合検知なし
- **プラン連動未適用** — 契約プラン別の表示制限・上限が効かない
- **本文文字数上限未設定** — 件名のみ 255 文字、本文は無制限
- **送信確認モーダルの A11y 属性欠落** — `role` / `aria-*` / フォーカストラップ未付与
- **期限切れバナーの A11y 属性欠落** — `role="alert"` / `aria-live` 未付与

## 11. 本番化要件

本書はモックアップ作成段階の要件として作成したが、最終的に本番環境構築を担う開発会社が実装に着手するために必要な情報を本セクションに集約する。

### 11.1 API 契約

想定エンドポイントと入出力。詳細スキーマ・パスは実装着手時に確定する（以下パスは想定）。

| エンドポイント | 用途 | リクエスト | 主要レスポンス |
| --- | --- | --- | --- |
| `GET /api/surveys/{surveyId}/thank-you-email` | 初期表示（設定・受信者・見積・クーポン・画面状態・version を一括取得） | — | 200 `{ settings, recipients, estimate, coupon, state, deadline, version }` ／ 403 ／ 404 |
| `POST /api/surveys/{surveyId}/thank-you-email/estimate` | 見積再計算 | `{ recipientIds: string[], couponCode? }` | 200 見積結果（§5.3）／ 400 |
| `PUT /api/surveys/{surveyId}/thank-you-email` | 設定保存 | `{ sendMethod, template, subject, body, recipientSelection, couponApplied, version }` | 200 `{ settings, version }` ／ 409 ／ 400 ／ 403 |
| `POST /api/surveys/{surveyId}/thank-you-email/send` | 送信実行 | `{ recipientIds, subject, body, estimateConfirmed }` | 202 `{ jobId }` ／ 409 画面状態不整合 ／ 410 期限切れ |
| `GET /api/surveys/{surveyId}/thank-you-email/send-status/{jobId}` | 送信進捗・結果取得 | — | 200 `{ status, sent, failed, excluded, details[] }` |
| `POST /api/surveys/{surveyId}/coupons/validate` / `DELETE /api/surveys/{surveyId}/coupons` | クーポン検証・解除（06 と共通） | 06 と同等 | 06 と同等 |

- 送信は非同期ジョブで実行し、進捗は polling（数秒間隔を想定、要確定）またはサーバプッシュで取得する。
- 画面状態判定（§4.1 の 5 状態）はサーバ側で算出し API 応答に含める。クライアント側は表示のみに責任を持つ。

### 11.2 権限・認可

| 利用者種別 | 取扱 |
| --- | --- |
| アンケート所有者・編集権限保持者 | 設定編集可。送信実行は別権限で制御可能とする |
| 送信実行権限のみ | 設定閲覧可、送信ボタンのみ活性 |
| 閲覧のみ権限 | 表示のみ、保存・送信は非活性 |
| 未ログイン | アクセス不可 |
| 契約プラン非対応 | プラン連動により機能制限（`plan-capabilities.json` 参照） |

### 11.3 業務エラー時の挙動

| エラー種別 | ユーザー表示 | システム挙動 |
| --- | --- | --- |
| 初期取得失敗 | エラーメッセージ＋再試行導線 | フォーム非表示 |
| 見積再計算失敗 | 右カラムにエラー表示、最後の成功見積を保持 | 入力値保持 |
| 保存バリデーション（400） | 該当項目にエラー表示 | 入力値保持 |
| 楽観ロック競合（409） | 競合通知＋再読込導線 | 入力退避を促す |
| 送信期限切れ（410） | 期限切れバナー表示、送信ボタン非活性 | 画面状態を `period_expired` に強制遷移 |
| 送信基盤 5xx | 「ただいま送信できません。時間をおいて再度お試しください。」 | 自動リトライなし、再送導線を出す |
| 送信部分失敗 | 行ごとに「失敗」ステータス表示、失敗件数を画面上部に集約 | 失敗分のみ再送できる導線を提供 |
| 認可エラー（403） | 「この画面を編集／送信する権限がありません」 | 対応操作を非活性 |

### 11.4 データ永続化

- お礼メール設定テーブルを新設（`survey_id`（主キー）／`send_method`（enum: manual/auto/none）／`template_id`／`subject`／`body`／`recipient_selection`（JSON）／`deadline_at`／`version`）。
- 送信履歴テーブルを新設（`send_job_id`／`survey_id`／`recipient_id`／`status`（送信済み/失敗/除外/データ化中）／`sent_at`／`failure_reason`／`rendered_subject`／`rendered_body`）。
- 差し込み変数を解決した最終送信内容（件名・本文）を履歴に保存し、再現性を担保する。
- クーポン適用状態は 06 と共通のサーバ管理テーブルを共有。

### 11.5 ログ・監査

- 操作ログ: 設定保存、送信実行、対象者選択の変更、クーポン操作（誰が／いつ／前後値）。
- 監査ログ: 送信実行（送信件数・合計金額・利用クーポン・ジョブ ID）。
- 失敗ログ: 送信失敗（宛先・失敗理由・タイムスタンプ）、API 5xx 応答。
- メールアドレスは監査ログ以外では部分伏字化（`a***@example.com`）を原則とする。
- 保管期間は運用方針に従う（要確定）。

### 11.6 想定データ量

- 1 アンケートあたり受信者数: 数十〜数千件（大規模キャンペーンで最大 1 万件超）。
- ページネーション（20/50/100 件）と一括操作に耐える設計を前提とする。
- 送信ジョブは非同期で、並列配送のスループット・メール基盤側レート制限との整合を設計時確定。
- 差し込み変数解決は受信者数 × 設問数 × 変数数の組合せで膨らむため、送信時に一括解決してログに保存する。

### 11.7 対応ブラウザ・デバイス

- 想定対応: 最新版 Chrome / Edge / Safari / Firefox（直近 1 年以内のメジャーバージョン）。
- レスポンシブ対応: PC（横幅 ≥ 1024px）／タブレット／スマートフォン（横幅 ≥ 375px）。
- IE および非 Evergreen ブラウザは対象外。

### 11.8 受け入れ基準

- [ ] 5 つの画面状態がサーバ応答の `state` に基づいて正しく切り替わる（現行のシナリオセレクタは本番から撤去）
- [ ] 送信期限経過でサーバ側が `period_expired` を返し、送信ボタンが非活性になる
- [ ] 送信実行時に対象者数・合計金額・クーポンがサーバ側で再評価され、クライアント改ざんで課金できない
- [ ] 送信結果（送信済み／失敗／除外）が履歴テーブルに保存され、画面再読込で再現される
- [ ] 送信部分失敗時、失敗分のみを再送できる導線が動作する
- [ ] 差し込み変数は送信時にサーバ側で解決され、送信ログに最終本文が記録される
- [ ] 保存・送信で二重送信が発生しない
- [ ] 06 画面と同じクーポンが共有され、サーバ側で一貫した適用状態になる（クライアント localStorage は使用しない）
- [ ] 送信確認モーダル・期限切れバナーが A11y 属性（`role` / `aria-*` / フォーカストラップ）を満たす
