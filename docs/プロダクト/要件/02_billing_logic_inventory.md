# 請求ロジック棚卸し（料金表中心）

## 1. 目的

本資料は、2026年3月13日時点でディレクトリ内に存在する請求関連資料を、`料金体系` を主軸に棚卸ししたものです。

- 決定事項の再定義は行わない
- 資料間で一致する内容は「現時点で言い切れる内容」として整理する
- 資料間で不一致の内容は「矛盾・未確定論点」として残す
- 実装とモックデータは正式仕様ではなく、`現行挙動の参考情報` として別節で扱う

## 3. 料金表中心の整理

### 3.1 プラン料金

#### 現時点で言い切れる内容

- `Premium` の基準価格として `月額10,000円（税別）` が複数資料で登場する。
- 契約単位は、少なくとも資料上は `個人アカウント` または `グループ` の両方が想定されている。
- 現行のユーザー向け請求書UIとモックデータは `STANDARD` `PREMIUM` `PREMIUM_PLUS` を扱う前提で作られている。

#### 根拠資料

- `docs/プロダクト/要件/01_billing_and_invoicing_logic.md`
  - プレミアム基本料金を `10,000円（税別）` と定義
  - 契約単位を個人アカウントまたはグループ単位と記載
- `docs/リファレンス/リソース/client-materials/service-plan-comparison.md`
  - プレミアムプラン（月額）を `月額10,000円（税抜）` と記載
  - スタンダードプランとプレミアムプランの比較表を掲載
- `docs/画面設計/仕様/04_invoice_screen.md`
  - `plan.code` の例として `STANDARD/PREMIUM/PREMIUM_PLUS` を記載
- `data/core/invoices.json`
  - 6件の請求書に対し、`STANDARD:3件 / PREMIUM:1件 / PREMIUM_PLUS:2件` を保持

#### 矛盾・未確定論点

- `PREMIUM_PLUS` は画面仕様とモックデータには存在するが、料金表や課金ルールは正式文書で定義されていない。
- `STANDARD` の課金単位は、比較表では `実数精算 @50円/枚` が中心だが、モック請求書では `月額基本サービス料 (Standardプラン)` が存在する。
- `プレミアム利用料` と `従量課金` の境界は、最新版の `docs/プロダクト/要件/01_billing_and_invoicing_logic.md` で「機能解放型」として定義された。
  - プレミアムは上位機能の利用権
  - 従量課金は原則継続
  - 明示的に内包と定義したもののみ例外扱い

### 3.2 従量課金

#### 現時点で言い切れる内容

- 名刺データ化費用として `50 / 100 / 150 / 200円` の速度別単価が複数資料に記載されている。
- お礼メール送信は `100通までは無料、101通目以降は1通1円` という記述が複数資料にある。
- オンデマンドプランは少なくとも比較表上 `プレミアム契約限定` とされている。

#### 根拠資料

- `docs/プロダクト/要件/01_billing_and_invoicing_logic.md`
  - 通常/特急/超特急/オンデマンドの単価を定義
  - お礼メールの無料枠と超過料金を定義
  - `想定件数の50%` を最低請求とするルールを定義
- `docs/リファレンス/リソース/client-materials/service-plan-comparison.md`
  - スピードプランごとの単価と納期目安を記載
  - 脚注で `実績が想定枚数の50%未満なら想定枚数の50%分を請求` と記載
  - メール送信の `101件目以降 1円/件` を記載

#### 矛盾・未確定論点

- `最低請求ルール` は資料で揺れている。
  - `01_billing_and_invoicing_logic.md` と `service-plan-comparison.md` は `想定件数の50%下限あり`
  - `【01.要件定義書】2025-01-14業務機能構成表の請求にまつわるメモ.txt` は `全て実数精算を基本、最低使用料金なし`
- `プレミアム契約中の従量課金有無` は最新版の `docs/プロダクト/要件/01_billing_and_invoicing_logic.md` で原則継続と整理された。
  - 名刺データ化費用は継続
  - お礼メール送信費用は継続
  - オンデマンドプランはプレミアム加入条件だが、料金は従量課金

### 3.3 追加料金

#### 現時点で言い切れる内容

- `追加アカウント料` と `クーポン` のルールは、現状では `01_billing_and_invoicing_logic.md` が最も明示的に定義している。
- モック請求書には `追加ユーザーライセンス` や各種オプション行が含まれており、少なくともUI上は追加料金行を表示できる。

#### 根拠資料

- `docs/プロダクト/要件/01_billing_and_invoicing_logic.md`
  - グループ利用時 `2名まで無料、3名以上は1名あたり300円/月`
  - クーポンを `定額割引 / 定率割引` として定義
  - `1請求につき1種類のみ適用`、有効期限ありと記載
- `data/core/invoices.json`
  - `追加ユーザーライセンス`
  - `追加ストレージ`
  - `API連携オプション`
  - `優先サポートオプション`
  - などの明細を保持

#### 矛盾・未確定論点

- `追加アカウント料 300円/月` は他資料で裏取りが少なく、現時点では単独定義に近い。
- モックデータ上の各種オプション料金は画面確認用の可能性が高く、正式な課金規則や適用条件を説明する要件書は見当たらない。
- クーポンは最新版の `docs/プロダクト/要件/01_billing_and_invoicing_logic.md` で、原則として従量課金項目へ適用し、プレミアム利用料および追加アカウント料金には適用しない整理となった。
- クーポンの運用フロー接続点は、請求書UIやOrder-to-Cash文書では依然として未接続である。

### 3.4 請求条件

#### 現時点で言い切れる内容

- 支払い方法は `銀行振込` が基本想定。
- グループ契約時は、少なくとも要件文書上 `グループ管理者（オーナー）宛てに一括請求` という考え方が存在する。
- 請求書の発行から送付、入金確認、消込までの業務プロセスは定義済みである。

#### 根拠資料

- `docs/プロダクト/要件/01_billing_and_invoicing_logic.md`
  - 締め日を `毎月末日`
  - 支払日を `翌月末払い`
  - 支払い方法を `銀行振込`
  - 個人契約は個人アカウントへ、グループ契約はグループ管理者へ一括請求
- `docs/プロダクト/プロセス/02_invoice_order_to_cash.md`
  - 受注登録 → 請求データ作成 → レビュー → 最終承認/発行 → PDF生成 → 送付 → 支払い → 入金確認/消込 のフローを定義
- `docs/リファレンス/リソース/client-materials/reference-docs/【01.要件定義書】2025-01-14業務機能構成表の請求にまつわるメモ.txt`
  - 月末締めベース
  - 銀行振込
  - 請求書は自動発行

#### 矛盾・未確定論点

- `請求タイミング` は資料で揺れている。
  - `01_billing_and_invoicing_logic.md`: 月末締め、翌月末払い
  - `【01.要件定義書】2025-01-14業務機能構成表の請求にまつわるメモ.txt`: 月末締め翌月頭請求、月初メール送信
- `請求書発行方法` は資料で揺れている。
  - 旧メモは `自動発行 + 元Excelから自動入力`
  - Order-to-Cash は `オペレーター作成 → 管理者承認` を含む
- `請求単位と請求先` は最新版の `docs/プロダクト/要件/01_billing_and_invoicing_logic.md` で、グループ契約時はグループ管理者へ一括請求し、プレミアム利用料・追加アカウント料金・従量課金を同一請求書内で別行表示する整理となった。
- 旧資料のうち、グループ全体の請求書へプレミアム費用を合算しない記述とは差分が残っている。

### 3.5 運用ステータス

#### 現時点で言い切れる内容

- 業務プロセス文書には、請求作成から入金・消込・アーカイブまでを含むライフサイクル定義がある。
- ユーザー向け請求書UIとモックデータは、より少ない表示用ステータスに縮約されている。

#### 根拠資料

- `docs/プロダクト/プロセス/02_invoice_order_to_cash.md`
  - `draft`
  - `awaiting_approval`
  - `issued`
  - `sent`
  - `paid`
  - `reconciled`
  - `overdue`
  - `cancelled`
  - `archived`
- `docs/画面設計/仕様/04_invoice_screen.md`
  - 画面用 `status` として `unpaid/paid/overdue/canceled`
- `data/core/invoices.json`
  - 実データ上も `unpaid / paid / overdue / canceled` の4種のみ

#### 矛盾・未確定論点

- `業務ステータス` と `画面表示ステータス` の対応表が未定義。
- `cancelled` と `canceled` の綴り揺れがある。
- `reconciled` や `archived` は業務文書には存在するが、ユーザー請求書UIやモックデータでは未使用。

## 4. 現行モック実装から読み取れる前提

### 4.1 一覧集計

#### 現時点で言い切れる内容

- 一覧画面は `請求書単位` ではなく `請求月 + 契約種別` の月次サマリーを表示する。
- 集計IDは `AGG-YYYYMM-{GROUP|PERSONAL}` 形式で生成される。

#### 根拠資料

- `02_dashboard/src/invoiceList.js`
  - `aggregateInvoices(...)` が `issueDate` の年月と契約種別で集計
- `docs/画面設計/仕様/04_invoice_screen.md`
  - INV-001 を月次サマリーと定義

#### 矛盾・未確定論点

- 契約種別の判定が `plan.code === 'STANDARD'` を個人、それ以外をグループとしており、契約実態と完全一致する保証がない。

### 4.2 詳細画面の金額計算

#### 現時点で言い切れる内容

- 詳細画面は `items[]` を正として `subtotalTaxable / subtotalNonTaxable / tax / totalAmount` を再計算する。
- 表示用請求書番号は `YY-ユーザーID5桁-連番3桁` 形式に変換される。

#### 根拠資料

- `02_dashboard/src/services/invoiceService.js`
  - `calculateInvoiceTotals(items)` で課税/非課税/税額/合計を再計算
  - `fetchInvoiceById(id)` 内で表示用IDへ整形
- `docs/画面設計/仕様/04_invoice_screen.md`
  - 表示用IDルールを明記

#### 矛盾・未確定論点

- 保存済みヘッダ金額と `items[]` 合計がズレている請求書がある場合、画面表示は再計算値に寄る。
- `data/core/invoices.json` では `INV-2025-09-002` の保存済み合計が `55,000円` に対し、`items[]` 再計算は `536,800円` となり大きく乖離している。

### 4.3 印刷ページ

#### 現時点で言い切れる内容

- 印刷要件書は `BASE → ADD_ON → ONE_TIME → CREDIT` のカテゴリ順出力とカテゴリ小計を求めている。
- 現行実装は `items[]` をそのまま順に描画しており、カテゴリ別の並び替えや小計計算は行っていない。

#### 根拠資料

- `docs/画面設計/仕様/05_invoice_document.md`
  - カテゴリ順とカテゴリ小計を要件化
- `docs/ハンドブック/テスト/invoice_manual_checklist.md`
  - 印刷ページでカテゴリ小計を確認項目化
- `02_dashboard/src/invoicePrint.js`
  - `renderItems(items)` が配列を順番に描画するのみ

#### 矛盾・未確定論点

- `印刷要件` と `現行実装` に差分がある。
- `AGG-...` の小計行は帳票要件上「印刷には出ない」前提だが、現行コードはカテゴリフィルタも小計制御もしていないため、今後の実装修正時に扱いを再確認する必要がある。

### 4.4 モックデータ品質

#### 現時点で言い切れる内容

- モック請求書は6件存在する。
- ステータス内訳は `unpaid:2 / paid:2 / overdue:1 / canceled:1`。
- プラン内訳は `STANDARD:3 / PREMIUM:1 / PREMIUM_PLUS:2`。
- `INV-2025-09-002` では `category` と `taxable` の欠損が集中している。

#### 根拠資料

- `data/core/invoices.json`
  - 6件の請求書データを保持
  - `INV-2025-09-002` は `24件中 category欠損21件 / taxable欠損17件`

#### 矛盾・未確定論点

- `docs/画面設計/仕様/05_invoice_document.md` と `docs/ハンドブック/テスト/invoice_manual_checklist.md` は `category` と `taxable` を帳票整合の必須条件としている。
- したがって現行モックデータは、帳票仕様の前提を満たしていない。

## 5. 現時点の論点一覧

### 優先度高

- 最低請求ルールは `想定件数50%下限あり` なのか、`完全実数精算` なのか
- 請求タイミングを `翌月頭請求` と `翌月末払い` のどちらで運用定義するのか
- 業務ステータスとユーザー表示ステータスの対応をどう整理するのか

### 優先度中

- `PREMIUM_PLUS` の料金・契約単位・請求条件
- 追加アカウント料 `300円/月` の正式化有無
- クーポンの請求フロー接続点
- 帳票における明細粒度を `プレミアム利用料 / 追加アカウント料金 / 名刺データ化費用 / お礼メール送信費用 / クーポン値引き` で固定し、実装とモックデータへどう反映するか
- 印刷ページのカテゴリ小計仕様を実装へ合わせるか、実装を要件へ合わせるか

## 6. 現時点の暫定結論

- `料金ポリシーのたたき台` としては `docs/プロダクト/要件/01_billing_and_invoicing_logic.md` が最もまとまっている。
- プレミアムと従量課金の境界は、最新版の `docs/プロダクト/要件/01_billing_and_invoicing_logic.md` を基準に `機能解放型` として扱うのが妥当である。
- `クライアント向けに近い価格表現` は `docs/リファレンス/リソース/client-materials/service-plan-comparison.md` が最も具体的である。
- `運用フロー` は `docs/プロダクト/プロセス/02_invoice_order_to_cash.md` を参照すべきである。
- `ユーザー請求書UIの前提` は `docs/画面設計/仕様/04_invoice_screen.md` 系と `02_dashboard/src/` 実装を分けて読む必要がある。
- 企画検討の次アクションとしては、まず `最低請求ルール` `請求タイミング` `ステータス体系` `PREMIUM_PLUS` の位置づけを意思決定し、その後に画面仕様・モックデータ・管理業務フローを揃えるのが安全である。
