# SPEED AD 実運用チーム体制

## 1. 目的

本ドキュメントは、SPEED AD の実サービス運用に関わる組織と主責務を最小限の粒度で整理し、通常案件時および障害・仕様変更時の連携経路を明確にすることを目的とします。

想定読者は、アブロードアウトソーシング株式会社、Rep 株式会社、運用関係者、および海外パートナー企業の管理担当者です。

## 2. 実運用に関与する組織

| 組織 | 主な役割 | 主責任 | 主な連携先 |
| --- | --- | --- | --- |
| アブロード プロダクトチーム | サービス方針・仕様判断の統括 | サービス方針、仕様判断、品質観点の確認、開発優先度の整理 | BPO部、営業部、Rep 株式会社 |
| アブロード BPO部 | 案件受注後の運用担当 | 顧客対応、案件進行管理、運用面の一次窓口 | プロダクトチーム、入力チーム |
| アブロード 営業部 | 販売販路の担当 | 販売活動、案件創出、顧客接点の初期形成 | BPO部、プロダクトチーム |
| 海外パートナー企業の入力チーム | サービス裏方業務の担当 | 名刺入力などの実務オペレーション | BPO部 |
| Rep 株式会社 | 技術面の対応窓口 | システム改修、技術調査、不具合解析、技術的改善提案 | プロダクトチーム |

## 3. 最小限の連携ルール

### 3.1. 通常案件時の相談経路

- 営業部が案件化した内容は、運用開始に向けて BPO部へ引き継ぐ。
- BPO部は、案件運用上の不明点や追加確認事項をプロダクトチームへ相談する。
- システム仕様や画面挙動に関する判断が必要な場合は、プロダクトチームが一次判断を行う。
- 技術的な改修や調査が必要な場合は、プロダクトチームから Rep 株式会社へ連携する。

### 3.2. 障害・不具合時の連携

- 顧客または運用現場で発見された不具合は、まず BPO部またはプロダクトチームで受け付ける。
- 運用手順や入力内容で解消可能な事象は、BPO部と入力チームで対応する。
- システム起因の疑いがある場合は、プロダクトチームが内容を整理して Rep 株式会社へエスカレーションする。
- 重大な障害や顧客説明が必要な場合は、アブロードアウトソーシング株式会社内で判断したうえで対外説明方針を統一する。

### 3.3. 仕様変更要望の扱い

- 顧客要望や営業起点の仕様変更要望は、営業部または BPO部からプロダクトチームへ集約する。
- 採否、優先度、対応時期の判断はプロダクトチームが行う。
- 実装を伴う場合は、プロダクトチームから Rep 株式会社へ依頼する。

## 4. 役割境界

| 境界 | 整理方針 |
| --- | --- |
| 営業と運用の境界 | 営業部は案件創出と初期提案を担い、受注後の案件進行と顧客運用窓口は BPO部が担う。 |
| 運用と開発の境界 | BPO部と入力チームは運用実務を担い、システム改修や技術調査は Rep 株式会社が担う。仕様判断と優先度整理はプロダクトチームが担う。 |
| 入力チームと BPO 管理の境界 | 入力チームは実作業を担い、案件管理、品質確認、顧客との調整は BPO部が担う。 |

## 6. 前提・想定

- 本書は実運用体制の責務整理を目的とし、日次・週次の詳細手順書は含みません。
- 海外パートナー企業の入力チームに関する個社名や担当者名は、本書では固定せず役割単位で記載します。
- 顧客との契約条件や運用時間帯の変更が生じた場合は、BPO部とプロダクトチームの責任境界を見直す可能性があります。
