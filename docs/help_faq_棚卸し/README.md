# help/FAQ コンテンツ棚卸しDB

ヘルプセンター（記事40本）とよくある質問（FAQ 23問）の全コンテンツを棚卸しし、
**用語の揺れ**と**記載が事実と食い違っていないか（嘘チェック）**を点検するためのスプレッドシートDB。

対象データの正本:
- FAQ … [05_support/assets/data/faq.json](../../05_support/assets/data/faq.json)
- ヘルプ記事 … [05_support/assets/data/help_articles.json](../../05_support/assets/data/help_articles.json)

## 仕組み（自動同期）

コンテンツ（help/FAQ）の正本は上記の **JSON**。ページはこのJSONから描画される。
スプレッドシートはその **自動ミラー**で、Google Apps Script が公開URLのJSONを取得して各タブへ書き込む。

```
[正本JSON]  faq.json / help_articles.json      ← ここを直す（＝HTML側の更新）
     │ 編集して push → GitHub Pages が再ビルド（公開URLが最新化）
     ▼
[Apps Script]  Code.gs  ← 公開URLのJSONをフェッチして各タブへ書き込む
     │  ・台帳 / 用語揺れ / 主張 を自動生成
     │  ・人の判断列は ID キーでマージして温存（同期しても消えない）
     ▼
[Google スプレッドシート]  共有 → 誰が見ても最新を確認できる
```

**データの向きは JSON → スプシ の一方向**。スプシ側で本文を直しても site には戻らない。
スプシで人が触るのは判断列（一致・推奨表記・担当など）だけで、ここは同期で保持される。

### レビュー記入表（HTML → スプシ／glossaryと同方式・2タブ）

[レビュー記入表.html](レビュー記入表.html) は **そのまま開いて使える単体HTML**
（Live Server でも `file://` でも可）。用語棚卸の
[glossary_用語棚卸_ユーザー側.html](../glossary_用語棚卸_ユーザー側.html) と同じ仕組み。タブで2種類のレビューを分離:

- **タブ1「回答内容レビュー」（カード形式・63件）** … help記事1本／FAQ1問ごとに
  **設問＋回答本文＋実画面リンク**をカードで表示（長文が読みやすいようカード）。
  正しければ「確認OK」、直すなら「原文コピー→修正後の内容」、気づきはコメント。判定（○△×）は無し。
- **タブ2「項目をまたいだ矛盾」（表形式・26件）** … 1項目だけ読んでも気づけない
  項目間の食い違い（カードブランドの差・存在しないプラン名・CSV説明の矛盾など）。コメント＋確認のみ。

```
[レビュー記入表.html]  ブラウザで開く（共有はファイルを配るだけ）
   │  タブ1: 63カード（FAQ23＋ヘルプ40）／タブ2: 26行の矛盾テーブル
   │  保存=POST(no-cors)／取得=JSONP（CORS回避）／localStorageにも保存
   │  15秒ごとに自動で最新取得（共同編集者の入力が反映される）
   ▼
[GAS decisions バックエンド]  gas/Code.gs  → スプシ factcheck_decisions シート
   （ID / 確認後の内容=修正後 / 判定メモ=コメント / 判定=対応ステータス(確認済み) / 記入者 / 更新日時）
```

行データは `_gen_inventory.mjs` が `_form_template.html` の `<!--CARDS-->`/`<!--CLASHROWS-->`
へ流し込んで生成。ID は回答レビュー＝`FAQ-xxx`/`HELP-xxx`、矛盾＝安定スラッグ（`gen001-svc` 等）。
「開く」リンクは実画面（例: `/05_support/help-content/?article=ts001`）。

**セットアップ:**
1. 記入表用の Google スプレッドシートを新規作成（台帳DBとは別でよい）
2. Apps Script に [gas/Code.gs](gas/Code.gs) を貼り、`SPREADSHEET_ID` を1のIDに変更
3. デプロイ → ウェブアプリ（実行=自分／アクセス=共有範囲）→ `/exec` URL を控える
4. [回答内容レビュー_記入表.html](回答内容レビュー_記入表.html) の `var GAS_WEB_APP_URL = '__GAS_URL__';`
   の `__GAS_URL__` を、その URL に差し替える
5. HTMLを配布（社内サーバ/共有フォルダ/GitHub Pages など）。各自開けば記入→自動同期

> GAS URL 未設定のあいだは、このブラウザ内（localStorage）に保存されるだけで動作確認はできる。
> URL を入れて再配布すると、全員のスプシ同期が有効になる。

## 同期（JSON → スプシ・任意）

同期の起こし方は3通り（併用可）:
1. **手動** … スプシのメニュー「棚卸しDB」→「今すぐ同期」（編集者なら誰でも押せる）
2. **毎日自動** … メニュー「棚卸しDB」→「自動同期を毎日ONにする」（時刻トリガー）
3. **編集即時（任意）** … JSONを push すると GitHub Action が Pages 再ビルドを待って GAS を叩く
   （[.github/workflows/sync-inventory.yml](../../.github/workflows/sync-inventory.yml)）

### セットアップ（GAS）

1. 共有用の Google スプレッドシートを新規作成
2. 「拡張機能」→「Apps Script」を開き、[Code.gs](Code.gs) の内容を貼り付けて保存
3. スプレッドシートを再読み込み →メニュー「棚卸しDB」→「今すぐ同期」（初回は権限承認）
4. 定期実行したい場合は「自動同期を毎日ONにする」
5. （任意）「編集即時」にしたい場合は `sync-inventory.yml` の冒頭コメントの手順で
   ウェブアプリ公開＋Secrets登録

> スプレッドシートを関係者に共有すれば、全員が最新の棚卸し結果を閲覧できる。
> 編集権限を渡した人は「今すぐ同期」で即時更新できる。

## オフライン版（CSV・任意）

ネット同期を使わず手元でCSVが欲しい場合、`_gen_inventory.mjs` を `node` で実行すると
GASと同じレイアウトの3CSV（UTF-8 BOM付き）を生成する。GASを使う場合は不要。

```
node _gen_inventory.mjs
```

> ⚠️ 抽出した主張（事実チェック）と用語グループの定義は、`Code.gs` と `_gen_inventory.mjs`
> の両方に同じ内容を持つ。**運用の正は `Code.gs`**。主張・グループを足すときは Code.gs を編集する
> （オフラインCSVも使うなら両方に反映）。

## ファイル構成

### 01_コンテンツ台帳.csv（DBの背骨・63行）
help/FAQ の全項目を1行ずつ。他の2シートはこの `台帳ID` で参照する。

| 列 | 内容 |
|---|---|
| 台帳ID | `FAQ-xxx` / `HELP-xxx`。両ソース横断で一意 |
| 種別 | FAQ / ヘルプ記事 |
| カテゴリID・カテゴリ名 | 元データのカテゴリ |
| 項目ID | 元データの id |
| 質問・タイトル | 設問文／記事タイトル |
| 要約 | ヘルプ記事の summary（FAQは空） |
| 回答本文 | 回答・本文（改行は空白化。markdown記法はそのまま） |
| 注目 | isFeatured なら ★ |
| 更新日 | ヘルプ記事の updatedAt（FAQは空） |
| 参照リンク | 公開ページのURL |
| 最終確認日／確認ステータス／担当 | **運用記入用（空欄）** |

### 02_事実チェック.csv ／ タブ「事実チェック」（嘘棚卸し・26行）
検証可能な主張（料金・期間・数量上限・機能有無/対応プラン・手順・対応環境・法務）を抽出。
**主張は原文ママの抜粋。一致判定は人間が行う前提で、判定列は空欄で渡している。**

| 列 | 内容 |
|---|---|
| チェックID | 安定キー（並び替えに強い。GAS版は `gen001-svc` 等、CSV版は `FC-001`〜） |
| 出典台帳ID | どの項目の主張か |
| 主張内容(原文ママ抜粋) | コンテンツに書かれている主張 |
| 主張種別 | 料金／契約・解約／数量上限／機能・対応プラン／エクスポート仕様 など |
| 照合先候補 | 確認先のルーティング（例: 料金プラン仕様書、plansページ、利用規約）。**判定ではない** |
| 備考(事実のみ) | 自己申告の「※仮文言」、他項目との差異など、**事実のみ**を記載（自動生成） |
| 仕様上の値 | **記入用（空欄）**：正本での実際の値 |
| 一致(○/△/×/要確認) | **記入用（空欄）**：照合結果 |
| 判定メモ | **記入用（空欄）**：判定の根拠・補足 |
| 判定者 | **記入用（空欄）** |

### 03_用語揺れ.csv ／ タブ「用語揺れ」（14グループ）
同じ概念が別表記になっているものをグループ化。**バリエーションは原文ママ。統一の判断はしない（推奨表記列は空欄）。**
既存の[用語洗い出し表](../glossary_用語洗い出し.md)と同じ方針（列挙のみ・判断は人間）。

| 列 | 内容 |
|---|---|
| グループ | G1〜 |
| 概念(意味) | そのグループが指す概念 |
| 出現バリエーション(原文ママ) | 実際に出現した表記を ` / ` 区切りで |
| FAQ/HELP/plansページ 出現(目安) | 各ソースでの出現数 |
| 出現箇所メモ | （任意） |
| 備考(事実のみ) | どこで割れているかの事実（自動生成） |
| 推奨表記 | **記入用（空欄）**：統一方針を決めたら記入 |
| 判定メモ | **記入用（空欄）** |

> ⚠️ **出現数は「目安」**。文字列の部分一致でカウントしているため、「特急」が「超特急」に含まれる、「プレミアム」が「プレミアムプラン」に含まれる等で重複・過大計上がある。**0か1以上か（そのソースに在るか無いか）の判断材料**として使い、厳密な件数としては扱わないこと。

## 初回点検で見えている主な論点（叩き台）

判定は人間が行う前提だが、抽出時点で目に付いた食い違いを事実として記録した。

- **お礼メールの三重表記** … plansページ「御礼メール」 / help・faq「お礼メール」 / help動画記事「サンクスメール」（G1）
- **カードブランドの不一致** … FAQ-bil004はJCBを含む4ブランド、HELP-bil001はJCBなし3ブランド（FC-022 / G8）
- **エンタープライズプランの存在** … HELP-misc004にのみ登場。plansページはスタンダード/プレミアムの2プラン構成（FC-024 / G7）
- **CSV文字コード説明の不一致** … FAQ-func008「UTF-8 BOM付き・直接読込可」 vs HELP-use004「UTF-8・文字化けする場合あり」（FC-010 / G12）
- **「無料プラン」という名称** … help・faqは多用するが、plansページには無い（スタンダード=月額なし）（FC-005 / G3）
- **自己申告の仮文言** … FAQ-bil004「※運用担当に要確認」、FAQ-bil005「※本番公開時に確定」（FC-008 / FC-009）

これらは「揺れ・要確認」として挙げているだけで、どれを正とするかは未判定。
