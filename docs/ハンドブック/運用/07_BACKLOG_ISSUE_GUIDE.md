# 07_BACKLOG_ISSUE_GUIDE.md: Backlog 起票手順

## 1. 目的
- Backlog へ課題を起票する際の確認順、転記ルール、記録先を統一する。
- 特に定期的に発生する `SPDAD_SCD` への展示会スケジュール起票を、毎回同じ粒度で実施できるようにする。

## 2. 対象プロジェクト
- `SPDAD2026`: 開発課題、仕様確認、UI差分整理、返答用課題。
- `SPDAD_SCD`: `SPEED AD展示会スケジュール`。展示会日程の管理用。

## 3. 起票前の確認
1. 起票先プロジェクトを確認する。
2. 元データの正本を確認する。
3. 既存課題の件名ルール、種別、優先度、本文項目を確認する。
4. 重複起票を避けるため、Backlog 上で件名検索を行う。

## 4. 展示会スケジュール起票の標準手順
### 4.1. 前提
- 起票先は `SPDAD_SCD`。
- 課題種別は `展示会`。
- 優先度は既存運用に合わせて `中`。
- 開始日には会期開始日、期限日には会期終了日を設定する。

### 4.2. 件名ルール
- 基本形は `展示会名_開始月/日_終了月/日`。
- 会場名や開催日数の末尾値が未確定、または運用上不要な場合は件名に含めない。
- 例: `関西物流展_4/8_4/10`

### 4.3. 本文ルール
- 元データで確認できる項目だけを記載する。
- 現在の標準は以下の 2 行。

```md
内容：LM/FA
トータル件数：900
```

- `利用企業社数` や `アンケート数` が元データにない場合は、無理に補完しない。

### 4.4. 起票フロー
1. 元データから `展示会名` `内容` `トータル件数` `会期` を整理する。
2. 日付を `YYYY-MM-DD` に正規化する。
3. 曜日表記がある場合は、実カレンダーと矛盾がないか確認する。
4. 既存課題を件名検索し、重複がないことを確認する。
5. 課題を起票する。
6. 起票後に `issueKey` を控え、必要に応じて会議メモや関連資料へ転記する。

## 6. 記録先
- 起票ルールの正本はこのファイルに残す。
- 課題の背景や判断理由など社内限定情報は、shared repo ではなく社内 private 管理先に残す。
- Backlog wiki へ転記する場合は、旧アーカイブではなく現行の正本資料から転記する。
