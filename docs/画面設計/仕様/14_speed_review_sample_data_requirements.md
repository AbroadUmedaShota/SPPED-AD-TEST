# SPEEDレビュー/詳細画面 模範回答データ要件（展示会想定）

## 1. 目的
SPEEDレビュー画面と詳細画面の画面設計・検証に使う「模範回答データ」の作成条件を定義する。  
本要件は、展示会運用（1日〜最大5日、1日300件想定）を前提とする。

## 2. ボリューム前提（更新版）
- 展示会日数: 1〜5日
- 1日あたり回答件数: 300件想定
- 総回答件数レンジ: 300〜1,500件

検証用データセットは最低でも次の3段階を用意する。
- `S`（1日想定）: 300件
- `M`（3日想定）: 900件
- `L`（5日想定）: 1,500件

画面設計レビュー用の基準セットは `M: 900件` とする。

## 3. 回答形式の網羅条件
1つのアンケート内で、サービス上で扱う全回答形式を含める。

- `single_choice`
- `multi_choice`
- `text`
- `free_text`
- `number`
- `date`
- `time`
- `matrix_sa`
- `matrix_ma`
- `handwriting`
- `image`
- `explanation`

## 4. データ構成条件
対象ファイルは以下3点を1セットとする。
- `docs/サンプル/demo_surveys/<surveyId>.json`
- `docs/サンプル/demo_answers/<surveyId>.json`
- `docs/サンプル/demo_business-cards/<surveyId>.json`

整合条件:
- `surveyId` が3ファイルで一致すること
- `answerId` が回答データと名刺データで1:1対応すること
- `details[].question` が設問定義と一致すること

## 5. 展示会らしい分布条件
- 回答日時は開催期間内に分布させる
- 時間帯は `09:00-19:00` を中心にする
- 昼休憩帯（12時台）でやや落ち、午前後半/夕方で山が出る分布を持たせる
- 日別件数は概ね300件だが、`±10%` の揺らぎを許容する

## 6. 周辺データ条件（画像含む）
- 名刺画像（表/裏）の実ファイル参照を含める
- `handwriting` と `image` は実在ファイル参照を含める
- 画像あり/なし（欠損）を混在させ、フォールバック表示を検証できるようにする

## 7. ステータス/例外パターン条件
- データ化完了とデータ化進行中（`businessCard` なし）を混在
- 未回答（空文字/`null`）を一部含める
- 境界値を含める（数値最小/最大、長文、日付・時刻境界）
- `isTest` の true/false を混在（表示影響確認）

## 8. 受け入れ基準
- 12形式すべてがレビュー一覧と詳細で崩れず表示される
- 設問切替時にテーブル列/KPI/内訳集計が同期更新される
- 日付・ステータスフィルタが期待どおりに動作する
- 300/900/1,500件の各セットで表示・操作が実用速度で成立する
- 画像読み込み失敗時にUIが破綻せず、コンソールエラーが致命化しない

## 9. 次ステップ（実データ作成）
次工程では以下を実施する。

1. `surveyId` を1つ確定し、12形式入り設問定義を固定する  
2. `S/M/L` 3セット（300/900/1,500件）を同一ロジックで生成する  
3. 画像参照パスと欠損率を定義して回答/名刺データへ反映する  
4. 画面で実表示確認し、受け入れ基準チェック結果を記録する
