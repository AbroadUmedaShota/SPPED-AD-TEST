# 管理者画面 サービス構造

## 1. 目的

管理者画面を、既存画面名ではなくSPEED ADのサービス構造と管理者責務から定義する。

管理者画面の中心単位は「アンケート案件」とする。アンケート案件を支える管理対象を整理し、後続の責務別要件と画面構成の基準にする。

## 2. 対象外

- 既存 `03_admin/` HTML構成の追認
- 個別画面のUIレイアウト、文言、HTML/JS実装
- 本番DB/API、認可、監査ログの詳細設計
- Backlog進捗、スケジュール、担当者整理
- 受注前の営業、商談、見積、提案管理

## 3. 判断根拠

| 種別 | 扱い |
| --- | --- |
| 本資料および後続の `10_`〜`70_` 資料 | 管理者画面要件の正本 |
| `03_admin/` | 参考: 現行モック。仕様上の正解とは扱わない |
| `data/` | 参考: モックデータ。実運用DBとは扱わない |
| `BY-*`、`sample/`、`Moved` HTML | 参考・移行・検証枠 |

## 4. 管理対象

| 管理対象 | サービス上の意味 | 管理者責務 | 後続資料 |
| --- | --- | --- | --- |
| 顧客・企業 | SPEED ADを利用する契約主体 | 契約・利用状態、連絡先、請求先の確認と補正 | `10_admin_account_management.md` |
| ユーザー・グループ | 顧客側の操作主体と共同利用単位 | 招待、停止、所属、請求先との関係確認 | `10_admin_account_management.md` |
| アンケート | 回答収集・名刺データ化・請求の中心案件 | 公開状態、回答状況、関連設定、履歴の確認と補正 | `20_admin_survey_management.md` |
| 設問・回答 | 回答者体験と分析の元データ | 構成確認、回答状況確認、例外時の調査 | `20_admin_survey_management.md` |
| 名刺データ | 回答に紐づく入力・照合・納品対象 | 入力、照合、例外、納品状態の管理 | `30_admin_data_operation.md` |
| 請求・クーポン・営業日 | 利用料金、割引、期限計算の前提 | 請求状態、割引条件、営業日/期限前提の管理 | `40_admin_billing_and_rules.md` |
| オペレーター・実績 | 社内/外部作業者と処理品質 | 作業者状態、権限、処理量、品質の確認 | `50_admin_operator_management.md` |

## 5. 管理者責務の分類

| 責務 | 定義 |
| --- | --- |
| 参照 | サービス状態、顧客状態、案件状態、作業状態を確認する |
| 登録 | 管理者が裏側で必要な管理対象を新規作成する |
| 編集 | 誤りや運用上必要な情報を補正する |
| 状態更新 | 公開、停止、入力中、照合中、請求済みなどの状態を更新する |
| 出力 | CSV、請求、一覧、作業用データなどを取得する |
| 通知 | 利用者、作業者、関係者へ必要な連絡を行う |
| 例外対応 | 通常フローで処理できない差異、エラー、問い合わせを処理する |

## 6. 利用者側サービスとの接点

| 接点 | 管理者側で見る理由 | 主な責務 |
| --- | --- | --- |
| 利用者アカウント | ログイン、契約、請求、権限の基礎になるため | 参照、登録、編集、状態更新 |
| アンケート作成・公開 | 顧客が作成した案件の運用状態を支えるため | 参照、状態更新、例外対応 |
| 回答者画面 | 回答収集結果と名刺データ化対象になるため | 参照、例外対応 |
| 名刺データ化設定 | 入力・照合・納品・請求条件に影響するため | 参照、編集、状態更新 |
| 請求書表示 | 利用者が確認する請求内容と管理側の請求状態を分離するため | 参照、登録、編集、出力 |

## 7. 完了条件

- 管理者画面全体を管理対象と責務で説明できる。
- 既存画面名を起点にせず、後続の責務別要件へ分解できる。
- 本文要件と参考情報の境界が分かる。

## 8. 未確定事項

- 管理者が利用者作成アンケートを直接編集できる範囲。
- 公開後の設問・回答データ変更可否。
- 管理者通知の送信対象、履歴、再送ルール。
- 本番API、権限、監査ログとして必要な画面外確認事項。
