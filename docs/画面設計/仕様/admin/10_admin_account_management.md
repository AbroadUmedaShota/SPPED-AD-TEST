# 管理者画面 顧客・ユーザー・グループ管理要件

## 1. 目的

顧客・企業、利用者アカウント、グループを管理する責務を定義する。

この領域は、ログイン、契約、請求、アンケート作成主体、共同利用単位の基礎になる。管理者は利用者側操作を代替するのではなく、サービス運用に必要な確認、補正、状態管理を行う。

## 2. 対象外

- 受注前の営業、商談、見積管理
- 本番認証基盤、MFA、IP制限の詳細
- 利用者側グループ編集画面のUI仕様
- 契約金額や個別商談条件の詳細

## 3. 判断根拠

| 種別 | 扱い |
| --- | --- |
| `01_admin_service_structure.md` | 管理対象と責務境界の根拠 |

## 4. 管理対象

| 管理対象 | 管理者が見る情報 | 管理者が変更してよい情報 |
| --- | --- | --- |
| 顧客・企業 | 会社名、連絡先、契約状態、利用状態、関連ユーザー | 基本情報、請求先、内部メモ、利用停止/再開 |
| ユーザー | 氏名、メール、所属、招待状態、利用状態 | 招待、再送、停止、所属補正、連絡先補正 |
| グループ | グループ名、管理者、メンバー、請求先、関連アンケート | 管理者補正、メンバー補正、請求先補正 |

## 5. 責務別要件

| 責務 | 要件 |
| --- | --- |
| 参照 | 顧客、ユーザー、グループを検索し、関連アンケート、請求、状態を確認できる |
| 登録 | 管理者が必要に応じて顧客、ユーザー招待、グループを登録できる |
| 編集 | 連絡先、請求先、所属、内部メモを補正できる |
| 状態更新 | 招待中、利用中、停止、退会相当の状態を更新できる |
| 出力 | 顧客一覧、ユーザー一覧、グループ一覧をCSV等で出力できる |
| 通知 | 招待、再送、重要連絡を利用者へ送信できる |
| 例外対応 | メール不達、所属誤り、請求先誤り、重複アカウントを処理できる |

## 6. 画面化する時の要点

- 一覧は検索、絞り込み、状態確認に限定する。
- 詳細は基本情報、関連アンケート、関連請求、操作履歴相当の確認に分ける。
- 危険操作は一覧から直接実行せず、詳細または確認導線へ寄せる。

## 7. 画面外確認事項

- 認証、認可、メール送信、招待トークンの保存先。
- 管理者による代理編集の監査ログ。
- 個人情報の表示範囲とマスキング要否。

## 8. 未確定事項

- 管理者がグループメンバー権限をどこまで変更できるか。
- 利用者自身の編集と管理者補正が競合した場合の優先順位。
- 削除を物理削除、停止、退会、非表示のどれで扱うか。
