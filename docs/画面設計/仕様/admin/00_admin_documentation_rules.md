# 管理者画面資料 作成ルール

## 1. 目的

管理者画面資料を完全新規で作り直すための共通ルールを定義する。

このルールは、管理者画面の個別仕様、サービス構造整理、責務境界整理、参考メモを作成する前提として使う。既存の削除済み資料はないものとして扱い、復元・継承しない。

## 2. 対象外

- 個別管理者画面の仕様作成
- UI再設計、HTML/JS実装、モックデータ改修
- 本番DB/APIの仕様確定
- Backlog進捗、スケジュール、担当状況の整理
- 旧26/27系リライト版資料の復元

## 3. 判断根拠

| 種別 | 扱い |
| --- | --- |
| 新規作成する管理者画面資料 | 正本 |
| `docs/画面設計/仕様/admin/` | 管理者画面資料の正本置き場 |
| `03_admin/` | 参考: 現行モック。仕様上の正解とは扱わない |
| `data/` | 参考: モックデータ。実運用DBとは扱わない |
| `BY-*` | 参考: 業務特化画面。正本候補にはしない |
| `sample/` | 参考: 検証・サンプル |
| `Moved` HTML | 参考: 移行導線 |
| 旧管理者資料、旧26/27系資料 | 削除済み。根拠にしない |

## 4. ファイル命名

ファイル名は `NN_english_slug.md` とする。本文は日本語でよい。

| 番号帯 | 用途 | 例 |
| --- | --- | --- |
| `00_` | 資料ルール、全体方針、用語 | `00_admin_documentation_rules.md` |
| `01_` | サービス構造、管理対象、責務境界 | `01_admin_service_structure.md` |
| `10_` | 顧客・企業・ユーザー・グループ | `10_admin_account_management.md` |
| `20_` | アンケート、設問、公開、回答 | `20_admin_survey_management.md` |
| `30_` | 名刺データ、入力、照合、エスカレーション | `30_admin_data_operation.md` |
| `40_` | 請求、クーポン、営業日、期限前提 | `40_admin_billing_and_rules.md` |
| `50_` | オペレーター、権限、実績 | `50_admin_operator_management.md` |
| `90_` | 参考、差分、移行メモ | `90_admin_mock_reference.md` |

## 5. 必須front-matter

各資料には次のfront-matterを付ける。

```yaml
---
owner: product
status: draft
last_reviewed: YYYY-MM-DD
---
```

`status` は `draft`、`review`、`approved`、`reference` のいずれかを基本とする。

## 6. 必須セクション

新規資料には、最低限次のセクションを置く。

| セクション | 目的 |
| --- | --- |
| `目的` | その資料で決めることを明示する |
| `対象外` | 混ぜてはいけない領域を明示する |
| `判断根拠` | 何を根拠にしたか、何を根拠にしないかを明示する |
| `未確定事項` | 次工程で決めることを本文要件と分ける |

## 7. 作成順序

最初に `01_admin_service_structure.md` を作成し、管理対象、利用者側サービスとの接点、管理者責務を定義する。

個別画面仕様は、サービス構造と責務境界が固まった後に作成する。既存画面名や既存HTML構造から先に仕様を組み立てない。

## 8. 既存HTMLの引用ルール

`03_admin/` のHTMLやモックを引用する場合は、必ず `参考: 現行モック` と明記する。

例:

```markdown
参考: 現行モック `03_admin/user-management.html` にはユーザー一覧がある。
```

この表現は、現行状態の確認には使えるが、仕様判断の根拠にはしない。

## 9. 未確定事項

- 個別画面仕様を画面単位で作るか、責務単位で作るか。
- 開発会社提出用の `docs/リライト版仕様書/admin/` を再作成するタイミング。
