# お礼メール設定画面 要件定義書

## 1. 文書目的
- 本書は、お礼メール設定画面の本番実装時における正本仕様を定義する。
- 現行の `02_dashboard/thankYouEmailSettings.html` は参照実装であり、本書の正本性は本番要件にある。
- 旧配置の `docs/要件定義/02_thankYouEmailSettings_requirements.md` は legacy とし、今後の更新は本書で行う。

## 2. 画面概要
- 対象画面は、アンケート回答者へのお礼メール送信設定、送信対象確認、送信実行、送信結果確認を行う画面とする。
- 画面は左側の設定・本文編集エリアと、右側の件数・費用・送信アクションエリアで構成する。
- 本画面の状態は画面内の擬似切替ではなく、アンケート状態、名刺データ化状態、送信期限、送信実績をもとにサーバーが返す状態を正とする。

## 3. 画面要件
### 3.1 上部情報
- パンくずと画面タイトルを表示する。
- アンケート基本情報として以下を表示する。
  - アンケート名
  - アンケートID
  - 会期期間
- 送信期限切れ時は上部に警告バナーを表示する。

### 3.2 左カラム
- 送信方法
  - `手動送信`
  - `自動送信`
  - `送信しない`
- メール内容設定
  - テンプレート選択
  - 件名入力
  - 差し込み変数一覧
  - 本文入力
- リアルタイムプレビュー
  - 件名
  - 本文
  - 差し込み変数不足の警告
- 送信対象者一覧
  - 一括選択
  - 個別選択
  - 会社名
  - 氏名
  - メールアドレス
  - 必要に応じて部署名・役職
  - 送信結果ステータス
  - ページング
  - 検索またはフィルタ

### 3.3 右カラム
- 送信対象件数
- 無料枠
- ご請求見込み金額
- クーポン入力と適用状態
- `設定を保存する`
- `お礼メールを送信する`

## 4. 状態要件
### 4.1 画面状態定義
サーバーは以下のいずれかの状態を返す。
- `before_or_during_event`
- `after_event_processing`
- `after_event_ready`
- `after_event_sent`
- `period_expired`

### 4.2 `before_or_during_event`
- 送信対象者一覧は表示せず、準備中メッセージを表示する。
- メール本文と送信方法の保存は可能とする。
- 送信実行は不可とする。

### 4.3 `after_event_processing`
- 名刺データ化または対象者確定処理中として扱う。
- 対象者一覧は未確定状態の案内を表示する。
- 本文・送信方法の保存は可能とする。
- 送信実行は不可とする。

### 4.4 `after_event_ready`
- 送信対象者一覧を表示する。
- 対象者の選択、件数確認、費用確認を可能とする。
- 送信実行を可能とする。

### 4.5 `after_event_sent`
- 送信設定は参照主体とし、再送は不可とする。
- 送信結果件数と各対象者の送信結果を表示する。
- 部分失敗がある場合はその旨を明示する。

### 4.6 `period_expired`
- 送信実行は不可とする。
- 対象者一覧は参照可能とし、必要に応じて代替対応を案内する。

## 5. 主要挙動
### 5.1 初期表示
- 画面表示時に以下を取得する。
  - アンケート基本情報
  - お礼メール設定
  - 利用可能テンプレート
  - 差し込み変数一覧
  - 送信対象集計
  - 現在見積
  - 送信可能期間
  - 状態
  - 競合制御に必要な識別情報

### 5.2 送信方法
- 送信方法は単一選択とする。
- `送信しない` を選択した場合はお礼メール機能を無効化した状態で保存する。

### 5.3 件名・本文
- 件名は最大 255 文字とする。
- 本文は複数行入力を可能とする。
- 差し込み変数はクリック操作で本文へ挿入できるようにする。

### 5.4 プレビュー
- プレビューは代表サンプルデータまたはプレビュー専用データを用いて生成する。
- 差し込み変数のうち、対象者データで欠損しうる項目がある場合は警告する。
- プレビュー専用データは本番送信対象とは分離する。

### 5.5 送信対象者一覧
- 一覧はサーバー連携で取得し、ページング可能とする。
- 一括選択は現在の絞り込み条件に対して明確な適用範囲を持つこと。
- 行ごとの選択変更に応じて件数と見積を更新する。
- 送信済み、対象外、失敗などの状態を視覚的に区別する。

### 5.6 クーポン
- クーポンはサーバー管理とする。
- 適用時はアンケートに対する適用可否と金額影響を返す。
- 解除時は確認を行い、解除後見積を再反映する。

### 5.7 保存
- 保存時は二重送信を防止する。
- 保存成功時は完了通知を表示し、最新状態を再反映する。
- 保存失敗時は入力値を保持したままメッセージを表示する。

### 5.8 送信実行
- 送信前に確認モーダルを表示する。
- 確認内容には対象件数と費用を表示する。
- 実行時は非同期ジョブを起動し、受理後はジョブ状況または最終結果を参照できるようにする。
- 同一対象への重複送信は防止する。

## 6. 費用見積要件
### 6.1 計算入力
- `activeCount`
- 無料枠残数または無料枠上限
- クーポン

### 6.2 計算結果
- 送信対象件数
- 無料適用件数
- 無料控除額
- クーポン値引き額
- 税別金額
- 税額
- 税込金額

### 6.3 計算責務
- 本番では見積計算ロジックの正はバックエンドとする。
- フロントは返却値を表示し、必要に応じて仮計算を補助利用するに留める。

## 7. 連携要件（参考）
### 7.1 初期表示に必要な情報
- アンケート基本情報
- 現在のお礼メール設定
- 利用可能テンプレート
- 差し込み変数一覧
- 送信対象集計
- 現在見積
- 送信可能期間
- 現在状態
- 他ユーザー更新との競合を検知するための識別情報

### 7.2 保存時に必要な情報
- 送信方法
- テンプレート
- 件名
- 本文
- クーポン適用情報
- 競合制御に必要な識別情報

### 7.3 送信対象者一覧で必要な能力
- ページング
- 検索またはフィルタ
- 一括選択の適用範囲を判定できる情報
- 選択可能件数と総件数の把握
- 行単位の状態表示に必要な情報

### 7.4 クーポン検証で必要な情報
- 入力されたクーポンコード
- 当該アンケートに対する適用可否
- 割引内容
- 利用者向けメッセージ
- 必要に応じて適用後見積

### 7.5 送信実行・結果確認で必要な能力
- 対象者指定情報をもとに送信実行を受け付けられること
- 非同期実行または同等の方法で、送信中と送信結果を追跡できること
- 送信成功件数、失敗件数、対象外件数、失敗理由を確認できること

### 7.6 設計上の論点
- エンドポイント命名
- HTTP メソッド
- 一覧取得のページング表現
- 競合制御方式
- エラー応答形式
- 非同期ジョブの表現方法
- 送信対象指定を条件指定にするか ID 指定にするか

## 8. 失敗系要件
- 対象者 0 件
  - 送信実行不可とし、理由を表示する。
- 送信期限切れ
  - 送信実行不可とし、代替手段を案内する。
- 再送禁止
  - 既送信対象への再送要求は拒否する。
- 重複送信防止
  - 送信実行中または受理済みジョブがある場合は制御する。
- 楽観ロック
  - 他ユーザー更新との競合を検知し、再読込または再取得導線を表示する。
- 部分失敗
  - 送信成功/失敗を分けて表示し、失敗理由を確認できるようにする。
- 認可エラー
  - 編集・送信権限がない場合は権限制御メッセージを表示する。

## 9. 非機能要件
- 一覧表示はページング前提で実用件数に耐えられること。
- 件数変更時の見積更新は利用者が待たされない応答時間であること。
- 保存・送信時は二重送信を防止すること。
- 競合更新検知ができること。
- 送信結果は監査上確認できる粒度で保持・表示できること。

## 10. 手動確認観点
- 初期表示で設定、集計、見積、状態が正しく表示されること。
- 送信方法切替、テンプレート変更、差し込み変数挿入、プレビュー更新が正しく動くこと。
- 対象者一覧のページング、選択、検索/フィルタ、件数反映が正しいこと。
- クーポン適用/解除で見積が更新されること。
- 保存成功/失敗、競合更新、認可エラーが正しく表示されること。
- 送信確認、送信実行、期限切れ、再送禁止、部分失敗が正しく制御されること。

## 11. 現行モックとの差分
- 現行モックのシナリオ切替 UI は本番要件に含めない。
- 現行モックの固定テンプレート、固定ダミー会期、ランダム送信結果、ブラウザストレージ連携は本番要件に含めない。
- 本番では状態判定、対象者一覧、送信結果、見積、クーポンはすべてサーバー連携を正とする。
- API の具体設計は設計チーム判断とする。
