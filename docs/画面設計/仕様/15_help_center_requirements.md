# ヘルプセンターページ 要件定義書

> **TL;DR（この文書の位置づけ）**
> - 対象は `02_dashboard/help.html`（トップ）と `help-content.html`（カテゴリ/記事/検索結果）の2画面。
> - 本書は「現行実装のトレースドキュメント」。HTML/JS/CSS のソースを根拠に書き、過去メモと実装が食い違っていた箇所は実装側に合わせて整合済み。
> - 将来拡張（KPI 計測、CMS 化、サーバー送信フィードバック等）は §9 に分離。
> - セキュリティ交渉不可項目（XSS 対策・`published` 配信制御・入力バリデーション）は §4.2.2 / §4.3 / §4.2.1 に集約。

## 1. 概要

### 1.1 優先度凡例

| 区分 | 意味 |
|------|------|
| **MVP** | 本版で必須。リリース条件。 |
| **Should** | 推奨。合理的理由があれば延期可。 |
| **Nice** | 任意。余力があれば対応。 |
| **Phase 2** | 本版対象外。§9 に集約。 |

### 1.2 目的・想定利用者・利用シーン

ヘルプセンターは、SPEED AD ダッシュボード利用者（広告運用担当・管理者）の疑問を自己解決させ、問い合わせ件数を削減する機能。想定利用者は（1）管理者（アカウント・権限管理）、（2）一般利用者（広告作成・レポート）、（3）多言語利用者（日本語・英語）。典型シーン: ダッシュボード操作中に詰まり別タブでヘルプを開いて自己解決。外部検索エンジンからの流入はスコープ外。

### 1.3 ユーザーストーリー

- 管理者として権限・アカウント関連を素早く検索したい（設定ミスの波及を防ぐため）。受入: 「アカウント」「権限」で関連記事 1 件以上（§4.2.2）。
- 一般利用者としておすすめやカテゴリから手順記事を見つけたい（問い合わせ回答待ち時間を避けるため）。受入: `isFeatured=true` 記事が最大 6 件表示（§4.1.5）。
- 多言語利用者として自分の言語で UI・記事を読みたい。受入: `getCurrentLanguage()==='en'` で英語翻訳ある記事は英語表示（§4.1.8）。

### 1.4 KPI 指標候補（Phase 2 で確定）

ベースライン・目標値・決定期限はすべて Phase 2 で確定。計測詳細は §9。候補: 自己解決率（ヘルプ閲覧後に問い合わせ未送信の割合）、検索ゼロ件率、フィードバック Yes 率、Contact 遷移率。

### 1.5 対象範囲 / 対象外

**対象**: `02_dashboard/` の `help.html` / `help-content.html`、および §8 の `support.speed-ad.com` 配下の共通アセット運用ルール。
**対象外**（§9 参照のみ）: サーバー送信フィードバック、閲覧ログベースおすすめ算出、ROI 試算、SaaS 代替ツール比較。

## 4. 機能要件

> **TL;DR**
> - §4.1 はトップページ（ヒーロー・検索・人気キーワード・おすすめ・カテゴリ・CTA）。
> - §4.2 はコンテンツページ（URL モード切替・検索・ソート・パンくず・フィードバック）。
> - §4.3 はデータソース（`help_articles.json` と `published` 配信制御）。
> - `support.speed-ad.com` 配下は別スコープ（§8）。

### 4.1 トップページ（`help.html`）

#### 4.1.1 ヒーローセクション（検索フォーム） [**MVP**]

- ヒーローに検索フォーム `<form method="GET" action="help-content.html">` を配置。
- `name="search"` 送信で `help-content.html?search=<keyword>` へ遷移。キーワードは `encodeURIComponent` でエンコード。
- 本機能はクライアント完結。サーバー化する場合はレート制限方針を別途定義する（TBD）。

#### 4.1.2 検索サジェスト [**MVP**]

- 入力中にインクリメンタル（逐次）でサジェスト表示。対象は `question`（タイトル）+ `answer`（本文）の連結。
- 最小入力文字数: 2 文字（UTF-16 code units、`help-center.js` L129）。
- debounce は**現行未実装**（`input` から直接発火、L102-L104）。Phase 2 導入検討（§9）。
- マッチ方式: 連結文字列を `toLowerCase()` 後、スペース区切りトークンで AND 部分一致（L134-L141）。全半角・ひらがな/カタカナ正規化は未対応（§4.2.2）。
- 表示件数: 最大 6 件。ランキングは AND マッチ全件を JSON 定義順で取得（タイトル前方一致優先等は未実装、Phase 2）。
- 0 件時: 「`{keyword}` の記事が見つかりません。Enter キーで全文検索できます。」を表示しドロップダウンは開いたまま（`renderSuggestionsEmpty()`、L178-L188）。
- キーボード: 上下矢印・Home・End で選択、Enter で `help-content.html?article=<articleId>` 遷移、Escape で閉じる（L229-L253）。
- アクセシビリティ: ARIA `combobox` / `listbox` パターンで実装。件数変化は `aria-live`（領域変化を支援技術に通知する仕組み）で読み上げ。`aria-label` で候補件数通知（L164）。

#### 4.1.3 スティッキー検索バー [**MVP**]

- ヒーローがビューポート上端を通過した時点で、画面上部に縮小版検索バーを固定表示。
- 出現トリガー: `window.scrollY > hero.offsetTop + hero.offsetHeight + 120`（px）。`scroll` イベント（passive）で監視（L263）。
- フェード時間: 250ms（`setTimeout(..., 250)` で `hidden` 化。CSS transition と同値、L271）。
- 表示/非表示は `is-visible` クラスと `hidden` 属性でフェード制御。
- SP/Tablet/Desktop で同一挙動（ブレークポイント別制御は現行なし）。

#### 4.1.4 人気キーワードチップ [**MVP**]

- ヒーロー下部に最大 5 件のチップ表示。クリックで当該キーワードの検索結果ページへ遷移。
- 現行は静的定義リスト。PV や検索ログからの動的算出は Phase 2（§9）。サーバー化時のレート制限方針は TBD。

#### 4.1.5 おすすめ記事 [**MVP**]

- `isFeatured=true` の記事から先頭 6 件を表示。
- **並び順（実装準拠）**: `isFeatured=true` で `filter` 後、JSON 定義順のまま `.slice(0, 6)`（L355-L358）。`order` は未使用。
  - `order` を使う場合の仕様: 昇順・欠落は末尾・タイブレークは `title` 昇順 → `articleId` 昇順。
- **欠落時既定値**: `isFeatured` → `false` / `updatedAt` → エポック / `order` → 末尾 / `published` → `false`（配信対象外）。
- **`published=true` のみ配信対象**（§4.3）。
- **6 件超運用**: `isFeatured=true` は同時 6 件まで。超過分は非表示になるため月次棚卸しでフラグを外す。
- レイアウト: CSS Grid（カルーセル・自動再生・ページネーションは非実装）。
- レスポンシブ: SP 1 列 / `≥640px` 2 列 / `≥1024px` 3 列。

#### 4.1.6 カテゴリから探す [**MVP**]

- 全カテゴリを Bento グリッド（異なるサイズのカードを格子状に並べるレイアウト）で表示。順序は `CATEGORY_ORDER`（`help-center.js` L22-L31）固定順。
- 各カードに記事プレビュー最大 2 件（L308）。**公開記事 0 件のカテゴリは非表示**。
- グリッド末尾に FAQ 専用カードを挿入（`FAQ_CARD` 定数 L33-L42、アイコン `help_outline`、リンク `faq.html`）。通常カテゴリ描画との干渉は §5.2 ビジュアル回帰で確認。

#### 4.1.7 CTA セクション [**MVP**]

- ページ下部に「お探しの答えは見つかりましたか？」セクション。
- リンク: 「お問い合わせ（不具合報告・ご質問）」→ `bug-report.html`、「よくある質問を見る」→ `faq.html`。
- `bug-report.html` は不具合・一般問い合わせ兼用で、問い合わせ種別選択を必須。
- 「回答には通常 〇 営業日いただいております」文言を表示（日数 TBD）。受付時間・SLA・緊急連絡は運用側定義（TBD）。

#### 4.1.8 多言語対応 [**Should**]

> 第 2 周レビューで MVP → Should に降格。i18n 共通化は Phase 2（§9）。

- 対応言語: 日本語・英語を想定（Phase 2 で確定）。
- `window.getCurrentLanguage()` は実装済み（`help-center.js` L6 / `help-content.js` L27）。サイト内切替 UI の所在は未調査（Phase 2）。
- `languagechange` はブラウザ自体の言語設定変更を示す Web 標準イベントで、サイト内切替 UI とは独立。サイト内 UI が `languagechange` を dispatch するかは未調査。
- `resolveLocalizedValue()` で現在言語のテキストを取得。**フォールバック順**: 現在言語 → 日本語 → キー名（L7）。
- 未翻訳時は日本語で代替表示しつつ、コンソールへ `[i18n warn] missing translation: {lang}/{articleId}` を出力。
- ローカライズ対象: `question` / `answer` / `summary` / カテゴリ名。`isFeatured` / `updatedAt` / `order` / `published` はロケール非依存。
- `languagechange` 購読で切替時にリアルタイム再レンダリング（`help-center.js` L74-L77、`help-content.js` L219-L221）。再レンダリング時は localStorage `sa.help.feedback.v1` を再読込して投票済み UI を保持。スクロール位置・展開状態・検索結果はリセット。
- 翻訳欠落検知・追加フローは運用引継ぎ資料で定義（TBD）。

### 4.2 コンテンツページ（`help-content.html`）

#### 4.2.1 URL クエリによる表示モード切替 [**MVP**]

優先順位（上が優先）:
1. `?search=<keyword>`: 検索結果一覧
2. `?article=<articleId>`: 記事詳細
3. `?category=<categoryId>`: カテゴリ内記事一覧
4. いずれも無い、または `search` が `trim()` 後空: `renderNotFound()`（404 相当）

**入力値バリデーション（セキュリティ交渉不可・必須）**:
- `article` / `category` は許可リストパターン `^[A-Za-z0-9_-]{1,64}$` に一致する場合のみ受け付ける。不一致は 404。**理由**: URL から DOM へ値を組み込む経路で、パス/スクリプト注入や想定外のリソース参照を防ぐため。
- `article` / `category` が JSON 内の既知 ID と一致しない場合は 404。
- `href` への値組み込みは URL コンストラクタで同一オリジン検証。異なるオリジン/パース失敗時は表示中止。**理由**: `javascript:` スキームやオープンリダイレクトを防ぐため。
- `search`: 長さ上限 200 字（UTF-16 code units）。制御文字（U+0000〜U+001F、U+007F〜U+009F）は除去。**理由**: 意図しないレンダリングや長大入力による処理負荷を抑えるため。
- 空検索語（`trim()` 後空）は §4.2.2 の 0 件フローではなく `renderNotFound()` を表示（現行実装準拠、`help-content.js` L184-L185）。Phase 2 で統一検討（§9）。

URL 正規化・canonical タグ付与は Phase 2 で確定。

#### 4.2.2 検索ロジック [**MVP**]

**クライアント完結**。将来サーバー化時のレート制限方針は別途定義（TBD）。

**検索条件**:
- 空白（半角・全角）で分割して AND 検索。
- 連続空白は単一区切り。
- クォート（`"`）囲みは未対応（通常トークン扱い）。
- **トークン上限**: 実装上は無制限（`split(/\s+/).filter(Boolean)`、`help-content.js` L505）。超長大キーワードは §4.2.1 の 200 字制限で遮断。Phase 2 で 10 トークン程度に制限予定（§9）。
- 対象フィールドはタイトル（`question`）と本文（`answer`）の連結文字列。

**正規化ポリシー**:
- 現行は小文字化（`toLowerCase()`）のみ。
- **NFKC**（Unicode 正規化の一種。全角英数/半角カナ等を統一形に揃える）・全半角統一・ひらがな/カタカナ同一視は現行未対応。
- リスク: 「ｷｰﾜｰﾄﾞ」と「キーワード」、「ゆうこう」と「有効」が別語扱いとなり検索漏れが発生する可能性。Phase 2（§9）。

##### 4.2.2 XSS 対策（セキュリティ交渉不可・必須）

検索ヒット箇所に `<mark>` を付与する場合、順序を誤ると XSS（悪意ある JS 実行）を招く。正しい順序は「先に全文エスケープ → エスケープ済み文字列に `<mark>` を挿入」。逆順では `<mark>` 自体が `&lt;mark&gt;` 化してハイライトが失われるか、エスケープ前にユーザー入力を HTML 展開する危険実装となる。

**手順（この順で実装する。`innerHTML` への未エスケープ文字列の直接代入は禁止）**:

1. 表示対象テキスト（タイトル・本文）の HTML 特殊文字（`&`, `<`, `>`, `"`, `'`）を `escapeHtml()` でエスケープする。
   - 理由: ユーザー入力や記事本文に `<script>` が入っていても、ただの文字列として無害化するため。
2. エスケープ済みの安全な文字列に対して、各トークンの正規表現（`escapeRegex()` で特殊文字エスケープ済み）でマッチ位置を特定する。
   - 理由: トークン側の正規表現特殊文字（`.` `*` `?` 等）が暴発しないよう、検索語もエスケープしておく。
3. マッチ位置に `<mark>$&</mark>` を `String.replace()` で挿入する。
   - ここで挿入される `<mark>` は開発者側が書いた固定文字列なので、HTML として安全に解釈されてよい。ユーザー入力は手順 1 で既に無害化済み。
4. 生成した HTML 文字列を `innerHTML` に代入する。

現行実装準拠箇所: `help-center.js` L152-L156 `renderSuggestions()`、`help-content.js` L538-L545 `renderSearchResults()`。`Range + textContent` 方式や DOM 断片 `appendChild` 方式への変更は、本方式からの乖離となるため §3 改訂履歴にエントリを追加したうえで検討する。

**検証用 XSS ペイロード**は §5.2 参照。

- `answer`（本文）はプレーンテキスト扱い。HTML タグを含む場合も上記手順 1 で必ずエスケープ。

**0 件時**:
- 「該当する記事は見つかりませんでした」を表示。
- 次アクション導線:
  - 代替検索語リスト（人気キーワードチップ。未設定なら非表示）+ 「キーワードを変えて再入力してください」のヒント。
  - FAQ リンク（`faq.html`）。
  - 問い合わせリンク（`bug-report.html`）。

#### 4.2.3 カテゴリ内の絞り込み・ソート [**MVP**]

- キーワード絞り込み: タイトル + サマリ（無ければ本文）で部分一致（小文字化のみ）。
- ソート順（`sortOrder`）:
  - `featured`（既定）: `isFeatured` desc → `updatedAt` desc → `title` 昇順 → `articleId` 昇順。
  - `newest`: `updatedAt` 降順。
  - `title`: `localeCompare` によるタイトル昇順。空タイトルは末尾扱い。記号・数字始まりはロケール仕様に従う（実装要確認）。
- 公開記事が無い場合は「このカテゴリにはまだ記事がありません」を表示。
- **`published=true` の記事のみ配信対象**（§4.3）。

#### 4.2.4 パンくず [**MVP**]

- 先頭に「ダッシュボード（`index.html`）→ ヘルプセンター（`help.html`）」を常に表示。
- モード別の末尾要素:
  - 検索結果: `検索結果: "<keyword>"`（リンクなし）
  - カテゴリ一覧: カテゴリ名（`help-content.html?category=<id>` へのリンク）
  - 記事詳細: カテゴリ名（リンク付き）→ 記事タイトル（リンクなし）
- 生成失敗時は最終階層を「...」または空文字で代替し、描画を停止しない。

#### 4.2.5 記事詳細: フィードバック UI [**MVP**]

- 「この記事は参考になりましたか？」セクション。`data-vote="yes"` / `data-vote="no"` の 2 ボタン。

**保存・集計方針**:
- 投票結果は **`localStorage` にのみ保存**。Phase 1 ではサーバー送信しない（`console.debug` のみ）。運営側は集計していない。
- 「端末内のみ保存。外部に送信されません。」旨を UI に表示（現行未実装、本書で明文化）。表示位置はボタン下部（投票前のみ）。多言語は Should。
- 投票データに PII・自由記述テキストを含めない。

**localStorage スキーマ（セキュリティ交渉不可・必須）**:

**設計根拠**:
- **キー名前空間 `sa.help.feedback.v1`**: localStorage は 1 オリジン内で全スクリプトが共有されるため、他機能との衝突・上書きを避けるためプロダクト名 + 機能名 + バージョンを接頭辞とする。スキーマ更新時は `v2` に上げて共存可能。
- **有効期限 180 日**: 記事更新後は「参考になった」判断が古くなるため、永久保存を避け 180 日で未投票へ戻して記事改善サイクルに追従させる。
- **スキーマバージョン `v`**: 将来フィールドを追加・削除する際に古いデータを安全に無視・移行できるようにするため。

- キー（移行先）: `sa.help.feedback.v1`。現行実装は旧キー `helpArticleFeedback`（`help-content.js` L31）。
- 旧スキーマ: `{ [articleId]: { vote: 'yes'|'no' } }`（`at` なし）。
- 移行後スキーマ: `{ [articleId]: { v: 1, vote: "yes" | "no", at: <ISO8601 UTC> } }`。
- 読み込み時は JSON パース + スキーマバリデーション（`v` 存在・`vote` 値・`at` フォーマット）必須。
- `at` に未来日時がある場合（端末時計改竄の可能性）は当該エントリを「未投票」扱い。時計改竄による期限延長リスクはサーバー検証（Phase 2）まで残る。
- 移行手順: 新キーが無い場合のみ旧キーを読み込んで新キーへ書き直す（1 回限り）。`at` 欠落時は移行実行時刻で補完し、180 日カウントを起算。

**有効期限**: `at` から 180 日（カレンダー日、UTC 基準）。期限切れは「未投票」扱い。現行は有効期限チェック未実装（`help-content.js` L463 は `existing` の有無しか見ない）。Phase 1 で実装。

**画面分岐**:
- 投票済み（期限内）はボタンを表示せず「ご協力ありがとうございました。」のみ。
- **「いいえ」投票後**: 「投票は記録されません。問い合わせはこちら（記事 ID（例: HELP-0123）を記載ください）」と `bug-report.html` / `faq.html` への誘導。`bug-report.html?articleId=<id>` 自動引き渡しは Phase 2（§9）。現行は「はい/いいえ」同一メッセージ（L209-L216）。Phase 1 で分岐実装。
- 投票取消 UI は Phase 2（§9）。
- **複数タブ**: `storage` イベント購読による他タブ投票反映は現行未実装。Phase 1 で実装。
- **容量超**: 現行は `catch (_e) { /* no-op */ }`（L43-L45）。仕様: `QuotaExceededError` 発生時はコンソール警告 + 投票ボタン非表示 + 「フィードバックを保存できませんでした。」表示。Phase 1 で実装。

サーバー送信化（Phase 2）実施時は、利用目的公表・法的根拠・保存期間・削除請求・越境移転・PII 伏字化レベル定義を §9 前提条件に含める。

#### 4.2.6 記事詳細: 記事 ID 表示 [**MVP**] / おすすめ記事 [**Should**]

**記事 ID 表示 [MVP]**:
- 記事詳細の記事ヘッダー内またはメタ情報欄に `articleId` を表示し、サポート担当者が問い合わせ内容と記事を照合できるようにする。
- 現行未実装（`help-content.js` L416-L434）。Phase 1 で実装。

**おすすめ記事 [Should]**:
- `isFeatured === true` から現在表示中記事を除外して抽出。
- 現行は全カテゴリ横断後、`Math.random()` でシャッフル（L477-L483）。同一カテゴリ優先は Phase 2（§9）。
- **シャッフル実装**: 現行は純粋乱数（`Math.random()`）。seeded 乱数（セッション ID または当日日付ベースのシードで、同一セッション中は同じ順序を返す擬似乱数）は未実装。Phase 2 で置き換え検討（§9）。
- 表示上限 3 件。除外後 3 件未満はその件数で表示し、0 件時はセクション自体を非表示。

### 4.3 データソース [**MVP**]

- `data/help_articles.json`: 唯一のデータソース。
- 期待構造: `{ categories: [{ id, name, questions: [{ id, question, answer, summary, isFeatured, updatedAt, order, videoUrl, published }] }] }`
- パス解決は共通ユーティリティ `resolveDashboardDataPath`。

**`published=true` のみ配信（セキュリティ交渉不可・必須）**:

**ビルド成果物から `published` フィールド自体を除去する理由**: `published=false` の記事が JSON に残っていれば、CI 検査で「未使用」と確認しても HTTP 越しに全文取得できてしまう。検査やアプリ側フィルタは二重防御にすぎず、**一次防御は「成果物に最初から含めない」こと**。フラグ名の存在自体が「公開保留記事あり」という情報漏洩となるため、フィールドごと落とす。

- `published=false` または未定義の記事は全モードで除外。
- 公開ビルドには `published=true` 記事のみを焼き込み、`published` フィールド自体を成果物 JSON から除去（**必須**）。
- CI で確認（**必須**）: (1) `published=false` 記事が成果物に混入していないこと、(2) 成果物 JSON に `published` キーが一切存在しないこと（JSON Schema + grep の両検査）。
- `help_articles.json` は認証済みのみに配信（§5.4）。未認証配信は禁止。認証未実装なら CI 検査に追加して追跡。

**運用フロー**: 公開前は `published=false` でレビュー（承認者 TBD。承認なしの `true` 化禁止）。緊急差替は `published` フラグ切替のみで対応できる構造を維持（SLA は TBD）。バックアップは Git。CMS 移行後は CMS 側に委譲（§9）。

**想定データ上限**: 記事 200 件 / 500 KB。超過時はカテゴリ分割または遅延ロード（TBD）。

## 5. 非機能要件

### 5.1 パフォーマンス [**MVP**]

**計測条件の根拠**: 社内開発環境（ハイスペック + WiFi）で問題なくとも、実利用者はより低性能な端末・回線を利用する。Moto G4 相当（中低端末）と 4G は Web パフォーマンス計測業界で標準的な「最悪ケース保証ライン」。この条件下で p95（昇順に並べた全計測値のうち上位 95% 以内に収まる値。外れ値を除く実用性能指標）が 1 秒以内であれば、実利用者の大半が快適に利用できる。

- 主要操作（検索遷移・サジェスト表示・カテゴリ/記事切替・ソート・絞り込み）は **p95 で 1 秒以内**。
- 計測条件:
  - データ件数: 200 件（§4.3）
  - 端末: Moto G4 相当
  - ネットワーク: 4G（12 Mbps / 70 ms RTT 相当）
  - キャッシュ: なし（初回読み込み）
  - 試行: 各操作 5 回以上、冒頭 1 回（ウォームアップ）は除外
  - ツール: Lighthouse CI または同等
  - 開始: ユーザー操作（クリック・Enter 等）発火時点
  - 終了: 対象 UI 描画完了後の `requestAnimationFrame` コールバック完了時点

### 5.2 レスポンシブ・ビジュアル回帰 [**MVP**]

- SP / Tablet / Desktop で表示を崩さない（ブレークポイントは §4.1.5）。
- WCAG 2.1 AA（Web アクセシビリティ国際基準の中堅レベル）達成閾値: axe-core などで critical / serious 違反ゼロ（§5.5）。
- ビジュアル回帰: 各ブレークポイントでの両 HTML、サジェスト表示/0 件、FAQ カードを含むカテゴリグリッド、`<mark>` ハイライト付き検索結果。

**検証用 XSS ペイロード**（§4.2.2 の手順が機能していることを確認する）:

| # | ペイロード | 期待結果 |
|---|-----------|---------|
| 1 | `<script>alert(1)</script>` | エスケープされて文字列として表示 |
| 2 | `"><img src=x onerror=alert(1)>` | エスケープされて文字列として表示 |
| 3 | `' OR '1'='1` | エスケープされて文字列として表示 |
| 4 | `<mark>test</mark>` | `&lt;mark&gt;test&lt;/mark&gt;` として表示 |
| 5 | `javascript:alert(1)` | `href` に設定されない（URL コンストラクタ検証で弾かれる） |

### 5.3 エラーハンドリング [**MVP**]

- 読み込み失敗時は「コンテンツの読み込みに失敗しました。」を表示。
- エラー UI 要件:
  - リトライボタン、問い合わせ導線（`bug-report.html`）リンク。
  - エラー ID 表示 + コピーボタン（多言語 MVP）。コピー成功時はトースト通知（**必須**）。
  - Clipboard API 失敗時は `<input readonly>` にエラー ID を入れて `select()` で自動選択表示（**必須**）。
- **エラー ID 書式**: `YYYYMMDD-HHMMSS-XXXX`（UTC、XXXX は大文字英数字 4 桁乱数）。`Date.now()` の UTC 文字列化 + `Math.random().toString(36).toUpperCase()` 上位 4 桁。衝突は照合用途のため許容。UUIDv4 化は Phase 2。
- ログ: コンソールへエラーログ出力（詳細 §5.6）。
- URL パラメータが既知 ID に一致しない場合は 404 にフォールバック。

### 5.4 対応環境 [**MVP**]

- **対応ブラウザ（暫定）**: Chrome / Firefox / Safari / Edge 最新 2 バージョン。Phase 2 で見直し。
- **同時アクセス数（暫定）**: CDN 帯域依存。目安値は TBD。
- **公開範囲・認証（セキュリティ交渉不可・必須）**:
  - ダッシュボードログイン済みのみアクセス可。未認証時はログインへリダイレクト。**理由**: 社内運用手順や契約情報を含み得るため公開ドメインへ露出させない。
  - `help_articles.json` は認証済みセッションのみに配信（§4.3）。CI で追跡。
- **CSP**（読み込めるリソース元をブラウザに制限させるセキュリティヘッダー）[Should]: `default-src 'self'` を基本、外部は明示許可（§8.3）。
- **HSTS**（HTTPS 強制接続を指示するヘッダー）[Should]: `max-age=31536000; includeSubDomains`（§8.3）。

### 5.5 アクセシビリティ [**Should**]

- WCAG 2.1 AA 準拠を目標。axe-core で critical / serious 違反ゼロを合格基準（§5.2）。
- 検索サジェストは ARIA `combobox` / `listbox` パターン。
- サジェスト件数・検索結果件数の変化は `aria-live` で読み上げ。
- `<mark>` ハイライトは `aria-label` または前後テキストで補完。
- キーボードのみで全操作可能。
- コントラスト比 4.5:1 以上（AA 基準）。

### 5.6 ログ要件 [**Should**]

**Phase 1 方針（セキュリティ交渉不可）**: サーバー送信を一切行わず、ログは `console.debug()` のみ。**理由**: 送信を始めた瞬間に利用目的公表・法的根拠・保存期間・削除請求などの法令対応が必要になるため、準備が整うまで封印する。Phase 2 収集時は §9 前提条件を満たす。

| 操作 | ログ対象 |
|------|---------|
| 検索実行 / 0 件 | クエリ、結果件数、タイムスタンプ（クエリに PII を含み得るため TBD のマスキング方針に従う） |
| 記事閲覧 | 記事 ID、カテゴリ ID、タイムスタンプ（記事 ID は非 PII） |
| フィードバック操作 | 記事 ID、投票値、タイムスタンプ（投票者識別情報は含めない） |

- 保存期間・マスキング方針はプライバシーポリシー担当と連携（TBD）。
- ブラウザ拡張がコンソール出力を傍受し得るため、セッション ID 等の機微情報は `console.debug` に出さない。
- エラー ID はユーザー自身が問い合わせフォームへ貼り付け、サポートが内部対応表（TBD）と照合する運用。

### 5.7 監視・障害対応 [**Should**]

主要メトリクス（暫定・Phase 2 開始前に確定）: JS エラー率（例: 1%超）、JSON 取得失敗率（例: 0.5%超）、ページロード p95（1 秒、§5.1）。通知先・通知チャネル・一次受け・エスカレーション先・SLO・RTO・RPO・ランブックはすべて TBD（運用引継ぎ資料）。重大障害復旧後、再発性があれば 2 週間以内に FAQ 記事化の要否を検討。

### 5.8 OSS ライセンス [**Should**]

- 依存ライブラリは MIT / BSD / Apache-2.0 互換に限定。
- **禁止**: GPL / AGPL / SSPL / CC-NC。**理由**: ソース開示義務や非商用限定などの制約が本製品の配布形態と衝突するため。
- NOTICE 同梱が必要なライセンス（Apache-2.0 等）は成果物に同梱。依存追加時はライセンス台帳（TBD）を更新。

### 5.9 デプロイ・ロールバック [**Should**]

静的ファイル配置 + CDN パージを想定（詳細 TBD）。ロールバックは旧バージョンの再デプロイ、`help_articles.json` は Git で戻す。CSP / HSTS はデプロイ時に付与（§5.4）。

## 6. 用語集

| 用語 | 説明 |
|------|------|
| `isFeatured` / `published` / `order` / `updatedAt` | 記事メタフィールド。`isFeatured=true` がおすすめ対象、`published=true` のみ配信対象、`order` は並び順、`updatedAt` は更新日時（ISO 8601） |
| `question` / `answer` / `summary` | 記事タイトル・本文（プレーンテキスト）・要約 |
| Bento グリッド | 異なるサイズのカードを格子状に並べるレイアウト（§4.1.6） |
| `resolveLocalizedValue()` | 現在言語のテキストを返すユーティリティ |
| `languagechange` | ブラウザ自体の言語設定変更を示す Web 標準イベント。サイト内切替 UI とは独立（§4.1.8） |
| NFKC | Unicode 正規化の一種。全角英数/半角カナ等を統一形に揃える（現行未対応） |
| seeded 乱数 | シードで同一順序を再現できる擬似乱数（現行未実装） |
| `sa.help.feedback.v1` | フィードバック投票の localStorage キー名前空間（移行先、§4.2.5） |
| PII | 個人識別情報（氏名・メール等） |
| MVP / Should / Nice / Phase 2 | 優先度凡例（§1.1） |
| p95 | 昇順に並べたとき上位 95% 以内に収まる値（§5.1） |
| SLO / RTO / RPO | 可用性目標 / 復旧時間目標 / 許容データ損失時間（§5.7） |
| CSP / HSTS | 読み込み元制限ヘッダー / HTTPS 強制接続ヘッダー（§5.4・§8.3） |
| ARIA | 支援技術向け HTML 属性仕様（§5.5） |
| debounce | 連続イベントを一定時間待って最後の 1 回だけ実行する制御（§4.1.2、現行未実装） |
| localStorage | ブラウザ端末内のキーバリューストレージ。外部送信されない（§4.2.5） |
| `<mark>` | 検索ハイライト用 HTML タグ（§4.2.2） |

## 8. Support 配下サイトの共通アセット運用ルール

> **TL;DR**: 本章は `support.speed-ad.com` / `stg.support.speed-ad.com` 配下（`/faq/`, `/help/`, `/tutorial/`）の共通アセット運用ルール。**§4 のダッシュボード配下ヘルプセンターとは別スコープ**（`SPDAD2026-73` 由来）。

### 8.1 対象 URL / 配置 / 参照

- 対象 URL: `/faq/`, `/help/`, `/tutorial/`。
- ページ本体は各配下、共通 `css` / `js` / `img` は `/assets/` 配下に集約。機能専用リソースは `/assets/img/faq/` のように用途別で分ける。
- HTML からの参照はルート相対で統一（例: `/assets/css/common.20260414.css`）。`../assets/...` のような相対参照は不採用。

```text
/
  faq/index.html
  help/index.html
  tutorial/index.html
  assets/
    css/  (common|faq|help|tutorial).YYYYMMDD.css
    js/   (common|faq|help|tutorial).YYYYMMDD.js
    img/  common/ faq/ help/ tutorial/
```

### 8.2 キャッシュ運用

**バージョン付きファイル名で運用する理由**: CDN やブラウザは同名ファイルを長期キャッシュするため、`common.css` を上書きしても旧版が配信され続ける。ファイル名を `common.20260414.css` のように日付付きにすれば HTML の参照先も変わり、キャッシュと無関係に新版が取得される。ロールバックも「HTML の参照を旧日付へ戻すだけ」で完了する。

- 共通 CSS / JS 更新時はバージョン付きファイル名で運用。サフィックス形式 `<name>.<YYYYMMDD>.css` / `.js`、同日複数回更新は `<name>.<YYYYMMDD>-<N>.css`。
- 旧版ファイルは新版リリース後 +24 時間保持。検証環境（stg）は本番より先にキャッシュパージ。
- 緊急パージ SLA: 重大不具合報告から 1 時間以内に CDN パージ（手順 TBD）。
- 本番・検証はドメインのみを変え、ディレクトリ構造とファイル命名は統一。

### 8.3 ドメイン分離・セキュリティ

- 本番・検証はドメイン分離のため、Cookie と localStorage は自動分離。
- **CSP [Should]**: `default-src 'self'` を基本、外部リソースは明示許可。**理由**: 万一 XSS 混入時も外部ドメインへのリソース取得・外部送信を防ぐため。
- **HSTS [Should]**: `max-age=31536000; includeSubDomains`。**理由**: HTTPS への強制接続で中間者攻撃を防ぐため。
- **`<meta>` robots [Nice]**: `/help/`, `/faq/`, `/tutorial/` 配下の `<head>` に `<meta name="robots" content="noindex, nofollow">` を必須付与（外部検索エンジンインデックス防止）。
- 第三者委託（CDN 等）運用は本要件書の対象外。

### 8.4 非採用方針

- 各機能配下への共通画像・共通 JS の重複配置、共通 CSS / JS の同名上書き運用、相対参照に依存した配置。
