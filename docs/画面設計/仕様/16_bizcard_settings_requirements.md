# 名刺データ化設定画面 要件定義書

## 1. 文書目的
- 本書は、名刺データ化設定画面の本番実装時における正本仕様を定義する。
- 現行の `02_dashboard/bizcardSettings.html` は参照実装であり、本書の正本性は本番要件にある。
- 旧配置の `docs/要件定義/03_bizcardSettings_requirements.md` は legacy とし、今後の更新は本書で行う。

## 2. 画面概要
- 対象画面は、アンケート単位で名刺データ化の実施有無、依頼条件、見込み件数、追加オプション、クーポン適用、社内メモを設定する画面とする。
- 画面は左側の設定エリアと、右側の見積・確定エリアで構成する。
- 本画面はアンケート基本情報、現在の設定値、最新の見積、クーポン適用結果をサーバーから取得して初期表示する。
- 本画面の更新はアンケート単位で永続化され、他ユーザー操作との競合を考慮する。

## 3. 画面要件
### 3.1 上部情報
- パンくずと画面タイトルを表示する。
- アンケート基本情報として以下を表示する。
  - アンケート名
  - アンケートID
  - 会期期間
- 必要に応じてアンケート名と会期期間には補助説明導線を設ける。

### 3.2 左カラム
- 名刺データ化の有効/無効切替
  - ラベルは「名刺のデータ化を行う」とする。
  - OFF 時は設定フォームを非活性化し、見積は 0 円表示とする。
- データ化プラン選択
  - プランは単一選択とする。
  - 各プランは名称、概要、単価、納期目安を表示する。
  - プラン詳細は補助モーダルまたは詳細導線で参照できるようにする。
- 見込み枚数入力
  - 数値入力と増減操作を提供する。
  - 0 以上の整数のみ受け付ける。
- プレミアムオプション
  - 多言語対応などの追加オプションを複数選択可能とする。
  - 金額加算対象と説明のみの項目は明確に区別する。
- 社内管理用メモ
  - 任意入力とする。
  - 利用者向け公開情報には含めない。

### 3.3 右カラム
- 料金見積もりサマリーを表示する。
  - ご請求見込み金額
  - 完了予定日
  - 内訳
- クーポン適用エリアを表示する。
  - 未適用時はコード入力と検証導線を表示する。
  - 適用中はクーポン内容と解除導線を表示する。
- アクションボタンを表示する。
  - `キャンセル`
  - `設定を保存して依頼を確定する`

### 3.4 モーダル・補助導線
- プラン詳細モーダル
- 請求・支払い注意事項モーダル
- 未保存離脱確認モーダル
- クーポン解除確認モーダル

## 4. 状態と挙動
### 4.1 初期表示
- 画面表示時に以下を取得する。
  - アンケート基本情報
  - 既存の名刺データ化設定
  - 適用中クーポン
  - 現在の見積
  - 完了予定日
  - 楽観ロック用 `version`
- 取得失敗時はエラーメッセージと再試行導線を表示する。

### 4.2 名刺データ化 OFF
- 入力欄は非活性化する。
- クーポン入力・検証・解除は不可とする。
- 見積は 0 円とする。
- 完了予定日は「実施しない」または非表示とする。

### 4.3 入力変更時
- プラン、見込み枚数、プレミアムオプション、クーポン変更時は見積を再取得または再計算する。
- 再計算中は右カラムに処理中表示を行う。
- 再計算失敗時は最後に確定した値を保持しつつエラーを表示する。

### 4.4 クーポン
- クーポンコード検証はサーバーに問い合わせる。
- 成功時はアンケートに対する適用可能性、割引種別、割引値、メッセージを返す。
- 失敗時は入力欄に紐づくエラーを表示する。
- 解除時は確認後にサーバー側の適用状態を更新する。

### 4.5 未保存離脱
- 取得時点から変更がある場合、画面離脱前に確認する。
- `キャンセル` 押下時も同様に確認する。

### 4.6 保存
- 保存時は画面を一時的に操作不可とし、二重送信を防止する。
- 保存成功時は最新状態を再反映し、完了通知を表示する。
- 保存失敗時は原因に応じたメッセージを表示し、入力値は保持する。

## 5. 見積要件
### 5.1 見積入力
- 見積に必要な入力は以下とする。
  - `bizcardEnabled`
  - `bizcardRequest`
  - `dataConversionPlan`
  - `premiumOptions`
  - `coupon`
  - アンケート会期情報

### 5.2 見積結果
- 画面表示に必要な見積返却項目は以下とする。
  - 税別金額
  - 必要に応じた税込金額
  - 単価
  - 見込み枚数
  - オプション加算額
  - クーポン値引き額
  - 最低請求条件に基づく注記
  - 完了予定日
- 完了予定日はバックエンド算出値を正とする。
- 納期計算は業務日・休業日・プラン条件を考慮できること。

## 6. 連携要件（参考）
### 6.1 初期表示に必要な情報
- アンケート基本情報
  - アンケート名
  - アンケートID
  - 会期期間
- 現在の名刺データ化設定
  - データ化有効/無効
  - 見込み枚数
  - 選択プラン
  - オプション
  - 社内メモ
- クーポン適用状態
- 現在の見積結果
- 完了予定日
- 他ユーザー更新との競合を検知するための識別情報

### 6.2 保存時に必要な情報
- データ化有効/無効
- 見込み枚数
- プラン
- オプション
- 社内メモ
- 競合制御に必要な識別情報

### 6.3 クーポン検証で必要な情報
- 入力されたクーポンコード
- 当該アンケートに対する適用可否
- 割引内容
- 利用者向けメッセージ
- 必要に応じて適用後見積

### 6.4 設計上の論点
- エンドポイント命名
- HTTP メソッド
- 競合制御方式
- エラー応答形式
- 見積を都度再計算するか、保存時に再評価するか
- 完了予定日を見積系レスポンスに含めるか、別責務に切り出すか

## 7. 失敗系要件
- バリデーションエラー
  - 見込み枚数未入力
  - 見込み枚数が範囲外
  - プラン未選択
- 競合更新
  - 他ユーザー更新との競合を検知し、再読込または再取得導線を表示する。
- 認可エラー
  - 編集権限がない場合は入力不可とし、理由を表示する。
- 一時障害
  - 取得・保存・クーポン検証失敗時に再試行可能とする。

## 8. 非機能要件
- 入力変更に対する見積更新は利用者が待たされない応答時間で行う。
- 二重送信防止を行う。
- 画面再読込後も保存済み設定が正しく再現される。
- 楽観ロックにより他ユーザー更新との競合を検知できる。

## 9. 手動確認観点
- 初期表示で既存設定と見積が正しく表示されること。
- ON/OFF 切替で入力可否と見積が正しく変わること。
- 見込み枚数・プラン・オプション変更で見積と完了予定日が更新されること。
- クーポン適用/解除がサーバー応答に応じて正しく反映されること。
- 保存成功/失敗、競合更新、認可エラーが正しく表示されること。

## 10. 現行モックとの差分
- 現行モックのブラウザストレージ保存、共有クーポンキー、単純日数加算は本番要件に含めない。
- 現行モックの内部プラン値やテスト用クーポンコードは正式契約に含めない。
- 本番では取得・保存・見積・クーポン検証はいずれもサーバー連携を正とする。
- API の具体設計は設計チーム判断とする。
