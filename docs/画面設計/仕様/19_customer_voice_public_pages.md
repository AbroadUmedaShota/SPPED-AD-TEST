# 公開「お客様のお声」ページ仕様

## 関連コード

- `screen_code`: `SCR-P-001`, `SCR-P-002`, `SCR-P-003`
- `related_feature_codes`: `FEAT-002`, `FEAT-003`

## 1. 概要

本書は、刷新されたログイン前トップ (`index.html`) から遷移する公開導線として追加する「お客様のお声」ページ群の表示仕様・導線・データ管理方針を定義する。

対象ページは以下の 3 画面とする。

- `customer-voices/index.html`: お客様のお声一覧ページ
- `customer-voices/company-monitor.html`: 先行導入企業の詳細ページ
- `customer-voices/university-survey.html`: アンケート機能を利用した大学の詳細ページ

## 2. 目的

- ログイン前トップで不足している信頼補強を行う
- `無料ではじめる` 導線へ自然に復帰できる公開導線を用意する
- 「フル活用」と「アンケート単機能活用」の両方を見せ、利用イメージの幅を伝える

## 3. 導線

### 3.1. ログイン前トップからの入口配置

`index.html` には以下の 3 導線を配置する。

1. ファーストビュー内主 CTA: `無料ではじめる`
2. ファーストビュー内補助 CTA: `お客様のお声を見る`
3. 機能紹介後のティザーセクション: 2 事例の要約カード + `詳細を見る`
4. フッター常設リンク: `お客様のお声`

`お客様のお声` 関連導線の遷移先は `customer-voices/index.html` とする。`無料ではじめる` は `?intent=signup#top` でログイン前トップの新規アカウント作成モーダルへ接続する。

### 3.2. ページ間導線

- 一覧ページから各詳細ページへ遷移できること
- 各詳細ページ下部に `他の事例を見る` を配置し、一覧へ戻れること
- 一覧ページおよび詳細ページの主 CTA は `無料ではじめる` とし、遷移先は `../index.html?intent=signup#top` とする
- 副 CTA は `ログインはこちら` とし、遷移先は `../index.html#top` とする

### 3.3. ログイン前トップ復帰時の挙動

- `index.html` は `intent=signup` を受け取った場合、既存の新規アカウント作成モーダルを自動表示する
- モーダル表示後は URL から `intent` を取り除き、再読込時の意図しない再表示を避ける

### 3.4. ログイン前トップとの現行接続状態

ログイン前トップの現行実装状況は `24_public_login_front_requirements.md` を参照する。
`customer-voices/` 配下は、ログイン前トップから接続される公開の信頼補強導線として扱う。

現行実装では、ログイン前トップのティザーは `data/customer-voices.json` の `label` / `listingSummary` を使い、公開事例ページ側では `voicePageLabel` / `voicePageSummary` があればそちらを優先する。
このため、ログイン前トップでは汎用表現を維持し、詳細ページでは承認済みの実名・詳細表現を出し分けられる。

## 4. 画面仕様

### 4.1. 一覧ページ

- 上部にヒーロー領域を表示し、「公開事例の入口」であることを明示する
- 中央に公開中の事例一覧を表示する
- 各事例カードには以下を表示する
  - 区分ラベル (`organizationType`)
  - 事例名 (`voicePageLabel` があればそれを優先し、なければ `label`)
  - 要約 (`voicePageSummary` があればそれを優先し、なければ `listingSummary`)
  - 利用機能 (`usedFeatures`)
- 下部に `無料ではじめる` / `ログインはこちら` を再掲する

### 4.2. 詳細ページ

詳細ページは共通構成を基本とし、必要に応じて専用レイアウトを許容する。基本の表示順は以下とする。

1. 概要
2. 導入前の課題
3. 利用した機能
4. 運用イメージ
5. 導入後の変化
6. 担当者コメント
7. 次の行動

補足:

- `company-monitor.html` は THK 事例専用のエディトリアルレイアウトを許容し、他の詳細ページは共通テンプレートを継続利用する
- 企業事例は「先行導入で得た価値」を主軸にする
- 大学事例は「アンケート単機能でも成立する手軽さ」を主軸にする
- 事例本文は要約表現を基本とし、`customer-voices/` 配下では承認済みの企業名・ロゴを掲載可とする
- 正式所属、個人フルネーム、会議経緯、承認前の具体数値は掲載しない

## 5. データ仕様

### 5.1. データソース

- 事例本文の正本は `data/customer-voices.json` とする
- 一覧ページ、詳細ページ、ログイン前トップのティザーは同一 JSON を参照する
- ログイン前トップのティザーは `label` / `listingSummary` を使い、`customer-voices/` 配下では `voicePageLabel` / `voicePageSummary` がある場合にそちらを優先する
- THK 事例の専用詳細ページでは、追加の任意項目を使ってヒーロー見出し・要点カード・公開用肩書きを描画する

### 5.2. 最低限の項目

各事例には最低限以下を持たせる。

- `slug`
- `label`
- `organizationType`
- `listingSummary`
- `usedFeatures`
- `challenge`
- `outcome`
- `quote`
- `ctaLabel`
- `publishStatus`

### 5.3. 任意項目

以下は任意項目として扱う。

- `overview`
- `operationImage`
- `heroImage`
- `accent`
- `accentStrong`
- `voicePageLabel`
- `voicePageHeadline`
- `voicePageSummary`
- `voicePageHighlights`
- `voicePageImageCaption`
- `bodyImagePrimary`
- `bodyImagePrimaryCaption`
- `bodyImageSecondary`
- `bodyImageSecondaryCaption`
- `publicQuoteAuthor`
- `quote.role`

任意項目が未入力の場合、対応する表示ブロックは非表示とし、空セクションを出さない。

### 5.4. 利用した機能の説明メモ

現行の `usedFeatures` は機能名の配列として扱い、一覧ページと詳細ページでは短いラベル表示に留める。
今後、詳細ページの「利用した機能」をより具体的に見せる場合は、`usedFeatureNotes` のような任意項目を追加し、各機能について「何を使ったか」ではなく「その事例でどう使ったか」を 1 文で補足する。

記載方針:

- 機能紹介ではなく、事例内の利用シーンとして書く
- `WEBアンケート作成` であれば、設問設計や会場導線の整理にどう使ったかを書く
- `名刺撮影` や `名刺情報のデータ化` であれば、回答情報と名刺情報を結び付けて確認しやすくした点を書く
- `QRコード` であれば、来場者自身のスマートフォンや会場掲示から回答へ入れる導線として書く
- `回答CSVダウンロード` であれば、会期後の集計・共有・振り返りに使った点を書く
- `御礼メールの送信` は現行トップで Coming Soon 表示のため、公開事例では提供済み機能として断定しない
- 承認前の成果数値、社内固有の運用名、担当者名、未公開の商談情報は含めない

文案例:

| 機能 | 記載メモ例 |
| --- | --- |
| WEBアンケート作成 | 展示会前にフォロー判断に必要な設問を整理し、来場者対応の流れに合わせた回答フォームを用意した。 |
| 名刺撮影 | 回答内容と名刺情報を同じ文脈で確認できるようにし、会期中から関係部門で来場者情報を見返しやすくした。 |
| QRコード | ブース掲示や配布資料から来場者自身のスマートフォンで回答できる導線を用意し、紙の回収負荷を抑えた。 |
| 回答CSVダウンロード | イベント後に回答データをまとめて確認し、速報共有や次回イベントの改善点整理に活用した。 |

## 7. 確認項目

- `index.html` の 3 導線から一覧ページへ到達できること
- 一覧ページの各カードから正しい詳細ページへ遷移できること
- `../index.html?intent=signup#top` でサインアップモーダルが自動表示されること
- `data/customer-voices.json` の任意項目が欠けてもレイアウト崩れや空セクションがないこと
- デスクトップ / タブレット / モバイルで表示崩れがないこと
- コンソールエラー、リンク切れ、画像切れがないこと

## 8. THK事例の文案・公開方針

### 8.1. 方針メモ

- 主軸は「展示会運用の整理と会期後初動の速さ」とする
- `customer-voices/` 配下では `THK株式会社` を実名表示し、ログイン前トップのティザーは汎用表現のまま維持する
- 詳細ページ冒頭は会社紹介より先に、何が変わったかが一読で伝わる成果見出しを置く
- THK らしさは「紙運用からの移行」「回答・名刺・メモの分断」「会期後フォロー判断の遅れ」で表現する
- 会社説明は補助情報に留め、`LMガイド` `機械要素部品` `メカトロ関連製品` までの公開安全な粒度に制限する
- 参考画像は展示会現場の雰囲気参照に使い、ロゴなし仮ビジュアルと承認後のロゴ入り最終版の両方に差し替え可能な構成にする
- THK 事例ページでは、ヒーロー画像 1 枚と本文サブ画像 2 枚を使い、展示会現場の空気感を補助する
- 正式所属、会議経緯、承認前の数値成果、複数展示会展開の詳細は shared repo に持ち込まない

### 8.2. 添削表

| 対象項目 | 現行課題 | 修正方針 |
| --- | --- | --- |
| `voicePageSummary` | 会社紹介が先に立ち、何が変わったかが弱い | 紙回収と情報分散の解消、翌営業日の初動共有を主メッセージにする |
| `overview` | 会社説明と関係性説明が中心で、展示会運用改善への接続が弱い | 企業背景は短く抑え、会期後フォローへ素早くつなげる必要性を先に置く |
| `challenge` | 抽象語が多く、現場での摩擦が見えにくい | 紙回収、設問設計、優先度判断の 3 点に絞って具体化する |
| `operationImage` | 流れはあるが、企画から会期後共有までの順序がやや弱い | 設問整理 → QR 回答と名刺取得 → 回答・名刺情報の確認の順に並べる |
| `outcome` | 成果の軸が散っており、読み手が変化を一段で掴みにくい | `初動` `共有` `判断` の 3 軸で、数値なしでも改善が読める形に整理する |
| `quote` | 関係性の良さが前に出ており、担当者の実感がやや弱い | 会期後の整理負荷が下がり、次の営業アクションにつなげやすくなった実感に寄せる |
| `voicePageHeadline` | ページの第一印象が要約文に依存しがち | 成果を一言で伝えるエディトリアル見出しを置く |
| `voicePageHighlights` | 読み始める前に全体像を掴みにくい | `導入前` `回収導線` `導入後` の 3 カードで要点を先に見せる |
