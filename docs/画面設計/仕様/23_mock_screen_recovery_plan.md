# モック画面回収計画

## 1. 概要

本書は、SPEED AD の静的モック画面を本番実装へ反映する際の回収方針を定義する。
ここでの「回収」は、モックHTMLをそのまま本番公開することではなく、現行モックに含まれる画面構成、文言、導線、静的コンテンツ、検証用データを棚卸しし、本番の責務別ホストへ取り込むか、参考資料として残すかを判定する作業を指す。

本番想定では、認証・業務操作を担うメインアプリ側と、ヘルプ・FAQ・規約・お知らせなどの静的コンテンツを担うサブドメイン側を分ける。
このため、モック回収では「どの画面をアプリへ入れるか」だけでなく、「どの内容を静的サブドメインへ移すか」を同時に判定する。

## 2. 前提

- ログイン前トップ (`index.html`) は公開入口として扱い、認証・サインアップ・公開LP導線の責務を持つ。
- 利用者向け画面は `02_dashboard/`、管理者向け画面は `03_admin/`、初回ログインは `04_first-login/` を現行モックの確認起点にする。
- ヘルプ、FAQ、問い合わせ、規約、更新履歴、お知らせなどの静的コンテンツは、将来 `support.speed-ad.com` 側の運用対象として扱う。
- `customer-voices/` はログイン前トップから接続される公開信頼補強コンテンツであり、サポート系静的コンテンツとは別に、公開LP/マーケティング導線として扱う。
- `data/*.json` はモック用データであり、本番APIや実DBの仕様確定を意味しない。
- `docs/画面設計/仕様/2026-04-24_dev_mock_diff_report.md` は、開発環境とローカルモックの差分確認結果として参照する。
- `docs/画面設計/仕様/18_screen_inventory_current.md` は、画面一覧と現行HTML/仕様の対応を確認する台帳として参照する。

## 3. 回収分類

モック画面は、以下の分類で扱う。

| 分類 | 対象 | 本番での扱い |
| --- | --- | --- |
| アプリ回収 | ダッシュボード、作成編集、設定、請求、管理者画面、初回ログインなど | `app.speed-ad.com` 相当の認証後アプリへ、UI/導線/文言を仕様化して反映する |
| 公開LP回収 | ログイン前トップ、お客様のお声、登録CTA、公開事例導線 | `speed-ad.com` 相当の公開入口へ、認証導線とマーケティング導線を壊さず反映する |
| 静的サブドメイン回収 | ヘルプ、FAQ、問い合わせ、規約、更新履歴、お知らせ | `support.speed-ad.com` 側へ移管し、ログイン前トップやアプリからは絶対URLで参照する |
| 参考保持 | サンプル、旧導線、Moved画面、検証専用UI | 本番導線には入れず、設計比較・検証用の参考として残す |
| 廃止候補 | 採用予定がなく、仕様にも接続しない画面 | 削除前に台帳へ理由を残し、別PRで整理する |

## 4. 対象別の回収方針

### 4.1. 公開・ログイン前

- `index.html` は公開入口として維持し、ログイン、サインアップ、公開LP、`customer-voices/` への導線を管理する。
- ログイン前ページへ追加するお知らせ欄は、本文正本を `support.speed-ad.com` 側に置き、LPでは最大3件のティザーとして表示する。
- `customer-voices/` は、静的サポートではなく公開信頼補強コンテンツとして扱う。事例本文、画像、JSONの公開安全性を確認したうえで、本番公開可否を判断する。

### 4.2. 利用者向けアプリ

- `02_dashboard/` 配下の業務画面は、静的サブドメインへ移さない。
- 開発環境に既に存在する画面は、URL体系・文言・情報設計の差分を確認し、モックのUI要素を採用するかを画面単位で判断する。
- プレミアム、SPEEDレビュー、グラフ分析など、モック先行で開発環境に未実装または未到達の領域は、機能要否を確認してから回収する。

### 4.3. 管理者向けアプリ

- `03_admin/` 配下は認証後の管理者向けアプリとして扱い、公開静的サイトへ移さない。
- `Moved` 画面、`BY-*`、`sample/` は現行正本に混ぜず、旧・サンプルとして別枠管理する。
- 管理者画面の回収は、利用者向け画面と同じPRに混ぜず、管理者仕様の正本と照合して別作業にする。

### 4.4. 静的コンテンツ

- ヘルプ、FAQ、問い合わせ、規約、プライバシー、特商法、更新履歴、お知らせは `support.speed-ad.com` 側の管理対象とする。
- アプリ側・ログイン前トップ側のリンクは、サポート側へ外部遷移する絶対URLを基本とする。
- 規約・法務ページは版管理と同意記録への影響があるため、本文差し替え前に法務確認を必須にする。
- お知らせは、LP上では最大3件を表示し、詳細は `support.speed-ad.com/news/` へ遷移する。取得失敗時はLP上のお知らせ欄を非表示にする。

## 5. 実行順序

1. 画面棚卸しを固定する。`18_screen_inventory_current.md` と `2026-04-24_dev_mock_diff_report.md` を基準に、回収対象を分類する。
2. 静的サブドメインへ移す対象を先に確定する。ヘルプ、FAQ、問い合わせ、規約、更新履歴、お知らせを `support.speed-ad.com` 管理に寄せる。
3. 静的サブドメインへ移す対象は、旧URLの保持・301リダイレクト・切り戻し期間を `21_support_site_separation_spec.md` と照合して決める。
4. 公開LP対象を確定する。`index.html`、`customer-voices/`、お知らせティザーの関係を整理する。
5. 利用者向けアプリの差分を画面単位で回収する。既存開発環境にある画面は差分吸収、未実装領域は機能要否確認から始める。
6. 管理者向けアプリの回収を別フェーズで実施する。利用者向け回収と混ぜない。
7. 旧・サンプル・Moved画面を整理する。削除ではなく、まず台帳で `参考保持` または `廃止候補` に分類する。

## 6. お知らせ欄との接続

ログイン前ページのお知らせ欄は、モック回収計画の最初の静的サブドメイン回収対象として扱う。

- 正本: `support.speed-ad.com/news.json` または同等の静的JSON
- 表示先: `index.html` のHero直下
- 表示件数: 最大3件
- 表示項目: 日付、タイトル、任意の短い概要、詳細URL
- 種別: 初期段階では持たない
- 失敗時: お知らせ欄を非表示にし、ログイン前LPの主要CTAを阻害しない
- 詳細導線: `support.speed-ad.com/news/` の一覧または詳細へ遷移

本番実装時は、`speed-ad.com` から `support.speed-ad.com` のJSONを読む場合にCORS設定とキャッシュ制御を定義する。
CORSが本番運用上難しい場合は、support側のJSONをビルド時にメイン側へ同梱する方式を代替案とする。

`news.json` と `/news/` は新しい公開静的面になるため、実装前に `18_screen_inventory_current.md` の回収判定台帳と `21_support_site_separation_spec.md` のURL振り分けへ追加する。

## 7. 回収時の禁止事項

- モック用のダミーデータを本番仕様として断定しない
- ローカルURL、localhost、127.0.0.1、検証用アカウントを公開導線に残さない
- 管理者画面や未承認の業務情報を公開静的サイトへ移さない
- 規約、価格、請求、法務文言を承認なしに書き換えない
- `customer-voices/` の実名・画像・成果表現を、承認前に本番公開前提として扱わない
- 画面回収と旧ファイル削除を同じ作業で行わない

## 8. 確認観点

- 回収対象が `アプリ回収`、`公開LP回収`、`静的サブドメイン回収`、`参考保持`、`廃止候補` のいずれかに分類されていること
- 静的サブドメインへ移す対象が `21_support_site_separation_spec.md` の方針と矛盾していないこと
- ログイン前トップの既存導線、モックログイン、`intent=signup`、`customer-voices/` 導線を壊していないこと
- 外部リンクには `target="_blank"` と `rel="noopener noreferrer"` を付与すること
- `data/*.json` の公開可否を確認し、非公開情報や実顧客情報を混入させないこと
- 画面実装を変更する場合は、対応する仕様書と台帳を同じPRで更新すること
- `index.html` を触る場合は、`127.0.0.1`、`localhost`、重複した規約リンク、`role="main"`、`fullPage.js` 初期化が再混入していないことを確認すること
- ログイン前ページのレビュー指摘対応 PR #289 (`codex/login-front-review-fixes-20260427`) とはブランチを分け、同じファイルを触る作業はPR #289の取り込み後に実施すること

## 9. ドキュメント実装フェーズのスコープ固定

本フェーズは回収計画の文書化を目的とし、対象は `docs/画面設計/仕様/` 配下に限定する。

このフェーズでは、以下を変更しない。

- `index.html`
- `02_dashboard/**`
- `03_admin/**`
- `04_first-login/**`
- `customer-voices/**`
- `data/*.json`

画面HTML、CSS、JavaScript、JSONの変更は、回収分類と回収先が確定した後の別フェーズで実施する。

## 10. 回収判定フロー

1画面または1モーダルごとに、以下の順で判定する。

1. 入力資料を確認する。主な入力は `18_screen_inventory_current.md`、`2026-04-24_dev_mock_diff_report.md`、該当する個別仕様書、現行HTML。
2. 回収分類を決める。分類は `アプリ回収`、`公開LP回収`、`静的サブドメイン回収`、`参考保持`、`廃止候補` のいずれかにする。
3. 回収先ホストを決める。候補は `speed-ad.com`、`app.speed-ad.com`、`support.speed-ad.com`、`repo-reference`、`retired` とする。
4. 判定状態を台帳へ記録する。判定根拠と次アクションを必ず残す。
5. `要確認` の場合は、確認相手、確認内容、期限を台帳へ記録し、実装へ進めない。

## 11. フェーズ1完了条件

- 主要な公開、利用者、管理者、初回ログイン、法務・ヘルプ、モーダルの各画面が回収分類を持っていること
- 各対象に回収先ホスト、判定状態、根拠資料、次アクションが記録されていること
- `support.speed-ad.com` へ移す対象は、旧URL保持または301リダイレクト方針が未決のまま実装に進んでいないこと
- `news.json` と `/news/` は新規静的面として台帳上に明示されていること
- `PR #289` と競合する `index.html` / `css/page.css` 変更を本フェーズに混ぜていないこと
