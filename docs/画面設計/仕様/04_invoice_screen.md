## 3. 用語・IDの定義

### 3.1 表示用ID（請求書番号）

- 画面上の請求書番号は `YY-ユーザーID5桁-連番3桁` 形式で表示する（例: `25-00001-001`）。
- 生成ルール（現行モック実装）:
  - `YY`: `issueDate` の西暦下2桁
  - `ユーザーID5桁`: `accountId` の数字部分を抽出し、0埋めで5桁
  - `連番3桁`: 元 `invoiceId`（例: `INV-2025-09-001`）の末尾要素

### 3.2 遷移ID（URLパラメータ `id`）

一覧→詳細/印刷で利用する `id` は以下のいずれか。

- 元請求書 ID: `INV-YYYY-MM-SEQ`（例: `INV-2025-09-001`）
- 集計 ID: `AGG-YYYYMM-{GROUP|PERSONAL}`（例: `AGG-202509-GROUP`、内部互換用）

一覧画面（INV-001）は **請求書単位** を表示するため、詳細遷移には原則として元請求書 ID（`INV-...`）を用いる。

補足（集計 ID の解決）:
- 集計 ID は legacy/internal 互換用の受け口として扱う。
- 集計 ID を指定した場合、詳細/印刷側では該当月・該当種別に属する複数請求書をまとめて1件として扱う。
- 明細は請求書ごとに連結し、各請求書の末尾に「小計」行（`isSubtotal: true`）を挿入する（現行モック実装）。

### 3.3 契約種別（個人/グループ）

現行モック実装の判定:
- `groups.json` 上で `billing.type === 'group'` の `accountId` に一致する請求書 → `グループ`
- 上記以外 → `個人`

## 4. データ契約（画面が期待する `Invoice`）

本要件は API 連携を想定したデータ契約でもあります（現状は `data/core/invoices.json` を参照）。

### 4.1 `Invoice`（請求書）

| 項目 | 型 | 必須 | 説明 |
| :--- | :--- | :---: | :--- |
| `invoiceId` | string | ◯ | 元請求書ID（画面では表示用に整形される） |
| `accountId` | string | ◯ | 請求先アカウントID。`1請求書 = 1accountId` の正本キー |
| `plan.code` | string | ◯ | `STANDARD/PREMIUM/PREMIUM_PLUS` 等 |
| `plan.displayName` | string | ◯ | 画面表示名 |
| `plan.billingType` | string | ◯ | `monthly/annual` |
| `addOns[]` | array |  | 追加オプション（要約表示に使用） |
| `issueDate` | string | ◯ | `YYYY-MM-DD` |
| `dueDate` | string | ◯ | `YYYY-MM-DD` |
| `billingPeriod.from` | string | ◯ | `YYYY-MM-DD` |
| `billingPeriod.to` | string | ◯ | `YYYY-MM-DD` |
| `status` | string | ◯ | `unpaid/paid/overdue/canceled` |
| `bankInfo` | object | ◯ | 振込先情報 |
| `notes` | string/null |  | 備考 |
| `items[]` | array | ◯ | 明細（計算・表示の正） |

### 4.2 `InvoiceItem`（明細）

| 項目 | 型 | 必須 | 説明 |
| :--- | :--- | :---: | :--- |
| `category` | string | ◯ | `BASE/ADD_ON/ONE_TIME/CREDIT`（印刷画面の出力条件） |
| `itemName` | string | ◯ | 品目1。アンケート依存の行ではアンケート名、アンケート非依存の行では具体課金名を表示する。`従量課金` は表示ラベルに使わない。 |
| `description` | string/null |  | 品目2。対象月、適用プラン、送信対象件数、無料100件控除、値引き理由などの課金根拠と対象条件を表示する。数量・単価は原則としてここに重複記載しない。 |
| `quantity` | number/null |  | 数量。課金種別ごとの意味は 4.4 を正とする |
| `unit` | string/null |  | 単位 |
| `unitPrice` | number/null |  | 単価 |
| `amount` | number | ◯ | 金額（未設定時は `unitPrice * quantity` 相当を推奨） |
| `taxable` | boolean | ◯ | 課税対象フラグ（合計計算に使用） |

### 4.3 明細粒度

画面上の主要な明細粒度は以下で固定する。

- `プレミアム利用料`
- `追加アカウント料金`
- `名刺データ化費用`
- `お礼メール送信費用`
- `クーポン値引き`

`プレミアム利用料` と `追加アカウント料金` は、同一請求書内で合算される場合でも別行で表示する。アンケート依存の明細では、`従量課金` を品目ラベルとして使わず、`名刺データ化費用` `お礼メール送信費用` `クーポン値引き` などの具体課金名を表示する。`クーポン値引き` を表示する場合は、対象課金の値引きとして独立行で表示する。

同一アンケート内の表示順は `名刺データ化費用` → `お礼メール送信費用` → `クーポン値引き` を正とする。

品目1 / 品目2 の表示ルールは以下とする。

- アンケート非依存の行:
  - 品目1 = 具体課金名（`プレミアム利用料` または `追加アカウント料金`）
  - 品目2 = 対象月、対象組織名、適用条件などの課金根拠。追加アカウント料金は `請求対象 Nアカウント` のように請求対象人数のみを表示する
  - 追加アカウント料金の無料アカウント数は `2` 固定とし、請求対象は `3アカウント目以降` とする。これは内部ルールであり、請求書表示文言には含めない
- アンケート依存の行:
  - 品目1 = アンケート名
  - 品目2 = `名刺データ化費用` `お礼メール送信費用` `クーポン値引き` などの具体課金名とその根拠
- 小計行を表示する場合:
  - 品目1 = 契約系小計名、または対象アンケート名
  - 品目2 = `小計`

### 4.4 アンケート単位の請求ロジック

- `名刺データ化費用.quantity` は、そのアンケートに付属する名刺のデータ化件数を表す。
- `お礼メール送信費用` は、データ化された連絡先のうち、お礼メール送信設定画面で送信対象に選択した件数を母数として扱う。
- `送信対象件数` は `名刺データ化件数` と一致しなくてよいが、`送信対象件数 <= 名刺データ化費用.quantity` を満たすことを必須とする。
- 無料枠はアンケート単位で `100件` 固定とする。
- `お礼メール送信費用.quantity = max(0, 送信対象件数 - 100)` とする。
- `お礼メール送信費用.quantity = 0` の場合は、`お礼メール送信費用` 行を出さない。
- `お礼メール送信費用.description` は、見栄えを損なわない範囲で `お礼メール送信費用 / 送信対象 N件 / 無料100件控除` のように、送信対象件数と無料枠控除を読み取れる文言を必須とする。

### 4.5 金額計算（現行モック実装）

画面/帳票は `items[]` を正として金額を再計算する。

- 課税小計: `sum(item.amount)`（ただし `item.taxable === true` のみ）
- 非課税小計: `sum(item.amount)`（上記以外）
- 消費税: `floor(課税小計 * 0.1)`
- 合計: `課税小計 + 非課税小計 + 消費税`

## 5. 画面要件

### 5.1 INV-001 請求書一覧

#### 5.1.1 表示要件

- 一覧は `data/core/invoices.json` の各請求書を **請求書単位** でカード表示する。
- 個人アカウント請求とグループアカウント請求は分割せず、同一時系列で混在表示する。
- 表示順は新しい発行日（`issueDate`）が先（降順）。
- カードには最低限以下を表示する。
  - 請求月（例: `2025年09月請求`）
  - 対象期間（例: `2025-08-01 ? 2025-08-31`）
  - 表示用ID（表示用に整形した請求書番号）
  - 契約種別バッジ（`個人` / `グループ`）
  - 請求先アカウント名（例: `個人アカウント` `営業部 (組織)`）
  - 請求先
  - 合計請求額（税込、3桁区切り）

#### 5.1.2 操作要件

- カード押下で INV-002 に遷移する（`invoice-detail.html?id={INV...}`）。
- キーボード操作で `Enter` / `Space` でも遷移できる。

#### 5.1.3 状態（ロード/空/エラー）

- ロード中は「読み込み中です…」を表示する。
- 0件の場合:
  - 「対象の請求書がありません。」を表示し、`再読み込み` アクションを提供する。
- 取得失敗の場合:
  - 「請求データの取得に失敗しました。ページを再読み込みしてください。」を表示し、`再読み込み` アクションを提供する。

#### 5.1.4 フィルタ

- 現行モック画面ではフィルタ UI は提供しない（将来拡張余地として内部ロジックは存在）。

---

### 5.2 INV-002 請求書詳細（Invoice Sheet + PDF出力）

#### 5.2.1 表示要件

- 詳細は、A4の「請求書レイアウト（Invoice Sheet）」を画面内に表示する（カード分割しない）。
- 明細が多い場合はページを自動追加する。
  - 改ページは固定件数ではなく、A4帳票内の実表示高さに基づいて動的に行う
  - 1ページ目も固定件数は設けず、`品名1` `品名2` の折り返し量を含む実表示高さに応じて収まる範囲まで表示する
  - 2ページ目以降も固定件数を保証せず、明細エリアの残余高さに収まる範囲で表示する
  - 実明細が少ないページでは、残余高さに応じて未記載行を補完し、ページ下端まで帳票らしい見た目を維持する
- 請求先（宛名）は `accountId` から解決した `corporateName` / `contactPerson` を表示する。
- 明細は、少なくとも `プレミアム利用料` `追加アカウント料金` `名刺データ化費用` `お礼メール送信費用` `クーポン値引き` を判別できる粒度で表示する。
- `プレミアム利用料` と `追加アカウント料金` は、同一請求書内で合算される場合でも別行で表示する。アンケート依存の明細では `従量課金` を表示ラベルに使わず、`名刺データ化費用` `お礼メール送信費用` `クーポン値引き` を具体課金名として表示する。
- 小計はカテゴリ単位ではなく、`プレミアム利用料 + 追加アカウント料金` の契約系小計と、アンケートごとの小計で表示する。
- 単票請求の明細の並びは `プレミアム利用料` → `追加アカウント料金` → 契約系小計 → アンケートごとの明細と小計とする。
- 利用者向けの主導線では `INV-...` を前提とし、`ご請求明細` は常に `1つの請求先アカウントに対する請求明細` として読める状態を正とする。
- `AGG-...` を直接指定した場合のみ、内部互換として集計請求表示を許容する。利用者向けの正本導線では前提にしない。
- 明細表の `品名1` は品目1、`品名2` は品目2として扱う。
- `品名1` はアンケート依存の行ではアンケート名、アンケート非依存の行では具体課金名を表示する。
- `品名2` は課金根拠、対象月、適用プラン、無料枠超過、値引き理由などを表示する。
- 数量・単価は専用列で表現するため、`品名2` には原則として重複記載しない。

#### 5.2.2 操作要件（帳票ダウンロード）

- `帳票ダウンロード` ボタンで PDF を生成しダウンロードする。
- PDF生成対象は `#invoice-sheet-container` とする。
- ファイル名は `invoice-{id}.pdf`（`id` は URL の `id` 値）とする。

#### 5.2.3 状態（ロード/エラー）

- ロード中はローディングオーバーレイを表示する。
- `id` 未指定: 「請求書IDが指定されていません。」
- データ未存在: 「対象の請求書が見つかりませんでした。」
- その他: 例外メッセージ（または「請求データの取得に失敗しました。」）

### 5.3 INV-003 印刷用ページ（ブラウザ印刷）

#### 5.3.1 表示要件

- `invoice-print.html` は `id` に該当する請求書を取得し、印刷用のHTMLを生成して表示する。
- 明細はカテゴリ単位ではなく、契約系小計とアンケートごとの小計で出力する。
- `AGG-...` は内部互換としてのみ扱い、利用者向けの主導線では `INV-...` を前提とする。
- A4印刷時に `項目` `内訳` の折り返しを許容し、ページ下端ではみ出さないよう高さ基準で改ページする。
- 実明細が少ないページでは、小計行の後ろに未記載行を補完し、ページ下端まで罫線を維持する。
- 明細の主要な粒度は `プレミアム利用料` `追加アカウント料金` `名刺データ化費用` `お礼メール送信費用` `クーポン値引き` とし、クーポン値引きは独立行として扱う。

#### 5.3.2 操作要件

- 画面表示後、印刷ダイアログを自動で開く（`window.print()`）。

## 6. 受け入れ基準（抜粋）

- INV-001: 請求書単位のカードが降順で表示され、個人/グループを識別でき、カード押下で詳細に遷移できる。
- INV-002: `id` に応じた請求書が表示され、PDFが生成できる。
- INV-003: `id` に応じた内容が表示され、契約系小計・アンケート小計を含めて印刷プレビューで崩れない。

詳細チェックは `docs/ハンドブック/テスト/invoice_manual_checklist.md` を参照。

## 7. 既知の留意点（引き継ぎ向け）

- 一覧カードの金額表示の通貨記号は実装上の表示差分が出やすい（提供時に `¥` 表示へ統一すること）。
- `items[].category` と `items[].taxable` は帳票整合のため必須項目として扱うこと（モックデータ側も揃えること）。
- `items[].itemName` または同等の明細表現は、`プレミアム利用料` `追加アカウント料金` `名刺データ化費用` `お礼メール送信費用` `クーポン値引き` の主要明細粒度と整合していること。
