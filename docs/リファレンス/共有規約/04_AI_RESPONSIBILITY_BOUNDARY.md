# AI・実装向け責任分界ルール

shared repo で AI と実装者が日常判断に使う責任分界の要約です。
社内体制そのものの正本ではなく、実装に必要な判断原則だけを再編集して残します。

## 1. 目的

- UI仕様、保存/API仕様、運用フロー仕様の判断先を揃える。
- shared/private の文書境界を崩さずに、AI が参照してよいルールだけを残す。
- 社内体制や会議経緯ではなく、実装判断に必要な結論だけを共有する。

## 2. 基本原則

- 画面体験、文言、表示制御、誘導、業務要件の一次整理は画面仕様を正本とする。
- 保存、API、認可、データ整合、サーバー側制御は UI と独立に成立する前提で扱う。
- 運用画面に関する割当、再割当、状態遷移、確認手順は、実装に必要な粒度まで shared 仕様へ昇格させる。
- 体制、人名、会議判断、社内事情は shared repo に直接書かない。

## 3. 領域別の判断原則

| 領域 | shared での主責任の扱い | AI/実装者の基本動作 |
| --- | --- | --- |
| 画面体験・UI制御 | 画面仕様を正本とする | 表示/非表示、文言、誘導、ロック表現は `docs/画面設計/仕様/` を優先参照する。 |
| 保存/API/認可 | サーバー側整合を必須とする | UI制御だけで完結させず、保存時検証・API権限・不正利用防止を前提に設計する。 |
| データ整合・フィルタリング | 実装ロジック側で保証する | プラン差分、出力制限、データ除外はサーバー側で再検証する前提で扱う。 |
| 運用フロー | 実装に必要な結論だけ shared に残す | 割当、再割当、確認導線、状態遷移が必要なら画面仕様か運用手順へ昇格させる。 |
| 社内体制・判断経緯 | private 正本のみ | shared へは持ち込まず、必要な結論だけ再編集して転記する。 |

## 4. 仕様未確定時の扱い

- UI上の見せ方が未確定なら、画面仕様の更新を優先し、コードや AGENTS のみで補完しない。
- APIや保存時の制約が未確定なら、UI仮実装で済ませず、サーバー側で成立する条件まで明文化する。
- 運用ルールが社内メモにしかない場合は、そのまま参照せず、shared 可能な粒度に再編集して `docs/画面設計/仕様/` または `docs/ハンドブック/` に昇格させる。
- 判断に迷う場合は private 管理を正本とし、shared には結論のみを残す。

## 5. shared に残してよい表現

- 表示条件、入力制約、権限制御、状態遷移、再割当条件
- 保存/API時の検証要件
- 実装直結の運用手順
- AI が迷わないための判断原則

## 6. shared に残してはいけない表現

- 人名、担当者評価、社内事情
- 会議経緯、検討途中メモ、意思決定の背景説明
- 価格、契約、未公開プラン、組織事情
- private 正本への依存がないと理解できない生の体制説明
