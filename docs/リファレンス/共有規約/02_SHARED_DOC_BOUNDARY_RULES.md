# Shared Doc Boundary Rules

shared repo に残す文書境界の正本です。実装会社と共有する前提で、`docs/` 配下に置いてよい情報と private 管理へ分離すべき情報をここで固定します。

## 1. 基本原則

- shared repo に残す基準は「実装直結のみ」です。
- 外部委託先にそのまま見せられないものは private 管理とします。
- 実装に必要な情報が社内文書にしかない場合は、shared 可能な粒度に再編集して `docs/画面設計/仕様/` か `docs/リファレンス/共有規約/` に昇格させます。

## 2. shared に置けるもの

- 画面仕様
- UI 文言、表示条件、入力制約、画面遷移
- 実装参照データとデータ読み込みルール
- 手動テスト手順
- コーディング規約、テンプレート、レビュー補助資料
- 外部共有前提で再編集済みの業務ルール

## 3. private に置くもの

- 価格、契約、法務、未公開プラン
- 社内体制、担当者評価、社内向け運用判断
- 会議メモ、議事録、走り書き、草案
- 企画検討メモ、営業判断、経営判断
- shared 仕様へ再編集していない背景説明

## 4. ディレクトリ方針

- shared 正本: `docs/画面設計/` `docs/ハンドブック/` `docs/リファレンス/`
- shared 補助資料: `docs/テンプレート/` `docs/サンプル/` `docs/変更履歴/CHANGELOG.md`
- internal stub: `docs/プロダクト/` `docs/会議録/` `docs/メモ/meetings/` `docs/legacy-要件定義/`
- アーカイブ: `docs/アーカイブ/`

## 5. 仕様補完ルール

- 画面実装に必要な判断は、社内資料への参照ではなく `docs/画面設計/仕様/` 本文に書きます。
- 仕様書内に社内判断を要約して載せる場合は、背景や意思決定過程ではなく、実装に必要な結論だけを残します。
- premium、回答画面、名刺設定、サンクスメール設定のように業務ルールを含む画面は、shared で読める制約条件を本文に明記します。

## 6. 新規文書追加時の判断

- 実装会社が日常的に参照するなら shared に置きます。
- shared に置く場合でも、社内の事情説明や会議経緯は落としてから記載します。
- 判断に迷う場合は private に置き、必要箇所だけ shared 文書へ転記します。
