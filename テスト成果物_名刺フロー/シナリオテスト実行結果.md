## 2. REG-001（E2E-BUG-001）詳細

| 項目 | 内容 |
|---|---|
| 重大度 | 高（業務停止・データ入力作業が中断、かつデバッグ情報漏えい併発） |
| 対象画面 | `/admin/data_input_screen?id=...`（名刺データ化 グループ1：メールアドレス 作業画面） |
| 再現手順 | メールアドレス入力欄にフォーカス→値を入力→**Enterキー（素のEnter）を押下** |
| 実際の結果 | ページ全体が遷移し、タイトル/本文が `MethodNotAllowedHttpException` / 「The POST method is not supported for route admin/data_input_screen. Supported methods: GET, HEAD.」（HTTP 405）の Laravel デバッグ画面（=Symfony HttpKernel=「カーネル画面」）になる |
| 期待結果 | 素のEnterで画面破壊が起きない（無反応 or 次フィールド移動 or 確定と同等の正常処理） |
| 原因（推定） | 入力フォームに既定のsubmitハンドラがあり、Enterで暗黙のフォーム送信が発火→GET/HEADのみ許可のルートへPOST→405。確定は `Ctrl+Enter` 設計のため、素のEnterは抑止されていない。 |
| 対比 | 正しい確定操作 `Ctrl+Enter` ではエラーは発生しない（正常系を実測） |
| 副次的懸念 | 本番ステージングで Laravel が **デバッグモードON**（スタックトレース全表示＝情報漏えい）。405画面にソースパス・スタックが露出。 |
| 証跡 | `report/20260603125038_REG001再現テスト/02_after_enter.png`（405デバッグ画面）, `REG-001_result.txt` |
| 推奨対処 | (1) フォームの Enter 既定submitを抑止（`preventDefault` or `type=button`化）。(2) 当該ルートに POST 405 時の安全なフォールバック。(3) 本番系のデバッグモードOFF（`APP_DEBUG=false`）。 |
