## E2E-BUG-001（REG-001）: 名刺データ化作業画面で Enter を押すと 405 エラー画面（カーネル画面）に遷移する

| 項目 | 内容 |
|---|---|
| ID | E2E-BUG-001 |
| 別称 | REG-001（回帰バグ番号） |
| 重大度 | **高** |
| 対応テストケース | TC-SC-05 |
| 発見日 | 2026-06-03 |
| 発見環境 | `https://stg.speed-ad.com`（ステージング） |
| 対象画面 | `/admin/data_input_screen?id=<n>`（名刺データ化 グループ1：メールアドレス 作業画面） |

### 再現手順

1. Lv4 管理者アカウントで管理者画面（`/admin/login`）にログインする
2. `/admin/data_input_list` を開き、グループ1（メールアドレス）リンクを押下する
3. `/admin/data_input_screen?id=<n>` が表示されることを確認する
4. メールアドレス入力欄にフォーカスし、任意の値を入力する
5. **素の Enter キーを押下する**

### 実際の結果

ページ全体が以下の Laravel デバッグ画面（カーネル画面）に遷移する。

- タイトル: `MethodNotAllowedHttpException`
- 本文: 「The POST method is not supported for route `admin/data_input_screen`. Supported methods: GET, HEAD.」
- HTTP ステータス: 405
- スタックトレース・ソースパスがデバッグ画面上に露出（SEC-R2-1 と連動）

### 期待結果

Enter キーを押下しても画面破壊が起きない（無反応 / 次フィールドへ移動 / 確定と同等の正常処理）

### 再現率

100%（本テスト実行中に再現を確認）

### 対比: 正常操作（Ctrl+Enter）

`Ctrl+Enter` による確定操作では 405 エラー画面への遷移は発生しない（TC-SC-04 で正常動作を確認済み）。

### 影響範囲

- 名刺データ化業務の全担当者（グループ1 のオペレーター全員）
- 作業中に誤 Enter を押すと作業画面が完全に破壊され、業務が中断する
- デバッグ情報（スタックトレース・ソースパス）がオペレーターに露出する

### 推定原因

入力フォームに HTML 既定の submit ハンドラが設定されており、Enter キー押下で暗黙のフォーム送信（`method=POST`）が発火する。当該ルート（`admin/data_input_screen`）は GET/HEAD のみを許可しており、POST が 405 となる。確定操作は `Ctrl+Enter` として設計されているが、素の Enter の押下を抑止するイベントハンドラが未実装の状態とみられる。

### 推奨対処

| 優先度 | 対処内容 |
|---|---|
| 高 | フォーム内のボタン/input に `type="button"` を設定するか、Enter キー押下時に `event.preventDefault()` を呼び出してフォームの暗黙 submit を無効化する |
| 高 | 当該ルートへの POST リクエストに対し、405 ではなく安全なフォールバック（リダイレクトまたはエラーメッセージ）を返すよう Laravel ルート設定を修正する |
| 高（環境対応） | ステージング本番相当環境で `APP_DEBUG=false` を設定し、デバッグ画面の情報露出を防ぐ（SEC-R2-1 参照） |

- `report/20260603125038_REG001再現テスト/REG-001_result.txt`

## SEC-R2-1: ステージング環境で Laravel デバッグモード ON（スタックトレース露出）

| 項目 | 内容 |
|---|---|
| ID | SEC-R2-1 |
| 重大度 | **高** |
| 対応テストケース | TC-SC-05（E2E-BUG-001 の副次確認） |
| 発見日 | 2026-06-03 |
| 発見環境 | `https://stg.speed-ad.com`（ステージング） |
| 対象 | 全管理画面（デバッグモード全体） |

### 発見経緯

TC-SC-05（REG-001 再現テスト）において、HTTP 405 エラーが発生した際に表示された Laravel のデバッグ画面（Symfony HttpKernel 例外ページ）に以下の情報が露出していることを確認した。

- スタックトレース（フレーム一覧）
- サーバー側のファイルシステムパス
- PHP バージョンおよびフレームワーク内部情報

### リスク

ステージング環境が本番相当のデータや設定を持つ場合、または攻撃者が意図的にエラーを引き起こした場合に、内部情報が漏えいするリスクがある。本番環境で同様の設定が残存していると、セキュリティインシデントにつながる可能性がある。

### 推奨対処

- ステージング環境（および本番環境）の `.env` で `APP_DEBUG=false` を設定する
- 本番環境では例外ハンドラがカスタムエラーページを表示するよう設定する

### 証跡

- `report/20260603125038_REG001再現テスト/02_after_enter.png`（405 デバッグ画面 — スタックトレース露出を含む）
